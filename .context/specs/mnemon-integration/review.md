### MF1 â€” Hook semantics premise wrong (D1, D7)
- **Sources:** Reviewer A (#1 HIGH conf), Reviewer C (#1)
- **Fact-check:** Verified mnemon's `user_prompt.sh` is a 5-line static reminder. It does NOT call `mnemon recall`. The LLM is the one that decides to invoke `mnemon recall` via Bash, prompted by the reminder. Confirmed by reading `~/.mnemon/prompt/skill.md` and the deployed hook script in this turn.
- **Issue:** Design D7 ("recall runs normally; hook captures stdout") and D1 ("TS hook calls mnemon recall directly") are both built on a false model of how mnemon works. There's nothing for the hook to capture because the hook doesn't run recall.
- **Resolution:** Reframe D1/D7. Two viable shapes for shadow mode:
  - (a) **Modified reminder text in shadow phase** â€” Phase 1 Remind hook prints "[mnemon] Phase 1: only consider remember; recall not yet active." LLM doesn't call recall. Day 8 hook flips to standard text.
  - (b) **PreToolUse Bash gate intercepts `mnemon recall`** â€” gate runs the recall, writes results to shadow.log, returns empty stdout to LLM. LLM proceeds without recall context.
  - Recommend (a) â€” simpler, no Bash interception complexity. Operator validates Phase 1 by inspecting `mnemon remember`-written facts directly.
- **Impact:** HIGH â€” design's entire shadow-mode mechanism doesn't exist in v0.1.2. Build would either fail or produce non-functional shadow mode.
- **Effort:** MEDIUM â€” D1 and D7 both rewritten.

### MF2 â€” Build boundary: host modules cannot import into container

- **Fact-check:** Confirmed via `package.json` inspection: host package is `nanoclaw` (Node + pnpm), container is `nanoclaw-agent-runner` (Bun, separate package tree under `container/agent-runner/`). - **Impact:** HIGH â€” code won't compile.
- **Effort:** LOW â€” relabeling the module structure section + path corrections.

## SHOULD-FIX (7) â€” Strongly recommend addressing; not blocking with operator waiver

### SF1 â€” D5 metrics file scheme insufficiently specified
- **Sources:** A (#7), B (drift MEDIUM), C (#6)
- **Issue:** Design says "two-file scheme" but multiple containers writing the same `data/mnemon-metrics-turns.jsonl` violates the one-writer-per-file rule. Per agent-telemetry-spec (T3) and AiOps JSONL guide (T3), per-process files in a directory is the recommended shape.
- **Resolution:** Per-container files at `data/mnemon-metrics/turns/<agent_group_id>__<session_id>.jsonl`; per-store host-written files at `data/mnemon-metrics/stores/<store>.jsonl`. Query CLI walks both directories. Atomicity guaranteed since each file has exactly one writer.

### SF2 â€” Shadow flag should have audit trail, not bare filesystem flag
- **Sources:** B (drift MEDIUM), C (#4)
- **Issue:** Bare flag at `~/.mnemon/data/<store>/.shadow_mode` lacks metadata (when enabled, when graduated, by whom). Silent graduation possible if file accidentally deleted.
- **Resolution:** Replace bare flag with host-side `data/mnemon-rollout.json` mapping `<store>` â†’ `{ phase: 'shadow' | 'live', enabled_at: ISO, graduated_at: ISO | null }`. Mount this file RO into containers; hooks read phase from it. Operator graduates via `pnpm exec tsx scripts/mnemon-phase2.ts <group>` which writes the timestamp + flips phase.

### SF3 â€” D3 flock should only wrap writes; recall is read-only
- **Sources:** A (#4 partial), C (#3)
- **Issue:** Design's hook-integration prose says "Remind acquires the write lock for recall-during-write" â€” internally inconsistent. Recall is read-only per mnemon's design; locking it adds avoidable hot-path latency.
- **Resolution:** Classify mnemon commands explicitly. Reads (recall, search, related, status) â€” no lock. Writes (remember, link, forget, embed, gc) â€” flock. Document this in `src/modules/mnemon/binary.ts` as a constant.

### SF4 â€” proper-lockfile dep choice unspecified; switch to shell `flock(1)`
- **Source:** A (#4)
- **Issue:** Adding `proper-lockfile` to `container/agent-runner/package.json` requires `bun install` + commit of `bun.lock`. Container is the right side (hooks run there), but the dep cost is unstated. Shell `flock(1)` is POSIX, already on Linux containers, zero deps.
- **Resolution:** Use shell `flock /tmp/mnemon-<store>.lock <command>` from container hook callbacks. Zero npm/bun dep added.

### SF5 â€” Host-gateway flag should be added unconditionally now
- **Source:** A (#5)
- **Issue:** Design defers `--add-host=host.docker.internal:host-gateway` to "Day 1 smoke test." Verified absent from container-runner.ts. Design's SOFT punt creates a Day-1 gotcha.
- **Resolution:** Add `--add-host=host.docker.internal:host-gateway` unconditionally in `container-runner.ts` regardless of mnemon enable state. No-op for non-mnemon groups (they don't curl the host); harmless but available when needed.

### SF6 â€” Hook error handling should differentiate failure classes
- **Source:** C (#8)
- **Issue:** Design says "catches errors; logs WARN; returns empty additionalContext (never fails the agent's turn)." Treats all failures as recoverable. Hides corruption, missing binary, permission errors that should block Phase 2 graduation.
- **Resolution:** Define failure classes:
  - **Recoverable (degrade):** Ollama unavailable, transient network, lock timeout, individual recall command failure â†’ log WARN, continue
  - **Blocking (mark store unhealthy):** binary missing, store DB inaccessible, schema-mismatch detected, repeated write failures (>3 in window) â†’ log ERROR, set `mnemon-rollout.json` to `phase: 'unhealthy'`, gate Phase 2 graduation off this state, surface in `task-host-deps-updates` report

### SF7 â€” busy_timeout limitation should be explicit
- **Sources:** B (drift LOW-MEDIUM)
- **Issue:** flock alone is half the SQLite-WAL solution per T1 sources. Mnemon's binary doesn't set busy_timeout. If a future second writer (e.g., metrics collector that opens the DB read-side with shared locks) appears, contention surfaces immediately.
- **Resolution:** In `docs/mnemon.md`, document that current design relies on host-side flock as the sole writer-serialization mechanism. If mnemon ever exposes a `--busy-timeout` flag or env, wire it through `applyMnemonEnv`. File upstream issue (already in scope per brief) requesting this.
