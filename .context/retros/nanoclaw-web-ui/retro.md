## Stage-by-Stage Analysis

### /team-brief — What requirements were missing that surfaced later?

**Skills marketplace search method.** The brief assumed a skills.sh HTTP API proxy. Dave corrected this mid-design: "the skills CLI has a find skill capability `npx skills find <query>`. Should we use this instead?" This changed the marketplace architecture from HTTP proxy to CLI execution. The brief should have asked about existing CLI tools before assuming an API approach.

**Standalone vs bundled deployment.** The initial brief defaulted to bundled mode. Dave asked for a pros/cons analysis mid-design, which led to the standalone-first architecture (C3 constraint). The brief should have identified deployment model as an explicit decision point.

### /team-design — What assumptions were invalidated?

**Theme palette.** The design initially referenced the existing web-ui.html amber palette. Dave corrected: "By nanoclaw's accent colors I meant https://nanoclaw.dev/, not our existing web UI." This required a browser visit to nanoclaw.dev and a full palette revision. The design assumed the existing code was the brand reference when the marketing site was.

**D15 decision record inconsistency.** The decision record for theme (D15) was updated in decisions.yaml but stale text remained in design.md. All 3 reviewers caught this — it's a process smell indicating the design went through multiple revisions without full reconciliation. Evidence: review findings from all 3 reviewers flagged D15.

### /team-review — What issues reached build/QA that review should have caught?

**Timing-safe token comparison.** The security-reviewer in /team-review flagged localStorage XSS and command injection but missed the timing side-channel in `checkAuth` using `===`. This was only caught by /team-qa's security validator. The review's security lens should have checked auth comparison methods — this is OWASP A02 (Cryptographic Failures).

**SSE connection limit.** The review caught the missing WS connection limit (flagged as BUG 5) but missed that SSE had the same problem. The WS cap was added but SSE was left unbounded until /team-qa caught it.

**Security headers.** No review stage flagged the missing `X-Content-Type-Options`, `X-Frame-Options`, or `Referrer-Policy` headers. These are baseline security hygiene that should have been in the design spec, not discovered in QA.

### /team-plan — What was under-specified?

**Session messages endpoint.** Task B4 specified "GET /api/sessions/:key/messages — getRecentMessages() from db.ts" but didn't specify how to look up a session by key when it's not in the active sessions map. The builder implemented a broken fallback (`getSessionsV2Full('', 1000, 0)`) with a comment "This won't work well." This was caught by all 5 reviewers in /review-swarm. The plan should have specified the lookup path for inactive sessions.

**Group A pagination gap.** The original plan (before Group A review) didn't include pagination wrappers for existing functions. The Group A review agent caught that `getShipLog`, `getBacklog`, `listMemories`, etc. lacked `offset` support. This required adding Task A3 (5 paginated wrappers) which wasn't in the initial decomposition.

### /team-build — What blockers occurred?

**Session reset wiped artifacts.** Mid-build, a session reset destroyed all uncommitted spec artifacts (brief, design, decisions.yaml, review). The nanoclaw repo was re-cloned from the host, losing everything in the working tree. This cost ~30 minutes to reconstruct the design from conversation context. Evidence: commit `f7beba9` ("docs: reconstruct nanoclaw-web-ui design spec").

**Background agents lost.** 3 of 4 plan review agents (Groups B, C, D) were running in background when the session reset occurred. Their results were lost. Had to review Groups B-D directly instead of in parallel.

**nanoclaw-ui path issue.** Group C builder created the SPA at `/workspace/group/nanoclaw-ui/` instead of `/workspace/extra/nanoclaw-ui/` because `/workspace/extra/` wasn't writable. Not a blocker but required noting in the report.

**No fix cycles on build.** Groups A, B, C, D all completed on first attempt — no rebuild cycles needed. Build quality was good.

### /review-swarm — What was caught?

10 BUGs + 7 SUGGESTIONs across 5 reviewers. The broken session messages endpoint was caught by all 5 — strongest signal. Key findings by category:
- **Security:** maxPayload missing (DoS), argument injection in marketplace search, ring buffer group leak
- **Correctness:** broken session messages endpoint, fake msgId, capability detection string matching
- **Robustness:** rate limiting bypass on reconnect, PATCH value validation, WS connection limit

5 SUGGESTIONs were promoted to fix-now based on Dave's direction to avoid tech debt.

### /simplify — What was found?

16 findings across 3 reviewers (reuse, quality, efficiency):
- **5 deduplication fixes** (origin validation, SSE broadcast, WS broadcast, error handlers, ActiveSession type)
- **4 quality fixes** (validation constants, unused params, regex gating, type narrowing)
- **5 efficiency fixes** (capabilities caching, EXISTS queries, skills caching, pagination fix, timer cleanup)

The deduplication findings suggest the build agents wrote code independently without awareness of shared patterns. This is expected given isolated task groups, but /team-plan could specify shared utilities as a pre-task.

### /team-qa — What MUST-FIX items were found?

5 MUST-FIX items:
1. Timing-safe token comparison (security — missed by /team-review)
2. SSE connection limit (symmetry with WS cap — missed by /team-review)
3. Security headers (baseline hygiene — never specified in design)
4. Capability COUNT → EXISTS (O(N) → O(1) — missed by /simplify efficiency reviewer)
5. PATCH prompt type validation (inconsistency — missed by /review-swarm)

Items 1-3 could have been caught earlier if the design had included a security baseline checklist.

## Key Learnings

1. **"Commit artifacts immediately after creation — never trust the working tree to survive."** The session reset wiped brief.md, decisions.yaml, and review.md because they were only in the working tree. The design was recoverable only because it was in conversation context. Next time, commit + push each artifact to a branch as soon as it's written. Evidence: 30 minutes lost reconstructing; 3 background agents lost.

2. **"Include a security baseline checklist in the design spec, not just in QA."** Timing-safe comparison, security headers, and SSE connection limits were all caught in QA but should have been in the design. The design focused on functional security (CORS, auth, input validation) but missed infrastructure security (comparison timing, response headers, resource limits). Evidence: QA MF-1, MF-2, MF-3.

3. **"Plan should specify shared utilities before task groups, not discover them after."** The /simplify pass found 5 deduplication opportunities (origin validation, broadcast loops, error handlers) that existed because Groups A and B built independently. A "Task 0: Shared Utilities" in the plan would have prevented this. Evidence: 16 simplify findings, 5 being deduplication.

4. **"Specify the full lookup path for every endpoint, including the 'not found' case."** The session messages endpoint was half-implemented because the plan said "getRecentMessages() from db.ts" without specifying how to resolve a session key to a chat_jid for inactive sessions. This is the most-flagged finding across the entire workflow (caught by all 5 review-swarm reviewers). Evidence: routes.ts:210 dead code with "This won't work well" comment.

5. **"Dave's corrections improve scope — build them into the process."** Dave's interventions (nanoclaw.dev vs web-ui.html for brand, `npx skills find` vs HTTP proxy, standalone vs bundled) were all improvements that prevented wrong-direction work. The brief stage should explicitly ask: "Are there existing tools/CLIs/sites I should reference?" Evidence: 3 mid-design corrections from Dave.

## Recommended Updates

### CLAUDE.md

**Add to "Pre-PR Quality Gates" or a new "Security Baseline" section:**
> All new HTTP endpoints must include: timing-safe auth comparison, connection limits (SSE and WS), security response headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy). These are non-negotiable baseline requirements, not QA findings.

*Reason:* QA MF-1, MF-2, MF-3 — all baseline security items that should be guardrails, not discoveries.

**Add to a "Session Persistence" note:**
> Commit spec artifacts (brief, design, plan, review) to a branch immediately after creation. Do not rely on the working tree surviving session resets.

*Reason:* Session reset wiped 3 artifacts; only design was recoverable from context.

### Workflow Skills

**/team-plan (Step 4):**
> Add a "Task 0: Shared Utilities" pattern. Before assigning tasks to groups, identify functions that multiple groups will need (auth helpers, broadcast utilities, error handlers, shared types) and assign them to the earliest group. Reference these shared utilities in dependent group task specs.

*Reason:* 5 deduplication findings in /simplify that would have been avoided with a shared utilities pre-task.

**/team-plan (Step 4e, named test cases):**
> For every API endpoint, the plan must specify the "not found" lookup path — not just the happy path. If an endpoint resolves an ID to a resource, specify which DB function or lookup handles the "resource doesn't exist" case.

*Reason:* Session messages endpoint was half-implemented because the plan didn't specify inactive session lookup.

**/team-review:**
> Add "Security baseline" to the required reviewer checklist: timing-safe auth comparisons, resource limits on all connection types (not just the new ones), security response headers. These should be checked by the security-reviewer lens, not deferred to QA.

*Reason:* 3 of 5 QA MUST-FIX items were baseline security that /team-review missed.

**/team-brief (Step 3):**
> Add a standard question: "Are there existing CLIs, tools, or reference sites (marketing site, brand guide) that should inform this feature?" This catches cases where the user has context the assistant doesn't know to ask about.

*Reason:* 3 mid-design corrections from Dave (nanoclaw.dev brand, `npx skills find`, standalone architecture).
