## Reference Audit

This rebuild replicates the architecture in **Vivian Balakrishnan's NanoClaw v1 gist** ([gist a7d4eec](https://gist.github.com/VivianBalakrishnan/a7d4eec3833baee4971a0ee54b08f322)). The prior attempt (PR #68) failed because it integrated against the wrong reference — `mnemon-dev/mnemon`'s own Claude Code integration pattern (agent-driven inline writes), not Vivian's three-layer pipeline. The two have fundamentally different control models. This brief is grounded in literal quotes from Vivian's gist for every architectural claim.

### What we replicate from the canonical reference

**Three-layer pipeline.** Verbatim from the gist:

```
Raw sources          →    mnemon graph         →    wiki pages
(transcripts/            (structured facts,         (narrative syntheses,
 articles,                graph nodes,               human-readable,
 web clips)               semantic retrieval)        cross-referenced)
         ↑                      ↑                          ↑
   ingest pipeline        auto-recall hook          synthesise operation
```

**Three operations.** Verbatim:
1. *"Extract — pull discrete facts, insights, and style preferences from raw documents (speeches, articles, conversations) into a graph database. Each entry is a self-contained, retrievable statement."*
2. *"Synthesise — compile those facts into human-readable wiki pages organised by entity, concept, and timeline."*
3. *"Recall — on every agent invocation, run a semantic query against the graph using the user's message as input. Relevant entries are injected as context before the agent responds."*

**The defining recall principle.** Verbatim: *"Every agent invocation triggers a semantic recall against the knowledge graph. Relevant facts surface automatically as a system reminder — **the agent never has to decide to 'look something up.'**"* This is the constraint that PR #68 violated by injecting *"only consider remember; recall not yet active"* during shadow phase.

**Source content includes conversations.** Verbatim from the Extract description: *"raw documents (speeches, articles, **conversations**)"* — chat is an explicit source type, not a side channel. The chat-stream classifier (Dave's v1 pattern: 60s polling, classify all messages since last watermark) is the implementation of conversation-as-source within Vivian's framework, not a separate hybrid. v1's gap was no wiki layer and no audit trail; Vivian's gist closes both gaps.

**Why mnemon + wiki, not just RAG.** Verbatim: *"RAG retrieves text chunks; mnemon stores synthesised facts as discrete nodes. The wiki layer adds human-readable narratives that can be reviewed and corrected. The two-tier design (mnemon for recall, wiki for synthesis) separates retrieval from presentation."*

**Local embeddings via Ollama + nomic-embed-text.** Already deployed on the host (per `docs/mnemon.md` from prior build); preserved in this rebuild.

### What we explicitly deviate from

Adding global tier is non-trivial (mount permissions, wrapper changes, recall merge logic) and not needed for v1-parity. The `MemoryStore` interface leaves room for a future global tier without breaking changes. |

Tool-output capture (any content the agent fetches via Granola, Pocket, gws Docs, URL fetch, attachments, Linear/GitHub threads) routes through the same source pipeline as Vivian's web clips — the *fetch* is the curation signal. Manual drop path retained for batch ops. |
| Subagent teams for parallel work | Not a memory-layer concern — out of scope for this rebuild | Vivian's Subagent Teams capability is orthogonal to memory. |

### What the canonical reference does not say (open decisions resolved in this brief)

- **Polling cadence + watermark mechanics** — Vivian's gist describes the operations but not the ingestion daemon's schedule. Dave's v1 used 60s polling with watermark-based "everything since last check." Adopted as baseline in this brief.
- **Chat-stream classifier prompt + cost shape** — gist doesn't specify. Defaulted to Haiku-class extraction with a `worth_storing` decision field; substantive-message pre-filter to skip trivial turns (length, single-emoji, ack-only).
- **Global vs local fact routing** — gist says global is "read-only for non-main agents" but doesn't specify how facts get *into* global. Deferred to next phase.
- **Recall failure semantics** — gist doesn't address what happens if Ollama is down or mnemon's embedding store is corrupt. Defaulted to fail-open (no facts injected, agent responds anyway, telemetry surfaces lag).
- **Recall latency budget** — gist doesn't quantify. Codex recommendation adopted: 750ms warm, 1500ms hard timeout.
- **Provider portability** — gist assumes Claude. NanoClaw v2 has a provider abstraction (Codex, OpenCode planned). Recall injector must live above provider boundaries.

## Requirements

### R1 — Auto-recall via inbound system message (the defining capability)

Every chat-kind agent invocation (chat, chat-sdk, agent-to-agent, webhook) must trigger a semantic recall against the relevant mnemon store. Top-K results are written to inbound.db as a `kind:'system'` row with `subtype:'recall_context'` BEFORE the user message is written, and the agent-runner formatter renders them as a `[Recalled context]` block visible to the agent before it processes the user message. The agent does not call recall itself.

**Scope decision (Q1 / cycle-2 M1 resolution):** scheduled tasks (synthesise, future scheduled prompts) do NOT trigger automatic recall. Synthesise specifically reads mnemon directly via Bash invocation of the mnemon CLI as part of its task flow. Other scheduled tasks may opt into recall via the same path if needed.

**Authority decision (Q2 / cycle-2 M2 resolution):** recall lands at transcript-level (visible to the agent as a system message in the conversation stream), NOT system-prompt-level. This is intentional: auto-extracted facts have classifier-derived provenance, not curator-audited authority. The agent reads recall context as informational, not as ground-truth that supersedes its reasoning. Vivian's gist used "system reminder" framing under the assumption of curated memory; we're running automated extraction without an audit step, so transcript-level treatment is the more honest semantics.

The noise pre-filter (`shouldRecall`: length, single-emoji, ack-only) still applies — recall does not fire on trivial turns.

### R2 — Chat-stream extraction via async classifier

A host-side daemon polls `data/archive.db` every 60s, batches new messages into conversation **turn pairs** (a stretch of consecutive user messages followed by the immediately-following stretch of consecutive assistant messages within a single thread), and runs a Haiku-class classifier per stable turn pair. A turn pair is *stable* when the last assistant message is older than the stability threshold (default 120s). Threads that span days produce many turn pairs — each user-burst → agent-burst exchange is one pair. The daemon does not wait for the thread to end. Extracted facts are written to the appropriate per-group mnemon store. The classifier never blocks a chat turn. The agent never invokes the classifier. *(Mirrors Dave's v1 pattern; "conversations" is an explicit source type per Vivian's gist Extract description.)*

### R3 — Tool-output capture as ingest signal

When the agent invokes a content-fetching tool (URL fetch, Granola/Pocket/gws/Gmail/Linear/GitHub readers, attachment downloads), the fetched content is automatically routed through the source ingestion pipeline — saved to `groups/<g>/sources/<type>/`, content-hash-deduped, and ingested by the daemon. The agent receives the content as normal tool output; no agent-side memory awareness required.

### R4 — Manual file drops (tertiary ingestion path)

Operator can drop files into `groups/<g>/sources/inbox/`; the daemon picks them up via filesystem event or next poll. Same content-hash dedup as R3.

### R5 — Wiki synthesis via agent-driven scheduled task per group

Daily synthesise scheduled task per group reads the group's mnemon store and compiles atomic facts into wiki pages organized as `entities/`, `concepts/`, `timelines/`. Synthesis is agent-driven (the group's own agent runs it via the existing scheduled-task mechanism) so domain context, voice, and existing wiki page content all inform output. Operator hand-edits in Obsidian must be preserved through future synth runs (merge, don't clobber). *(Vivian's gist: Synthesise is one of the three core ops.)*

### R6 — Two-store interface, single backend now

The memory layer must be addressed by the rest of the codebase through a `MemoryStore` interface, not directly via `mnemon` CLI calls. mnemon is the implementation today; the interface must permit a future swap to sqlite-vec or another backend without touching the recall injector, the chat-stream daemon, or the tool-output capture path.

### R7 — Cross-provider recall injection

Auto-recall must work identically when the agent provider is Claude (today), OpenCode, or Codex (future). The injector lives above the provider abstraction layer (in or above `container/agent-runner/src/providers/`) — not inside Claude SDK hooks. Adding a new provider must not require duplicating the recall pipeline.

### R8 — Surgical removal of all PR #68 broken pieces

Every component from the prior failed build must be either removed or repurposed via the new interface. No dead code, no commented-out blocks, no orphan feature flags, no unreachable scheduled tasks. Specifically:
- Delete: `container/agent-runner/src/modules/mnemon/{hooks,binary-classes,failure-class,rollout-reader,metrics-emit,block-mnemon-real-hook,*.test}.ts`, `src/modules/mnemon/*.ts`, `container/skills/mnemon-companion/SKILL.md`, `scripts/mnemon-phase2.ts`, `scripts/mnemon-metrics-collector.ts`, `data/mnemon-rollout.json`, `data/mnemon-health.json`, all `groups/*/.mnemon-metrics.jsonl`, the entire phase-gating block in `container/mnemon-wrapper.sh`.
- Keep: mnemon binary install in `container/Dockerfile`, Ollama on host (existing systemd unit), wiki autopush cron, GitHub wiki repos under `groups/*/wiki/`.
- Repurpose: the existing `wiki/SKILL.md` (container skill) — needs revision to reflect that synthesis happens via scheduled task, not on-demand by agent during chat.

### R9 — Per-group enable/disable

Operator controls per-group memory enablement via a single switch in `groups/<g>/container.json` (e.g., `"memory": { "enabled": true }`). When disabled: no recall injection, no chat-stream classification, no tool-output capture, no synthesise task scheduled. When enabled: all four behaviors active. No phase concept; no shadow mode.

### R10 — Health + observability

The host daemon emits structured metrics (per-group): chat-stream lag (oldest unprocessed message), classifier success/failure/dead-letter counts, source-file ingest success/failure counts, recall query p50/p95 latency, recall fail-open events. A single `data/memory-health.json` aggregates current state for operator inspection. Lag alarm: if oldest unprocessed message is older than N minutes (default 10), surface it.

## Constraints

### Architectural constraints

- **[HARD]** Memory writes are async and must never interrupt the main conversation. Agent never blocks on a memory write.
- **[HARD]** Agent never decides what to remember. Classifier (host-side daemon) is the only fact-write decider.
- **[HARD]** Recall is automatic system-reminder push on every agent invocation, with noise pre-filter (length under N chars, single-emoji, ack-only words like `ok`/`yes`/`👍`).
- **[HARD]** Recall fail-open: if mnemon or Ollama is unavailable, no facts are injected, agent responds normally, telemetry surfaces failure.
- **[HARD]** Recall p50 latency target ≤ 750ms warm; hard timeout 1500ms.
- **[HARD]** No phase gating, no shadow/live distinction. Memory is on or off per group.
- **[HARD]** Memory layer accessed through a `MemoryStore` interface; no direct mnemon CLI dependency outside the implementation file.
- **[HARD]** Cross-provider portable: recall injection lives above the provider abstraction, not inside Claude SDK hooks.

### Cleanup discipline

- **[HARD]** Surgical removal of PR #68 broken pieces (R8). No dead code, no commented-out blocks, no orphan feature flags, no unreachable scheduled tasks.
- **[HARD]** Existing wiki repos under `groups/*/wiki/` and `scripts/wiki-autopush.sh` cron must remain operational throughout the rebuild — no disruption to wiki autopush.
- **[HARD]** Existing Ollama systemd unit on host must remain unmodified; this rebuild depends on it.

### Ingestion + dedup

- **[HARD]** Chat-stream classifier batches turn pairs. A turn pair is: a stretch of consecutive user messages within a single thread + the immediately-following stretch of consecutive assistant messages. Turn pairs can span poll windows. Single-message classification is forbidden — turn pair is the minimum unit. Threads that span days produce many turn pairs.
- **[HARD]** Turn-pair stability threshold: 120 seconds. A pair is classifiable only when the last assistant message is older than 120s (no further agent activity expected).
- **[HARD]** Turn-pair orphan threshold: 10 minutes. A user-only run (no agent response) is classifiable on its own after 10 minutes, with `is_orphan: true` provenance. Captures user-stated facts even when the agent never triggered.
- **[HARD]** Turn-pair classification fires on the next 60s sweep after stability is reached, not event-driven. Up to 60s latency between pair-stable and fact-written is acceptable; this keeps the daemon's logic uniform (poll-driven only).
- **[HARD]** Chat-stream classifier idempotency keyed on `(message_archive_id, classifier_version)`. Re-running over the same messages must not double-write facts.
- **[HARD]** Source ingestion idempotency keyed on `sha256(canonicalized_content) + extractor_version + prompt_version`. Re-fetching or re-dropping the same content produces zero new ingests.
- **[HARD]** Substantive-message pre-filter for chat-stream classifier: skip trivial turns (under N chars, single-emoji, ack-only). Threshold tunable.
- **[HARD]** Classifier output must include a `worth_storing: boolean` field; defaults to `false` (the classifier must affirmatively decide a fact is durable).
- **[HARD]** Classifier must refuse to extract secrets, credentials, tokens, or transient session state. Operator-defined deny patterns honored.

### Polling + watermarking

- **[HARD]** Chat-stream daemon polls `data/archive.db` every 60s.
- **[HARD]** Per-group, per-thread watermark on `sent_at` (or message_archive_id) tracks "last classified message." Watermark advances only on classifier success.
- **[HARD]** Classifier failure → message stays unclassified; retry on next poll with exponential backoff. Dead-letter after N retries.

### Storage + isolation

- **[HARD]** Per-group local mnemon stores, no cross-group access. Cross-tenant isolation enforced at filesystem (mount narrowing) and CLI (`MNEMON_STORE` env var pinning) levels.
- **[HARD]** No global store in this build. `MemoryStore` interface leaves room for one in a future build without breaking changes.

### Operational

- **[HARD]** Single PR, end-to-end working build, demonstrated on all 7 production groups before merge gate. No phased rollout.
- **[HARD]** Pre-merge smoke test: send substantive message to one group (illysium recommended); within next 60s sweep, verify chat-stream classifier ran, fact landed in mnemon, recall injected on next message.
- **[HARD]** Host-prerequisite verification at build start: Ollama service active, nomic-embed-text model pulled, mnemon binary at expected version, host disk space adequate for daemon state.
- **[SOFT]** Daemon should be observable in real time (`memory-health.json` updated within seconds of state change); polling at 5s for the health file is acceptable.

### Supply chain

- **[HARD]** No additions to `pnpm-workspace.yaml` `minimumReleaseAgeExclude` without operator approval.
- **[HARD]** No new dependencies on the `onlyBuiltDependencies` list without operator approval.
- **[HARD]** Anthropic SDK version pinning preserved; Haiku model version pinned (current: `claude-haiku-4-5-20251001`).

## Key Concepts

- **Conversation turn pair**: a stretch of consecutive user messages within a single thread + the immediately-following stretch of consecutive assistant messages, batched together for one classifier call. Bounded by **role-flip + response stability**, not by thread closure. A long-running thread produces many turn pairs (one per user-burst → agent-burst exchange). The classifier processes a pair when its assistant run is stable (last assistant message > 120s old). User-only orphans (no agent response) are classified alone after 10 minutes.
- **Source content**: any non-chat content ingested into mnemon — webpages, transcripts, documents, attachments. Always content-hash-deduped at ingestion.
- **MemoryStore interface**: the abstraction boundary between the rest of the codebase and mnemon. Recall injector, chat-stream daemon, tool-output capture all depend on the interface.
- **Watermark**: per-group, per-thread cursor on `data/archive.db` tracking the last classified message. Advances only on success.
- **Fact (atomic)**: a discrete, retrievable statement extracted from a turn or source. Has content, category, importance, entities, source provenance. Unit of mnemon storage.
- **Synthesise task**: daily scheduled task per group that wakes the group's agent to compile mnemon facts into wiki pages. Agent-driven for domain context.
- **Substantive turn**: a chat turn that survives the noise pre-filter (length, content shape). Eligible for classifier extraction.

## Open Questions (resolved by defaults — flag if assumption wrong)

- **Recall query strategy**: defaulting to **raw user message, normalized** (strip mentions, attachment metadata, length-cap). No LLM-rephrasing in the recall hot path. *(Per Codex recommendation; gist says "using the user's message as input.")*
- **Recall top-K**: defaulting to **K=5**, with token-budget cap of 500 tokens for injected context.
- **Classifier model**: defaulting to **`claude-haiku-4-5-20251001`**.
- **Classifier prompt cache**: defaulting to **enabled** (system prompt + extraction instructions cached; per-turn payload is the only varying part).
- **Wiki synthesis cadence**: defaulting to **daily at 03:00 UTC per group** (matches existing scheduled-task pattern).
- **Substantive-message threshold**: defaulting to **≥ 20 chars OR ≥ 4 words after stripping mentions/whitespace**, plus ack-list `[ok, yes, no, sure, k, 👍, 🙏, lol, cool, nice, thanks, thx, np, …]`. Tunable via daemon config.
- **Health JSON polling cadence by operator tooling**: defaulting to **5s** when polled.

## Success Criteria

### Functional (must work end-to-end before merge)

1. **Chat-stream extraction lands facts.** Send 5 substantive turns in one production group (e.g., illysium); within 60s of the last turn, ≥ 1 fact appears in that group's mnemon store with `source: "agent" | "classifier"` (or whatever the source attribution becomes — distinguishable from manual writes).
2. **Auto-recall surfaces facts.** After step 1, send a follow-up message in the same group that semantically references one of the stored facts. Verify the agent's response demonstrates awareness of the recalled fact (operator confirms by reading response, or by inspecting the injected `additionalContext` in agent-runner logs).
3. **Tool-output capture lands facts.** Have agent fetch a URL/article in chat; within 60s of the fetch, the source file appears in `groups/<g>/sources/<type>/` and ≥ 1 fact extracted from it lands in the local mnemon store.
4. **Manual drop captures.** Drop a markdown file in `groups/<g>/sources/inbox/`; within next poll, the file is processed, fact(s) extracted, source moved to a processed/ subdirectory.
5. **Synthesis produces wiki pages.** After enough chat-stream + tool-output activity to populate ~10 facts in a group's mnemon store, the next-day's synthesise task produces at least one new or updated wiki page under `groups/<g>/wiki/{entities,concepts,timelines}/`.
6. **Wiki autopush still working.** Throughout the build, `groups/<g>/wiki/` repos auto-push every 5 min as before — no regression to the orthogonal pipeline.
7. **Cross-provider stub.** Even though Claude is the only provider today, the recall injection code path must be exercisable from a non-Claude entrypoint (smoke test via a unit test that mocks an OpenCode/Codex provider call).

### Cleanup (must be visibly clean)

8. `git diff` of merged PR shows **zero** files matching `container/agent-runner/src/modules/mnemon/{hooks,rollout-reader,binary-classes,failure-class,metrics-emit,block-mnemon-real-hook}.ts`, `src/modules/mnemon/*.ts`, `container/skills/mnemon-companion/SKILL.md`, `scripts/mnemon-{phase2,metrics-collector}.ts`, `data/mnemon-{rollout,health}.json`, `groups/*/.mnemon-metrics.jsonl`, or any `phase`/`shadow`/`live` references in `mnemon-wrapper.sh`. (Files removed, not just emptied.)
9. `grep -rE 'phase|shadow|live' container/mnemon-wrapper.sh src/modules/memory/ container/agent-runner/src/modules/memory/` returns only references unrelated to the dropped phase concept.
10. Health daemon (`data/memory-health.json`) reports green for all 7 groups within 5 min of post-merge restart.

### Operational

11. Smoke test on all 7 groups within 24h of merge: each group has ≥ 1 chat-derived fact in its mnemon store. (Confirms the daemon is iterating across all groups, not just one.)
12. No regression in chat latency: post-merge response p50 within 5% of pre-merge baseline (recall fires before `provider.query()`; daemon writes are off the hot path).
13. Operator can disable any single group's memory layer via `groups/<g>/container.json` flip + restart — no facts ingested, no recall injected, no synthesise task scheduled for that group while disabled.

### Observability

14. `data/memory-health.json` exposes per-group: chat-stream lag, classifier success/fail/dead-letter counts, source-ingest success/fail counts, recall query latency p50/p95, recall fail-open count.
15. Classifier prompt + version traceable in mnemon insight metadata (`source_provenance` field) so future re-runs can identify which classifier version produced a fact.
