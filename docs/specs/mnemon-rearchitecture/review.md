## MUST-FIX Findings

### [M1] `block-mnemon-real-hook.ts` retained but its claude.ts wiring is deleted (M3/M7 conflict)

- **Raised by:** Reviewer A (F1), Reviewer C (#2) — convergence
- **Finding:** §Cleanup Inventory §M7 retains `block-mnemon-real-hook.ts` as defense-in-depth. But §Cleanup Inventory `claude.ts` edits explicitly say to REMOVE: the import line, the `mnemonStore` field, the field init, and the conditional `...(this.mnemonStore ? [createBlockMnemonRealHook()] : [])`. With those edits applied verbatim, the hook exists but is never wired to any PreToolUse matcher. Defense layer becomes dead code.
- **Effort to Fix:** Low (replace gate with new env var)
- **Impact if Ignored:** High — agent can write to mnemon despite RW mount + wrapper enforcement bypass via direct `mnemon-real` invocation
- **Suggested resolution:** §Cleanup Inventory `claude.ts` edits change "REMOVE the import" to "UPDATE the import path to `../modules/memory/block-mnemon-real-hook.js` (file moved per M7)." Replace the gate from `this.mnemonStore` to `process.env.MNEMON_READ_ONLY === '1'`. Keep the conditional Bash matcher entry.
- **Status:** Open — mechanical

### [M2] `writeSessionMessageRaw` referenced but doesn't exist

- **Raised by:** Reviewer A (F2)
- **Finding:** §Detailed Design 2 `maybeInjectRecall` calls `writeSessionMessageRaw(...)` — this function doesn't exist in the codebase (`src/session-manager.ts` has only `writeSessionMessage`). Pseudocode primitive is fictional.
- **Effort to Fix:** Low (use existing primitives or specify new helper)
- **Impact if Ignored:** High — same retro-#2 root cause as PR #68's wrapper-jq-path bug
- **Suggested resolution:** Replace with explicit call sequence: `insertMessage(sessionDB, { ... })` (existing in `src/db/session-db.ts`) followed by `updateSessionLastActive(sessionId)`. OR add `writeSessionMessageRaw` to `§Detailed Design 1` + new-files list as a private helper in session-manager.ts that bypasses the recall hook (to prevent recursion).
- **Status:** Open — mechanical

### [M3] `MemoryStore.remember()` interface signature missing `idempotencyKey` param

- **Raised by:** Reviewer A (F3), Reviewer C (#3) — convergence
- **Finding:** §Detailed Design 1 declares `remember(agentGroupId, fact): Promise<RememberResult>` (2 args). M10 cycle-2 resolution + §Detailed Design 3 daemon pseudocode call `memoryStore.remember(agentGroupId, fact, { idempotencyKey })` (3 args). Interface and call site out of sync.
- **Effort to Fix:** Low (one signature update)
- **Impact if Ignored:** High — daemon won't compile; M10 cycle-2 fix unverified
- **Suggested resolution:** Update interface to `remember(agentGroupId: string, fact: FactInput, opts?: { idempotencyKey?: string }): Promise<RememberResult>`. Document idempotency contract in mnemon-impl.
- **Status:** Open — mechanical

### [M4] S5 typing-indicator placement contradicts M1 hook location

- **Raised by:** Reviewer A (F4)
- **Finding:** S5 cycle-2 says "Typing indicator activated BEFORE recall call in router." M1 places the hook in `writeSessionMessage` (session-manager), not router. Router currently calls `startTypingRefresh` AFTER `writeSessionMessage`. To activate typing before recall, typing call must move ahead in router, OR S5 only applies to router-path writes. Design body doesn't reconcile.
- **Effort to Fix:** Low (scope clarification + one router edit)
- **Impact if Ignored:** Medium-High — UX latency masking guarantee unclear; non-router writers (a2a, approvals) have no typing indicator anyway
- **Suggested resolution:** Add to §Cleanup Inventory `src/router.ts` edit: move `startTypingRefresh()` BEFORE `writeSessionMessage` for chat-kind paths. Document that S5 applies only to platform user message writes (which is also the only kind where typing indicator is meaningful).
- **Status:** Open — mechanical

### [M5] Daemon classifier missing `channel_type='agent'` filter on archive.db read

- **Raised by:** Reviewer A (F5)
- **Finding:** M5 cycle-2 correctly gates `maybeInjectRecall` by `channelType !== 'agent'`. But §Detailed Design 3 classifier loop's archive.db read doesn't apply the same filter. `delivery.ts` archives a2a assistant replies with `channel_type='agent'`. Classifier will pair them as "assistant runs" alongside real chat, generating false facts.
- **Effort to Fix:** Low (one WHERE clause)
- **Impact if Ignored:** High — a2a internal traffic gets classified into both stores, generating false facts
- **Suggested resolution:** Add `WHERE channel_type != 'agent'` to classifier's archive query in §Detailed Design 3. Add to test plan.
- **Status:** Open — mechanical

### [M6] Scan cursor + retry queue creates unbounded re-work

- **Raised by:** Reviewer A (F6), Reviewer C (#4) — convergence
- **Finding:** §Detailed Design 3 retry pseudocode: when item is in `dead_letters` not yet due for retry, "continues without advancing scan_cursor." But scan_cursor is also the read pointer for archive.db. A single stuck retry-pending item at sent_at=T causes every subsequent sweep to re-read everything from T onward across every group. Repeated archive scans every 60s.
- **Effort to Fix:** Low (decouple retry from scan)
- **Impact if Ignored:** High — performance regression under stuck retries; defeats the watermark/scan-cursor split's intent
- **Suggested resolution:** scan_cursor advances past ALL inspected items including pending-retries. Pull retries from `dead_letters` directly via `idx_dead_letters_retry` index where `next_retry_at <= now`. Update §Detailed Design 3 watermark/scan-cursor model to reflect three concepts: scan_cursor (advances past everything inspected), success_watermark (advances only on success), retry_queue (separate index on dead_letters).
- **Status:** Open — mechanical

### [M7] Host needs Anthropic SDK; not in package.json — supply-chain + auth path unspecified

- **Raised by:** Reviewer A (F7), partial F15
- **Finding:** §Detailed Design 3 classifier loop calls Haiku from the host daemon. Host `package.json` has no `@anthropic-ai/sdk` dependency. Adding it requires supply-chain ack (`minimumReleaseAge: 4320`). Also: design doesn't specify HOW host calls Haiku — direct SDK, OneCLI gateway proxy + native fetch, or shelling to container. Each has different supply-chain implications. C32 says "Anthropic SDK pinning preserved" but only addresses container.
- **Effort to Fix:** Low (decision + spec)
- **Impact if Ignored:** High — daemon can't be built; auth path undefined
- **Suggested resolution:** Decide between:
  - (a) `@anthropic-ai/sdk@<version>` added to host `package.json`, with `minimumReleaseAgeExclude` operator-approved for the pinned version
  - (b) Native `fetch` against OneCLI gateway (existing host capability) — no new SDK dep, leverages existing credential proxy
  Recommend **(b)** — reuses existing infra. Document the auth path.
- **Status:** Open — mechanical (but requires explicit choice)

### [M8] `getAgentGroupFolder` referenced in M8 attachment-downloader patch — doesn't exist

- **Raised by:** Reviewer A (F8)
- **Finding:** §Detailed Design 4 Surface (d) hook code uses `const groupFolder = getAgentGroupFolder(agentGroupId);`. This function doesn't exist. Actual lookup is `getAgentGroup(agentGroupId)?.folder` from `src/db/agent-groups.ts`. Also: `persistInboundAttachments` writes to **session attachments dir** (`sessionDir(agentGroupId, sessionId)/attachments/<msgId>/`), not the group folder — separate paths.
- **Effort to Fix:** Low
- **Impact if Ignored:** High — builder will get `getAgentGroupFolder is not defined` and guess; same retro-#2 root cause
- **Suggested resolution:** Replace `getAgentGroupFolder(agentGroupId)` with `getAgentGroup(agentGroupId)?.folder` (with null-check). Import from `./db/agent-groups.js`. Verify `GROUPS_DIR` from `./config.js`.
- **Status:** Open — mechanical

### [M9] systemd unit `node dist/scripts/memory-daemon.js` won't exist with current tsconfig

- **Raised by:** Reviewer A (F9)
- **Finding:** §Detailed Design 3 systemd unit: `ExecStart=/usr/bin/env node dist/scripts/memory-daemon.js`. `tsconfig.json` has `"rootDir": "./src"` and `"include": ["src/**/*"]` — `pnpm run build` (just `tsc`) will not compile `scripts/memory-daemon.ts` to `dist/scripts/`. S6 cycle-2 resolution claim "Build produces this artefact" is unsupported.
- **Effort to Fix:** Low (move daemon entry under src/ OR adjust tsconfig)
- **Impact if Ignored:** High — systemd unit fails at startup; daemon can't run
- **Suggested resolution:** Move daemon entry to `src/memory-daemon/index.ts`; ExecStart becomes `node dist/memory-daemon/index.js`. Mirror the existing `nanoclaw-v2.service`'s additional fields: `WorkingDirectory`, `StandardOutput`, `StandardError`, `Environment=PATH`.
- **Status:** Open — mechanical

## SHOULD-FIX Findings

### [S1] `Requires=ollama.service` but only `After=nanoclaw-v2.service` — first-boot race

- **Raised by:** Reviewer A (F10)
- **Finding:** Memory-daemon depends on archive.db (created by host on init). Currently starts in parallel with host on boot, may hit missing `data/archive.db`.
- **Suggested resolution:** Add `Requires=nanoclaw-v2.service`, OR document defensive open behavior (skip sweeps when archive.db missing).
- **Status:** Open

### [S2] `maybeInjectRecall` couples every `writeSessionMessage` caller to memory module

- **Raised by:** Reviewer A (F13)
- **Finding:** Every host writer (approvals, scheduling, self-mod, a2a) now executes `memoryEnabledForGroup` check. Hidden coupling.
- **Suggested resolution:** Cache per-group memory-enabled boolean in-process with 60s TTL. Document the cache invariant.
- **Status:** Open

### [S3] enable-memory.ts dirs vs existing illysium dirs naming mismatch + no backfill

- **Raised by:** Reviewer A (F14)
- **Finding:** New layout `inbox,articles,docs,transcripts,clips,media,processed/` vs existing `articles,docs,threads/`. `transcripts` ≠ `threads`. No backfill mechanism for existing files.
- **Suggested resolution:** Reuse existing dirs OR document rename + one-shot migration. Make first sweep optionally backfill from `articles/`, `docs/`, `threads/` (operator flag).
- **Status:** Open

### [S4] No semantic retrieval evaluation harness (Phase-2)

- **Raised by:** Reviewer B (LOW-MEDIUM drift)
- **Finding:** Production memory systems benefit from a small curated benchmark of (query → expected-fact-id) pairs to track recall precision over time. Design tracks distribution metrics but no labeled regression harness.
- **Suggested resolution:** Add to `decisions.yaml` as a Phase-2 deferral; build a 20-50 query golden set after 2-4 weeks of production data accumulates.
- **Status:** Open — Phase-2 deferral

### [S5] No decay / TTL model

- **Raised by:** Reviewer B (LOW drift)
- **Finding:** Aura uses 0.5%/day decay → 50% after 138 days. Brief and design don't surface a decay model.
- **Suggested resolution:** Phase-2 deferral; mnemon may already implement decay internally — verify.
- **Status:** Open — Phase-2 deferral

### [S6] Conflict resolution / supersession is interface-only in v1

- **Raised by:** Reviewer B (LOW drift)
- **Finding:** `supersedes` and `validTo` fields exposed in interface but v1 mnemon-impl ignores them. Mem0's ADD/UPDATE/DELETE/NOOP function-call pattern not implemented.
- **Suggested resolution:** Document v1 limitations explicitly; Phase-2 implement consolidation pass.
- **Status:** Open — Phase-2 deferral

### [S7] Container can write source files via tool-output capture — memory-poisoning path

- **Raised by:** Reviewer C (#5)
- **Finding:** Tool-output capture lets agent (via the tool result middleware) decide which content lands in `sources/inbox/`. Even though the classifier gates writes, the agent can choose what to "show" the classifier — adversarial or careless agents could poison memory.
- **Suggested resolution:** Acknowledge the trust boundary explicitly: agent-controlled tool execution → writes to inbox → classifier decides. Add to assumptions log: "tool-output capture trusts the agent's tool-fetching judgment to the extent the classifier and secret-redactor scrubber catch poisoning attempts." Phase-2: signed manifests for capture provenance.
- **Status:** Open
