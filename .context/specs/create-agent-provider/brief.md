# Brief: `create_agent` provider + per-provider config

> Generated by `/team-brief` — 2026-04-24
> Saved to: `.context/specs/create-agent-provider/brief.md`

## Requirements

- Extend the `create_agent` MCP tool (`container/agent-runner/src/mcp-tools/agents.ts`) so an admin agent can choose, at spawn time, which provider the new sub-agent will run on.
- Accept an optional `provider` string parameter that selects the registered agent provider (e.g. `"claude"`, `"codex"`).
- Accept an optional `provider_config` object whose schema is **provider-specific** — each provider owns the set of keys it accepts. Initial schemas:
  - **Claude:** `{ model?: string, effort?: string }`
  - **Codex:** `{ model?: string, reasoning_effort?: string }`
- Validate `provider` against the host's registered providers (`getProviderContainerConfig`) **before** writing any persistent state. If the provider is not registered, return `isError` with a message that names the missing skill (e.g. `"Provider 'codex' is not registered. Run /add-codex on the host first."`).
- Validate `provider_config` against the chosen provider's schema before any persistent state is written. Unknown keys, wrong types, or invalid enum values → `isError`, no DB write.
- Persist the selected provider on the new `agent_groups` row (set `agent_groups.agent_provider` instead of the current hardcoded `null`).
- Persist the chosen `provider_config` so it survives container restarts and overrides any host-wide `.env` defaults. The new agent group's effective `model` / `effort` / `reasoning_effort` must match what was passed to `create_agent` on every wake, not just the first.
- When `provider` is omitted, behavior must be identical to today's `create_agent` (defaults to the host's default provider — currently Claude — and writes `agent_provider: null` for back-compat with existing rows).
- The implementation must land on a feature branch (`feat/create-agent-provider`) suitable for opening a PR against `qwibitai/nanoclaw` upstream — minimal divergence from current file/module conventions, no parallel registries.

## Constraints

- [HARD] Validation must occur before any persistent state is created. No orphan `agent_groups` row, no `agent_destinations` rows, no `groups/<folder>/` directory may exist after a failed `create_agent` call.
- [HARD] Tool remains admin-only. Existing access gating in `agents.ts` must not be weakened.
- [HARD] Existing `create_agent` callers (no `provider`, no `provider_config`) must keep their current behavior — back-compat is non-negotiable.
- [HARD] Per-provider `provider_config` schemas live alongside each provider in `src/providers/<name>.ts` (or co-located there). A single provider must be addable in one PR, one file diff. No central enum / parallel registry of "all valid keys for all providers."
- [HARD] Initial scope is Claude + Codex provider_config schemas. OpenCode (and any future providers) accept `provider` but their `provider_config` schema is empty in this PR (any non-empty `provider_config` for them → validation error). Each is a one-file follow-up.
- [HARD] Fire-and-forget semantics from the agent's POV are preserved. The MCP `handler` returns synchronously; the host writes a follow-up `notifyAgent` system message on validation failure or on successful creation.
- [HARD] Supply-chain rules apply (`minimumReleaseAge: 4320`, `frozen-lockfile` in CI, `onlyBuiltDependencies` allowlist). No new package dependencies expected; if any are needed, they must be flagged for human approval.
- [SOFT] Codebase conventions: the module is registry-based (`src/modules/agent-to-agent/index.ts`). New code should fit that pattern. New tests should match existing test framework (`vitest` host-side, `bun:test` container-side).
- [SOFT] Per-provider config persistence mechanism (env vars in `container_config.env` vs. a dedicated `agent_groups.provider_config` JSON column vs. provider-specific files like `~/.codex/config.toml`) is design's call, but whichever mechanism is chosen must satisfy the HARD persistence constraint above.

## Non-Goals (Explicitly Out of Scope)

- Installing providers from the MCP tool. `/add-codex`, `/add-opencode`, etc. remain user-run host-side skills. The MCP only consumes what's already installed.
- An `update_agent_provider` MCP tool to retrofit the provider/config of existing agent groups. Out of scope; future work.
- Exposing OpenCode `provider_config` keys (`model`, `small_model`, `provider`, etc.) in this iteration. OpenCode accepts `provider: "opencode"` but with empty `provider_config`. Follow-up PR.
- Per-session or per-turn overrides of provider/model/effort from the MCP tool. The data model already supports `sessions.agent_provider`, but this brief only addresses the per-agent-group spawn-time path.
- Exposing Claude SDK options beyond `model` and `effort` (e.g. `permissionMode`, `disallowedTools`, `systemPrompt`). These are intentionally fixed by the host as security/UX baselines.
- Managing OneCLI credential allow-lists for the new agent. The existing OneCLI flow ([CLAUDE.md "Gotcha: auto-created agents start in `selective` secret mode"](../../CLAUDE.md)) still applies and is the user's responsibility.
- A UI / dashboard for managing agent groups. Chat-only.

## Key Concepts

- **Agent provider:** Backend that drives an agent's reasoning loop inside the container. Selected by the `AGENT_PROVIDER` env var. Resolution precedence (existing): `sessions.agent_provider → agent_groups.agent_provider → container.json provider → 'claude'`. Implementation in `src/container-runner.ts::resolveProviderName`.
- **Provider registry:** Host-side `src/providers/<name>.ts` files self-register via `provider-container-registry.ts`. `getProviderContainerConfig(name)` is the lookup.
- **`provider_config`:** New optional object on `create_agent` whose **shape is defined by the chosen provider**. Each provider declares its own schema and validator. Conceptually parallel to how the existing `container_config.env` overrides work, but typed and exposed in chat.
- **`agent_destinations`:** Existing ACL table. The bidirectional rows created by `create_agent` (parent→child and child→parent) are unchanged by this work — provider choice is orthogonal to addressing.
- **Fire-and-forget:** Existing pattern. `create_agent` MCP returns synchronously with a "creating…" message; success or failure surfaces as a follow-up system message via `notifyAgent`. Validation errors must arrive in chat, not just in host logs.
- **Admin-only:** Existing constraint. Non-admin containers do not see `create_agent` in their tool list. This work does not change the gating.

## Open Questions (if any remain)

None. All scope and surface decisions are settled above. Implementation-level decisions (where exactly `provider_config` persists, how each schema is registered, whether validation lives in `agents.ts` or `create-agent.ts`) are deferred to `/team-design`.

## Success Criteria

1. From an admin Claude agent, calling `create_agent({ name: "Coder", instructions: "...", provider: "codex", provider_config: { model: "gpt-5.5", reasoning_effort: "high" } })` results in a new agent group whose container, on first wake and on every subsequent wake, runs Codex with model `gpt-5.5` and reasoning effort `high`.
2. From an admin Claude agent, calling `create_agent({ name: "Reviewer", instructions: "...", provider: "claude", provider_config: { model: "claude-opus-4-7", effort: "high" } })` results in a new Claude agent whose model + effort are sticky across container restarts.
3. Calling `create_agent({ name: "X", provider: "codex" })` on a host where `/add-codex` has not been run returns an `isError` content block in chat naming the missing skill, AND no row exists in `agent_groups`, AND no row exists in `agent_destinations`, AND no `groups/X/` directory has been created.
4. Calling `create_agent({ name: "X", provider: "codex", provider_config: { effort: "high" } })` (wrong key — Codex uses `reasoning_effort`, not `effort`) returns an `isError` content block in chat naming the unknown key, with the same orphan-state guarantees as (3).
5. Calling `create_agent({ name: "X", instructions: "..." })` with no `provider` and no `provider_config` produces a Claude agent identical to today's behavior. Existing tests for the current path still pass with no modification.
6. The diff is contained in a feature branch `feat/create-agent-provider`, builds cleanly (`pnpm run build` host, `pnpm exec tsc -p container/agent-runner/tsconfig.json --noEmit` container), passes both test suites, and is ready to open as a PR against `qwibitai/nanoclaw`.
