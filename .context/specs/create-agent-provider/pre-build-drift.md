## Blocking Mismatches

### [B1] Step-2 (`updateContainerConfig`) failure handling: plan is tighter than design

- **Claim:** Design's D10 ordering spec says `updateContainerConfig` (step 2 of the sequenced write): *"On failure: propagate; no DB row references this folder yet. Caller may want to clean up the folder from step 1, but even if they don't, there's no DB pointer to a dangling config."*
- **Class:** DIVERGED
- **SOT says:** On step-2 failure, propagate the exception. Caller may clean up the folder or not.
- **Target says:** Task B2 approach wraps step 2 in `try/catch`: on failure, call `safeRemoveFolder(newGroup.folder)`, call `notifyAgent(session, "create_agent failed: could not write config for \"${name}\".")`, and `return` (no exception propagated).
- **Gap:** The plan extends the design's design-level invariant. Where the design tolerates "may or may not clean up," the plan mandates cleanup; where the design propagates, the plan notifies and returns.
- **Found by:** Agent B (Codex, Claim #47).
- **Status:** **ACKNOWLEDGED as intentional tightening** — see `drift-acks.json`. Rationale: review finding S14 required full-folder rollback on failure to prevent user-supplied prompt content (in `CLAUDE.local.md`) from persisting on disk. The design's own S14 response mandated this for step-3 failures; extending the same invariant to step-2 failures is the consistent application of S14's intent. Propagating to the delivery loop without notifying the agent also produces worse UX than the plan's explicit `notifyAgent` error.
