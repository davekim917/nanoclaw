### Task B2: CORS middleware and body parser utility

**File:** `src/api/cors.ts` (CREATE)

**What to build:**
CORS middleware function that runs on every request before route matching. Behavior per design spec (section A.5): when `WEB_UI_ORIGINS` is empty, no CORS headers sent. When set, respond to `OPTIONS` preflight and add headers to matching-origin requests. Non-matching origins get no headers.

Also create a shared `parseJsonBody(req, maxBytes)` utility that replaces the duplicated body parsing in `handleIntervene` and `handleSend` (both have identical 65KB limit, abort-on-error patterns). This utility returns a `Promise<T>` and rejects on errors.

**Interface:**
```typescript
import { IncomingMessage, ServerResponse } from 'http';

/**
 * Apply CORS headers if the request's Origin matches WEB_UI_ORIGINS.
 * Returns true if this was an OPTIONS preflight (response already sent).
 * Returns false if request should continue to route matching.
 */
export function handleCors(req: IncomingMessage, res: ServerResponse): boolean;

/**
 * Parse JSON body from request. Rejects if body exceeds maxBytes or is invalid JSON.
 */
export function parseJsonBody<T = unknown>(
  req: IncomingMessage,
  maxBytes?: number,  // default 65536
): Promise<T>;
```

**ASSERT:**
- `handleCors` reads `WEB_UI_ORIGINS` from `config.ts` (import at top, not re-read each time).

- `Access-Control-Allow-Origin` is the specific matched origin, never `*`.
- Same-origin requests (no `Origin` header) skip CORS entirely.
- `parseJsonBody` rejects with a typed error that includes an HTTP status code.

### Task B3: WebSocket handler

**File:** `src/api/ws.ts` (CREATE), `src/web-ui.ts` (MODIFY)

**What to build:**
WebSocket upgrade handler using the `ws` library in `noServer` mode. Attached to the HTTP server's `upgrade` event. Auth via `?token=` query param at upgrade time (reuse `checkAuth` from `web-ui.ts`). Origin validation against `WEB_UI_ORIGINS` (reject with 403 if mismatch; allow if origins unset or origin matches or no Origin header in bundled mode).

Implements the full WS protocol from the design (section A.4):
- On successful upgrade: send `{ type: 'connected', capabilities }`.
- Handle `send_message`: validate fields, apply trigger prefix injection (same logic as `startSession` in `index.ts:2056-2059`), store message, enqueue processing, respond with `message_stored`.
- Handle `subscribe`: update per-connection group filter set.
- Rate limit: 10 `send_message` per minute per connection. Excess gets `error` with `code: 'rate_limited'`.
- Backpressure: drop `progress` events when `ws.bufferedAmount > 1MB`, preserve lifecycle events.
- Event ring buffer: last 500 events for reconnection replay.

Modify `web-ui.ts` to call an `initWebSocket(server, deps, token)` function from `ws.ts` during startup. The broadcast/notify functions in `web-ui.ts` must also forward events to WebSocket clients (in addition to SSE clients).

**Interface:**
```typescript
import { Server } from 'http';
import type { Capabilities } from './types.js';

export interface WsDeps {
  checkAuth: (req: IncomingMessage, token: string) => boolean;
  getCapabilities: () => Capabilities;
  startSession: (groupJid: string, text: string, senderName: string, senderId: string) => boolean;
  broadcastToSse: (sessionKey: string, group: string, threadId: string | undefined, event: ProgressEvent) => void;
}

export function initWebSocket(
  server: Server,
  deps: WsDeps,
  token: string,
): {
  broadcastWs: (sessionKey: string, group: string, threadId: string | undefined, event: unknown) => void;
  notifyWsSessionStart: (sessionKey: string, group: string, groupJid: string, threadId?: string) => void;
  notifyWsSessionEnd: (sessionKey: string) => void;
  notifyWsSkillInstall: (jobId: string, output: string, status: string) => void;
};
```

**ASSERT:**
- Uses `ws` in `noServer` mode (no separate HTTP server).
- Auth validated BEFORE WebSocket handshake completes (HTTP 401 for bad token).
- Origin validated BEFORE handshake (HTTP 403 for mismatched origin).
- `send_message` applies trigger prefix injection matching `index.ts:2056-2059` logic.
- Rate limit is per-connection, resets every 60 seconds.
- Ring buffer is in-memory, max 500 entries, shared across all connections.
- `subscribe` replaces (not merges) the previous subscription set.
