## Brief Reference

Approved brief: `.context/specs/nanoclaw-web-ui/brief.md` (2026-03-20)

**Summary:** Build a standalone web UI for NanoClaw with chat, conversation history, skills marketplace (skills.sh), and workflow management (Town.com-inspired). Connects to any NanoClaw instance via WebSocket/REST API. Optional bundled mode. Capability discovery for graceful degradation. Dark mode default. Multi-user data model from day one.

### Flagged Constraints

- **C4 (multi-user data model):** The current schema has no `user_id` on any table. Adding `user_id` to new tables is straightforward, but retrofitting existing tables (messages, sessions, tasks) would be a breaking migration. Recommend: new tables get `user_id`; existing tables get `user_id` via optional column additions (nullable, default null) in Phase 2 when multi-user auth ships. This satisfies the constraint's intent without breaking the current schema.
- **C12 (no new deps):** Node.js has no built-in WebSocket server. Options: (a) use the `ws` npm package (~3.5KB, standard, battle-tested), (b) implement WebSocket handshake manually over `http`. Recommend `ws` — it's the standard Node.js WebSocket library and is worth the dependency.

## Research Summary

### Project patterns (CLAUDE.md, codebase)

- **Single process architecture:** NanoClaw is a single Node.js process. Channels self-register at startup via `registerChannel()` (`src/channels/registry.ts:19`). All coordination happens in-process.
- **Existing web server:** `src/web-ui.ts` runs an HTTP server on port 3002 using Node.js built-in `http`. Has bearer token auth, SSE streaming, and 4 REST endpoints (`/events`, `/api/intervene`, `/api/groups`, `/api/send`, `/api/sessions`). Binds to `0.0.0.0` when token is set, `127.0.0.1` otherwise.
- **Message injection pattern:** `web-ui.ts:2054-2070` shows how to inject a web message into the system: store in DB via `storeMessage()`, then `queue.enqueueMessageCheck(groupJid)` to trigger processing. The agent processes it through the normal message loop.
- **IPC command set:** 30+ commands in `src/ipc.ts` (line 302-1397). Two patterns: fire-and-forget tasks (write JSON to `data/ipc/{group}/tasks/`) and request-response queries (write to `queries/`, read from `responses/`). All commands return structured JSON.
- **Progress streaming:** `web-ui.ts:53-75` broadcasts `ProgressEvent` objects via SSE to connected clients. Events include tool calls, text output, and session lifecycle.
- **Database schema:** `src/db.ts:23-289` defines 14 tables. Key ones for the UI: `messages`, `sessions_v2`, `scheduled_tasks`, `task_run_logs`, `memories`, `backlog`, `ship_log`, `thread_search` (FTS5), `thread_metadata`, `registered_groups`.

### Library research

- **Lit 3.3.x:** Actively maintained (OpenJS Foundation + Google). ~5KB gzipped. Web components standard. Reactive properties, reactive controllers, decorators. Source: lit.dev, npm.
- **@vaadin/router:** Deprecated — Vaadin moved to React Router. Do not use for new projects. Source: GitHub issues, npm activity.
- **URLPattern API:** Browser-native (Baseline 2025). Lightweight routing possible without dependencies. Source: MDN, web.dev.
- **@lit-labs/router:** Pre-1.0, labs status. Functional but has known gaps (no imperative `Router.go()`). Acceptable for simple routing. Source: npm, Lit GitHub discussions.
- **@lit-labs/signals:** TC39 Signals integration for Lit. Fine-grained cross-component reactivity. Source: npm, Lit docs.
- **ws npm package:** Standard Node.js WebSocket library. 3.5KB, zero deps, 50M+ weekly downloads. Used by Socket.IO, Express-WS, etc. Source: npm.
- **skills.sh API:** Public endpoint at `https://skills.sh/api/skills` returns skill metadata sorted by install count. CLI search tool available. Source: skills.sh/docs, npm `skills` package.

### Existing code read

- `src/web-ui.ts` (full file) — HTTP server, auth, SSE, REST endpoints. 302 lines.
- `src/ipc.ts:57-69` — `IpcDeps` interface: the dependencies needed to process commands.
- `src/ipc.ts:270-299` — Task/query data structures passed to `processTaskIpc` and `processQueryIpc`.
- `src/ipc.ts:953-1032` — `processQueryIpc` function: dispatches query commands and writes JSON responses.
- `src/db.ts:23-289` — Full schema creation. 14 tables with indexes and migrations.
- `src/config.ts:188-191` — `WEB_UI_PORT` (3002) and `WEB_UI_TOKEN` config.
- `src/index.ts:2042-2074` — How `startWebUI` is called with deps for message injection.
- `src/types.ts:31-170` — All TypeScript interfaces for domain objects.
- `src/channels/registry.ts` (full file) — Channel factory pattern, 30 lines.
- `src/router.ts` (full file) — Message formatting, XML envelope, `<internal>` stripping, channel routing.

## Design Decisions

This design covers two workstreams: (A) NanoClaw-side API gateway, and (B) standalone SPA.

---

## A. NanoClaw API Gateway

### Option A: Evolve web-ui.ts into API Gateway

**Approach:** Expand the existing `web-ui.ts` HTTP server to serve as a full API gateway. Add WebSocket support (via `ws` package) for real-time chat streaming. Add REST endpoints that wrap the existing IPC command logic. Add `/api/capabilities` for feature discovery. Optionally serve SPA static assets for bundled mode. Rename to `api-gateway.ts` and split route handlers into a `api/` directory.

**Pros:**
- Minimal change to NanoClaw — evolves existing server, single port
- Reuses existing auth pattern (Bearer token + query param)
- Already integrated into the startup sequence (`index.ts:2042`)
- SSE activity monitor stays backwards-compatible
- One port = simpler deployment, no CORS in bundled mode

**Cons:**
- `web-ui.ts` was designed as a simple monitor — significant expansion needed
- Adds `ws` dependency to NanoClaw
- File grows large → needs to be split into modules

**Pattern alignment:** Follows NanoClaw's single-process architecture (CLAUDE.md: "Single Node.js process"). Extends existing HTTP server pattern rather than adding a new one.

**Confidence:** High — all assumptions validated by reading existing code. The `startSession` pattern (`index.ts:2054-2070`) proves message injection works. SSE broadcast pattern (`web-ui.ts:53-75`) proves real-time streaming works.

**Assumptions:**
- `ws` WebSocket upgrade can coexist with the existing `http.createServer` on the same port
- Progress events currently broadcast to SSE clients can also be forwarded to WebSocket clients

### Option B: Register "Web" as a Channel

**Approach:** Create a `web.ts` channel that registers via `registerChannel()`, treating the browser like Discord or Slack. The channel manages WebSocket connections and maps messages to/from the NanoClaw message format. A separate HTTP server handles REST endpoints for CRUD operations (tasks, memories, etc.).

**Pros:**
- Follows NanoClaw's channel abstraction pattern cleanly for chat
- Chat routing reuses the existing channel infrastructure
- Web channel can participate in the normal message loop

**Cons:**
- Channel interface (`types.ts:178-220`) is designed for messaging platforms — methods like `ownsJid()`, `setTyping()`, `addReaction()` don't map cleanly to a web UI
- CRUD operations (list tasks, search memories, browse skills) don't fit the channel abstraction — still need a separate REST server
- Two integration points: channel for chat + REST server for everything else = more complexity
- Would need a separate HTTP server for REST endpoints (channel interface doesn't handle HTTP)

**Pattern alignment:** Partially aligned — chat routing fits the channel pattern, but admin/query operations don't. Would stretch the channel abstraction beyond its design intent.

**Confidence:** Medium — chat part is clean, but the REST/query part requires a second integration pattern. The channel interface would need methods it wasn't designed for.

**Assumptions:**
- Channel interface can be extended with WebSocket transport without breaking other channels
- REST operations can coexist with channel-based chat routing

### Option C: IPC Proxy (File-based)

**Approach:** Create a lightweight HTTP/WS server that translates web API calls into file-based IPC operations — writing JSON files to the `data/ipc/` directories and reading responses. Zero changes to NanoClaw's internal processing.

**Pros:**
- No changes to NanoClaw's IPC processing code
- Complete decoupling — proxy can run as a separate process

**Cons:**
- File-based IPC has polling latency (~1s `IPC_POLL_INTERVAL`) — unacceptable for interactive chat
- IPC was designed for container→host communication with directory-based identity — web clients don't have IPC directories
- Fragile — depends on file system timing and cleanup
- No streaming — IPC responses are complete JSON files, not streams

**Pattern alignment:** Reuses IPC infrastructure but in a way it wasn't designed for. The IPC system assumes container identity from directory paths — web clients would need a synthetic identity.

**Confidence:** Low — file-based IPC fundamentally can't support real-time streaming or low-latency interactive chat.

**Assumptions:**
- File-based IPC could be made fast enough for interactive use (unlikely)
- A synthetic IPC identity could be created for web clients

### Recommendation: Option A — Evolve web-ui.ts into API Gateway

**Justification:** Option A is the minimal-change, high-confidence approach. The existing `web-ui.ts` already has the foundation — HTTP server, auth, SSE streaming, message injection, group listing. Expanding it with WebSocket + REST is a natural evolution, not a rewrite. Option B forces the channel abstraction to serve a purpose it wasn't designed for (CRUD operations). Option C can't meet real-time requirements.

The single-port, single-process pattern aligns with both NanoClaw's architecture (CLAUDE.md) and OpenClaw's proven approach.

**NanoClaw-side changes (summary):**

1. **Evolve `web-ui.ts` in place** — extract new route handlers into `src/api/` directory. #### Skills Install Security

`POST /api/skills/install` executes `npx skills add <repo>` on the NanoClaw host. This is a **privileged operation** that breaks the container isolation boundary.

**Required safeguards:**
- **Input validation:** `repo` parameter must match strict pattern: `^[a-zA-Z0-9_-]+/[a-zA-Z0-9_.-]+$` (GitHub `owner/repo` format only). Reject all other input with 400.
- **Execution:** Use `execFile('npx', ['skills', 'add', repo])` with argument array — never shell interpolation or `exec()`/`execSync()`.
- **Async model:** Return `{ status: 'installing', jobId: '<uuid>' }` immediately. Run install in a child process. Stream stdout/stderr via WebSocket progress events with `type: 'skill_install_progress'`. `GET /api/skills/install/:jobId` returns status (pending/running/completed/failed).
- **Timeout:** 120s max. Kill child process if exceeded.
- **Output cap:** Capture max 1MB of stdout/stderr.
- **Restart notification:** Response includes `requires_restart: true`. SPA displays prominent "Restart NanoClaw to apply changes" notification. Consider adding `POST /api/restart` (Phase 2).
- **Authorization:** Phase 1 — bearer token auth (admin-level). Phase 2 — restrict to admin role only.

4. **Add WebSocket upgrade handler** on same HTTP server. **Authentication happens at upgrade time** — matching the existing SSE `?token=` pattern. Unauthenticated upgrade requests receive HTTP 401 and are never promoted to WebSocket.

```typescript
// WebSocket message protocol
type WsClientMessage =
  | { type: 'send_message'; groupJid: string; text: string; senderName?: string; senderId?: string; threadId?: string }
  | { type: 'subscribe'; groups?: string[] }  // filter events to specific groups

type WsServerMessage =
  | { type: 'connected'; capabilities: Capabilities }  // sent immediately after successful upgrade
  | { type: 'progress'; sessionKey: string; group: string; event: ProgressEvent }
  | { type: 'session_start'; sessionKey: string; group: string; groupJid: string }
  | { type: 'session_end'; sessionKey: string }
  | { type: 'message_stored'; id: string }  // ack for send_message
  | { type: 'skill_install_progress'; jobId: string; output: string; status: string }
  | { type: 'error'; code: string; message: string }
```

**WebSocket behavioral spec:**
- **Auth:** Token validated during HTTP upgrade via `?token=<value>` query param. No in-band auth message. On auth failure, server responds with HTTP 401 before WebSocket handshake completes. **Origin validation:** During the `upgrade` event, validate the `Origin` header against `WEB_UI_ORIGINS` (same allowlist as CORS). Reject mismatches with 403 to prevent cross-site WebSocket hijacking. In bundled mode (no `WEB_UI_ORIGINS` set), only allow same-origin or missing `Origin` header.
- **Connected:** Immediately after successful upgrade, server sends `{ type: 'connected', capabilities }`.
- **send_message:** Server applies trigger prefix injection (same logic as `startSession` in `index.ts:2056-2059` — prepends `@{assistantName} ` when `group.requiresTrigger !== false`). `senderName` defaults to a configurable value (env var `WEB_UI_SENDER_NAME`, default: 'Web User') for Phase 1; Phase 2 uses authenticated user identity. `senderId` defaults to `'web-ui-<token-hash-prefix>'`.
- **subscribe:** Replaces previous subscription (not additive). Server maintains `Set<string>` of subscribed groups per connection. Default when no `subscribe` sent: receive all events (matching current SSE behavior). New groups registered after subscription require a fresh `subscribe` to opt in.
- **Rate limiting:** Max 10 `send_message` per minute per connection. Excess messages receive `{ type: 'error', code: 'rate_limited', message: '...' }`.
- **Reconnection (client-side):** Exponential backoff with jitter (1s, 2s, 4s, 8s, max 30s). On reconnect, client re-sends `subscribe` with last-seen event timestamp. Server replays missed events from an in-memory ring buffer (last 500 events, ~60s retention). If gap exceeds buffer, server sends `{ type: 'resync' }` and client fetches full state via REST.
- **Backpressure:** If `ws.bufferedAmount` exceeds 1MB, server drops non-critical events (progress) but preserves session lifecycle events (start/end) and message acks.

5. **Add CORS middleware** — configurable `WEB_UI_ORIGINS` env var (comma-separated allow-list). Added to `config.ts`'s `readEnvFile()` call.

**CORS behavior:**
- **When `WEB_UI_ORIGINS` is unset:** No CORS headers sent. Cross-origin requests are rejected by the browser. Only same-origin (bundled mode) works.
- **When set (e.g., `http://localhost:5173,https://ui.example.com`):** Middleware handles `OPTIONS` preflight requests, returning: `Access-Control-Allow-Origin: <matched origin>` (not `*`), `Access-Control-Allow-Methods: GET, POST, PATCH, OPTIONS`, `Access-Control-Allow-Headers: Authorization, Content-Type`, `Access-Control-Max-Age: 86400` (24h). Non-matching origins get no CORS headers.
- **Same-origin requests** (bundled mode, no `Origin` header): skip CORS entirely.
- Middleware runs before route matching on all requests.

6. **Add static file serving** for bundled mode — if `nanoclaw-ui` package is found (check `require.resolve('nanoclaw-ui/dist/index.html')` or a known path), serve its `dist/` contents on `/ui/*`. **SPA fallback routing:** All `/ui/*` paths that don't match a static file serve `dist/index.html` (required for client-side routing — e.g., `/ui/chat/group-123` must not 404). The `nanoclaw-ui` package must export built assets in its `dist/` directory with `index.html` as the entry point.

7. **Capability discovery** — `GET /api/capabilities`. Feature detection uses **meaningful availability checks**, not just table existence (since `createSchema()` creates all tables unconditionally):
```typescript
interface Capabilities {
  version: string;           // from package.json
  features: {
    memory: boolean;         // sqlite-vec extension loaded (vec_memories usable) AND memories table has >0 rows
    backlog: boolean;        // backlog table has >0 rows OR backlog IPC handler is registered
    ship_log: boolean;       // ship_log table has >0 rows OR ship_log IPC handler is registered
    thread_search: boolean;  // thread_search FTS5 table has >0 rows (indicates indexing is active)
    tone_profiles: boolean;  // tone-profiles/ directory exists AND contains >=1 .md file
    gate_protocol: boolean;  // pending_gates table has >0 rows (gate feature has been used)
    activity_summary: boolean; // daily-notifications system task handler is registered
    commit_digest: boolean;  // commit-digest system task handler is registered
    ollama: boolean;         // OLLAMA_HOST env var is set and non-empty
  };
  channels: string[];        // from channel registry
  groups: GroupInfo[];        // from registered groups
}
```
Note: Features with 0 rows but whose code is loaded (e.g., backlog module exists but never used) should still show as available — check handler registration as primary signal, row count as secondary.

## B. Standalone SPA

### Option A: Full SPA with Client-Side Routing

**Approach:** Single-page application with Vite + Lit 3.x. Client-side routing via URLPattern API (browser-native). All pages render in-browser. API communication via WebSocket (real-time chat) + REST (CRUD operations). Thin API client layer wraps both transports.

**Layout: Sidebar navigation** — persistent sidebar on the left with page links (Chat, Sessions, Skills, Workflows). Main content area on the right. Sidebar collapses to icons on mobile. Chat page is the default landing.

**Pros:**
- Standard SPA pattern — well-understood, easy to maintain
- URLPattern is browser-native (Baseline 2025) — no routing dependency
- Sidebar navigation gives clear page structure and quick switching
- All rendering happens client-side — NanoClaw serves static files only

**Cons:**
- Initial load fetches entire SPA bundle (mitigated by code splitting)
- Client-side routing requires fallback handling (all routes → index.html)

**Pattern alignment:** Matches OpenClaw's WebChat approach (Vite + Lit SPA). PinchChat also uses this pattern as a standalone client.

**Confidence:** High — standard SPA architecture with well-tested tools.

**Assumptions:**
- URLPattern API has sufficient browser support for target audience (self-hosted users → modern browsers)

### Option B: Tab-Based Single Page (No Routing)

**Approach:** Single page with tab navigation. All sections (Chat, Sessions, Skills, Workflows) are tab panels within one page. No URL routing — the current tab is stored in component state. Simpler architecture, no routing library needed.

**Layout: Top tab bar** — horizontal tabs across the top. Active tab's content renders below. Tabs stay in DOM (hidden) for instant switching.

**Pros:**
- Simpler — no routing, no URL management, no history API
- Instant tab switching (all panels stay in DOM)
- No fallback/redirect configuration needed

**Cons:**
- No deep linking — can't share a URL to a specific session or workflow
- Browser back/forward doesn't work across tabs
- All panels load upfront (no lazy loading)
- Doesn't feel like a "real" app — tabs feel like a widget, not a workspace

**Pattern alignment:** Simpler than OpenClaw's approach. More like the existing `web-ui.html` (single page with sections).

**Confidence:** High — technically simpler, but worse UX for a workspace product.

**Assumptions:**
- Users don't need deep linking or browser history navigation

---

### Recommendation: Option A — Full SPA with Client-Side Routing

**Justification:** Deep linking is essential for a workspace product — users need to share URLs to specific sessions, workflows, or skill pages. Browser back/forward navigation is expected. The overhead of client-side routing with URLPattern is minimal (no dependencies). Option B's simplicity comes at the cost of UX that doesn't meet the "rich experience" goal from the brief.

### Routing

URLPattern-based router with simple pattern matching:

```typescript
const routes = [
  { pattern: new URLPattern({ pathname: '/' }), page: 'chat' },
  { pattern: new URLPattern({ pathname: '/chat/:groupId?' }), page: 'chat' },
  { pattern: new URLPattern({ pathname: '/sessions' }), page: 'sessions' },
  { pattern: new URLPattern({ pathname: '/sessions/:key' }), page: 'sessions' },
  { pattern: new URLPattern({ pathname: '/skills' }), page: 'skills' },
  { pattern: new URLPattern({ pathname: '/workflows' }), page: 'workflows' },
  { pattern: new URLPattern({ pathname: '/workflows/:id' }), page: 'workflows' },
];
```

Navigation uses `history.pushState()` + `popstate` event listener. `<app-shell>` listens for route changes and swaps the active page component.

### State Management

Three layers (no pre-production dependencies):

1. **Local component state:** Lit `@state()` decorators for UI-only state (input values, dropdowns open, etc.)
2. **Shared state via EventTarget bus:** A thin `AppStore` class extending `EventTarget` for cross-component state. Components subscribe via `addEventListener` and unsubscribe in `disconnectedCallback`. This avoids depending on `@lit-labs/signals` (TC39 Signals is Stage 1 — pre-production, API may change). The store holds:
   - `authState` — token, connection status, capabilities
   - `activeGroup` — currently selected group
   - `chatMessages` — messages in current session
   - `streamingText` — live streaming output from agent
   - Getters return immutable snapshots; setters dispatch typed events.
3. **API client:** Thin wrapper that updates the store on response

### Auth Flow

1. On load, check `localStorage` for saved connection URL (token stored in `sessionStorage` for reduced XSS exposure)
2. If none, show `<auth-form>`: user enters NanoClaw URL + token
3. On submit: `GET /api/capabilities` with `Authorization: Bearer <token>`
4. If 200: save URL to `localStorage`, token to `sessionStorage`, open WebSocket, render app
5. If 401: show error, stay on auth form
6. WebSocket connects with `?token=<value>` — auth validated at HTTP upgrade time. On success, server immediately sends `{ type: 'connected', capabilities }`. On failure, server rejects upgrade with HTTP 401 (no WebSocket connection established).

**Security:** Token stored in `sessionStorage` (cleared on tab close) rather than `localStorage` to reduce XSS exposure window. The markdown renderer **must** use DOMPurify (or equivalent strict HTML sanitizer) — this is a hard security requirement, not optional. Agent output contains markdown, code blocks, and tool call results which are rendered as HTML; without sanitization, XSS would grant full admin access via the stored token.

### Chat Flow (End-to-End)

1. User selects group from `<group-picker>`
2. User types message in `<chat-input>`, hits send
3. UI sends via WebSocket: `{ type: 'send_message', groupJid: '...', text: '...', senderName: '...' }`
4. **Server-side processing:** Server applies trigger prefix injection (prepends `@{assistantName} ` if `group.requiresTrigger !== false`, matching `index.ts:2056-2059` logic), stores message in DB via `storeMessage()` with `sender_name` from the WS message (not hardcoded), enqueues processing via `queue.enqueueMessageCheck()`, responds with `message_stored`
5. UI optimistically renders user's message in chat
6. Server streams `progress` events as agent runs (tool calls, text chunks)
7. UI renders streaming text in real-time via `<streaming-text>` component
8. Server sends `session_end` when agent finishes
9. UI finalizes message, fetches full response from REST if needed
