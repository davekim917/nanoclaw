## SHOULD-FIX (4)

### S14 — D10 rollback-failure mode is underspecified

**Sources:** Reviewer A finding #1; Reviewer B cosmetic item #2 (corroborating).
**The issue:** D10 says "rollback by removing `groups/<folder>/` (or at minimum: delete `container.json`)" but doesn't say what happens when the rollback itself fails (`fs.rm` throws: permission, concurrent handle, etc.). In that case: no DB row, but a half-applied `container.json` and partial `CLAUDE.local.md`/skills symlinks persist. A later `create_agent` with the same name passes the DB-keyed collision check and gets a *different* folder — the first one becomes permanently orphan.

Also: the "delete just `container.json`" fallback wording is dangerous — it leaves `CLAUDE.local.md` with the agent's initial instructions on disk indefinitely (that file contains user-supplied prompt content).

Reviewer B corroborates: every saga/compensation pattern reference uniformly treats idempotency and rollback-failure handling as first-class. The design's guarantee ("degrades safely on any failure") overstates what's actually delivered.

**Resolution:**
- Remove the "or at minimum: delete container.json" fallback. Mandate full-folder `fs.rm`.
- Add one sentence to D10: "Rollback is best-effort; on failure, the orphan folder is logged at error severity for manual cleanup. It does not affect subsequent `create_agent` calls because folder naming is DB-deduplicated."
- (Optional, deferred) Add a sweep-time reconciler that GC's orphan folders with no matching DB row — tracks as R9.

**Effort:** Low (design wording) · **Impact:** Medium (observability + predictable failure mode) · **Confidence:** High.

### S15 — D11 "paired commit in same PR scope" conflates PR mechanics with commit mechanics

**Sources:** Reviewer A finding #2; Reviewer B cosmetic item #3 (corroborating).
**Verification:** A verified against actual upstream state — `upstream/providers` has 5 commits from `upstream/main` not yet merged (`a4346f5`, `1df8dec`, `82baa39`, `5845a5a`, `ce28e7f`). It's a long-lived sibling branch that periodically merges `main`, not a clean extension of it. Reviewer B confirms: GitHub Stacked PRs require same-repo + PR-to-PR relationship; they do not support "one PR touching two long-lived sibling branches."

**The issue:** GitHub PRs target a single base branch. The design's language "one coherent PR spanning two branches" is mechanically impossible — the implementer must open **two** PRs (trunk → `main`, codex schema → `providers`). Additionally, ordering matters: if the trunk PR merges first, Codex installs during the window get D8 strict-fallback rejections; if providers merges first, the schema sits unused until the registry lands. Neither is catastrophic but the design claims D11 "removes the window entirely" — it only removes the window if both land close together, which is not guaranteed by "paired PRs."

**Resolution:** Reword D11 and the Implementation Surface to use established vocabulary:
- "**Two coordinated linked PRs** (trunk → `main`, codex schema → `providers`), cross-referenced in PR descriptions, **merged in coordination** — ideally both land the same day. During any interim window where only one has merged, behavior is safe-but-incomplete: main-first rejects Codex config, providers-first has a schema nobody reads. Neither delivers SC #1 until both land."
- Matches Well-Architected polyrepo "linked PRs with enforced merge order" pattern (Reviewer B's T1 source).

**Effort:** Low (design wording) · **Impact:** Medium (avoids misleading the implementer about what GitHub will accept) · **Confidence:** High.

### S16 — D10 ordering has an implicit step-1 FS write that should be named

**Sources:** Reviewer A finding #3.
**The issue:** The design says "write `container.json` FIRST, before the DB insert" — but `initGroupFilesystem` (step 1) already calls `initContainerConfig` at `src/group-init.ts:169`, which writes an empty `container.json`. So the actual sequence is: step 1 creates a baseline config → step 2 mutates it → step 3 DB insert. An implementer reading "first" might be confused about which write is "first."

Also: step 1 creates more than `container.json` — it writes `CLAUDE.local.md`, `.claude-shared/settings.json`, skills symlinks. Rollback MUST remove the whole folder; anything less leaves prompt content on disk.

**Resolution:** Clarify the two-phase FS write in D10:
- "Step 1: `initGroupFilesystem` creates `groups/<folder>/` with a baseline (empty) `container.json`, `CLAUDE.local.md`, and skills symlinks.
- Step 2: `updateContainerConfig` mutates the existing `container.json` to set `provider` + `providerConfig`.
- Step 3: `createAgentGroup` DB INSERT.
- On step-3 failure: full-folder `fs.rm` (not just `container.json`); log on rollback-failure (see S14)."

**Effort:** Low (3-sentence rewrite) · **Impact:** Low-Medium (implementation clarity; combined with S14, closes the rollback story completely) · **Confidence:** High.

### S17 — Specify which test file gets the `test.todo('admin enforcement')` marker

**Sources:** Reviewer A finding #4.
**The issue:** The design calls for a `test.todo` marker but doesn't name the file. The new test surface spans `bun:test` (container-side validation) and `vitest` (host-side persistence). The admin gate gap is host-side (`handleCreateAgent` has no role check), so the marker belongs in the **vitest** file.

**Resolution:** One-word edit — specify "the host-side `vitest` file" for the S9 marker.

**Effort:** Low · **Impact:** Low (placement clarity) · **Confidence:** Medium.

**Source:** Reviewer A finding #6; Reviewer B cosmetic item #4.
The Zod enum hand-transcribes the SDK's `EffortLevel` literal list. A6 mitigates with a comment citing `sdk.d.ts:462`. Valuable but not required for this PR; file as a small follow-up or fold into the implementer's judgment. Also: A notes the implementer should anchor to the **exported** `EffortLevel` at `sdk.d.ts:462`, not the narrower internal `effortLevel` at `:4626` — add one word to D9 if writing the drift-detector test.

### W7 — S12 description/schema twin-source drift risk

**Source:** Reviewer A finding #5; Reviewer B note on S12.
The hand-written `description` string on `provider_config` and the Zod `configSchema` contain the same information. R6 already acknowledges migration to `z.toJSONSchema()` as future work; this is that gap manifesting in the interim. Acceptable given two MCP SDK issues (#1562 `$ref` emission, #702 transforms dropped) currently make `z.toJSONSchema()` risky for immediate adoption.
