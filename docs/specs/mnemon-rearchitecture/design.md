# Design: mnemon-rearchitecture

> Generated by `/team-design` — 2026-04-29

## Brief Reference

Implements the architecture described in `docs/specs/mnemon-rearchitecture/brief.md` — replicating Vivian Balakrishnan's NanoClaw v1 gist ([gist a7d4eec](https://gist.github.com/VivianBalakrishnan/a7d4eec3833baee4971a0ee54b08f322)). Three layers (raw sources → mnemon graph → wiki pages), three operations (extract, synthesise, recall), with auto-recall as system-reminder push and chat-stream + tool-output + manual-drop as the three ingestion sources. 31 HARD + 1 SOFT constraints, 11 Q&A decisions resolved.

This design replaces the failed PR #68 integration (`feat/mnemon-integration`, merged `681b143`, see `.context/retros/mnemon-integration/retro.md`).

## Research Summary

### Project patterns (from CLAUDE.md + project-scope.md)

- **Two-DB session split** (`docs/db.md`): host writes inbound.db, container writes outbound.db. Memory writes belong on the host side because they're host-driven (no agent involvement). Recall injection must respect this split — cannot have container writing memory facts back through outbound.db.
- **Provider abstraction** (`container/agent-runner/src/providers/`): trunk ships `claude` + `codex` + `mock`. AgentProvider interface (`types.ts:1-52`) has `query(input: QueryInput): AgentQuery`. QueryInput.systemContext.instructions is the per-session injection point. The provider abstraction is real and load-bearing — recall design must not break it.
- **MCP tool boundary** (CLAUDE.md § Skills): tool-output capture must respect that MCP tool definitions live in `container/agent-runner/src/mcp-tools/`; host doesn't see tool calls until they land as messages.
- **Schema migrations** (`docs/db-central.md`): new host DB tables get numbered migration files in `src/db/migrations/`.
- **Supply chain hygiene** (`pnpm-workspace.yaml minimumReleaseAge: 4320`): any new npm dep on host side waits 3 days before resolution; container-side bun deps are pinned manually.

### Library docs (verified)

- **`@anthropic-ai/claude-agent-sdk`** (Context7 / live): `query()` accepts `systemPrompt: { type: 'preset', preset: 'claude_code', append: instructions }` per `claude.ts:946`. The `append` field is the documented mechanism for per-call system prompt extension. Per-turn injection via `systemContext.instructions` change at the agent-runner layer is supported.
- **mnemon CLI v0.1.2** (existing host install): `mnemon recall <query> --store <name> --limit <n>` returns ranked JSON with `results[].insight.{content,category,importance,entities,...}`. `mnemon remember <text> --cat <cat> --imp <1-5> --entities "<csv>" --source <agent|user|external>` writes a fact and returns dedup verdict + edge candidates. Confirmed via direct CLI call in prior session.
- **Anthropic Haiku 4.5** (current pin `claude-haiku-4-5-20251001`, per .env review): supports system prompt + JSON output schema via prompt; ~75-150 t/sec input on small payloads. Cost: ~$0.25/M input, $1.25/M output. Classifier prompt ~500 tokens in, ~50 tokens out per turn pair = ~$0.0002 per call. At 200 turns/day across 7 groups → ~$0.04/day.
- **Ollama nomic-embed-text** (existing host service): 274MB model, 768-dim, ~60-216ms warm on Ampere ARM aarch64. Used by mnemon for hybrid recall when `MNEMON_EMBED_ENDPOINT` + `MNEMON_EMBED_MODEL` env vars are set.
- **better-sqlite3 + TRUNCATE journal** (`src/message-archive.ts:54-57`): chosen specifically because containers read archive.db without WAL sidecar mounts. Confirmed compatible with concurrent host-daemon reader.

### Existing code (key files read)

- `src/message-archive.ts:67-81` — `messages_archive` schema. Has `id`, `agent_group_id`, `messaging_group_id`, `thread_id`, `role`, `text`, `sent_at`. Indexes on `(agent_group_id, sent_at)` and `(agent_group_id, thread_id, sent_at)` already exist for the daemon's queries.
- `src/router.ts:695-724` — router writes inbound user messages to inbound.db AND mirrors to archive.db in the same flow. Recall injection (Option B) hooks here.
- `container/agent-runner/src/providers/claude.ts:18-19, 968, 976-980` — currently imports + wires PR #68 mnemon hooks. All three import lines + the conditional hook attachment block need surgical removal (Cleanup Inventory C9).

## Options

The architecturally meaningful choice is **where the recall injection runs**: container-side (touching the agent-runner / provider abstraction) or host-side (touching the router and inbound.db). The chat-stream classifier daemon, source-ingestion daemon, and tool-output capture are common to all three options with minor variation; the design comparison is dominated by the recall injection placement.

### Option A: Container-side recall injection via QueryInput extension

**Approach:** Add `systemReminder?: string` to `QueryInput`. Agent-runner poll-loop calls a new `recallContext()` function before each `provider.query()` invocation; it queries mnemon (via either local CLI or HTTP to host) with the user message, formats top-K facts, and passes them in `systemReminder`. Each provider translates: Claude appends to `systemPrompt`, Codex prepends to system message, mock no-ops. Host runs the chat-stream classifier daemon and source-ingestion daemon.

**Pros:**
- Recall is per-turn, naturally tied to the user message (no message-ordering concerns)
- Provider abstraction stays in control of prompt assembly — each provider knows best how to inject system context
- Container has direct access to mnemon binary + Ollama (via existing host:11434 reachability) — low latency

**Cons:**
- Requires changes to `QueryInput` interface, plus implementation in every provider (claude.ts, codex.ts, mock.ts)
- Each provider's recall code is a potential drift surface (we just lived through this with PR #68's Claude-only hooks)
- Container-side recall code must depend on mnemon CLI shape (or call host HTTP endpoint) — which leaks the storage primitive into the container
- C7 (MemoryStore interface) becomes harder: the abstraction has to span container + host

**Pattern alignment:** Echoes PR #68's design at a higher level — agent-runner is the recall orchestrator. Codex's earlier recommendation cited this option ("put recall injection before provider.query() in the poll loop") but with the qualifier that it must be provider-agnostic — which forces the QueryInput interface change.

**Confidence:** Medium — provider implementations are straightforward but the cross-provider drift risk is the same root cause as PR #68's failure (Claude-specific hooks). Adding a field to QueryInput is supposed to make it provider-agnostic, but every provider has to actually implement it correctly.

**Assumptions:**
- All current providers (claude, codex, mock) can be modified in this PR without scope creep
- A future OpenCode/anthropic-via-bedrock provider would correctly implement the new field

---

### Option B: Host-side recall injection via inbound.db `system` message

**Approach:** Recall lives entirely host-side in `src/router.ts`. After router resolves the agent group + session for an inbound user message, but before writing the user message to inbound.db, host calls `MemoryStore.recall(agentGroupId, userMessageText)` with a 1500ms hard timeout. On success, host writes a `kind: 'system'` message containing the recall context to inbound.db FIRST, then writes the user message. The agent-runner's poll loop already batches all pending messages_in into a single prompt — both rows get formatted together and the agent sees the system context immediately before the user message. Container has no recall code at all. Container still has mnemon binary access, but only for the daily synthesise task (read mnemon → write wiki pages).

**Pros:**
- Provider-agnostic by construction — recall lives entirely on the host; container never sees mnemon during chat turns
- Zero changes to AgentProvider interface, QueryInput, or any provider implementation file
- Container changes are minimal: just delete PR #68 hook code, add tool-output capture for content tools (separate concern)
- MemoryStore is a single Node module on the host (`src/modules/memory/store.ts`) — no IPC, no cross-runtime boundaries
- Fault isolation is excellent: recall failure (Ollama down, mnemon error) just means the system message isn't written — agent processes user message normally without context
- Reuses an existing message kind (`system`) that the agent-runner formatter (`container/agent-runner/src/formatter.ts`) already handles

**Cons:**
- Recall happens in the router's hot path. Adds 100-750ms (typical) before the user message lands in inbound.db. For warm containers (most of Dave's groups), this adds perceptible latency to "first agent response."
- The synchronous recall in the router blocks subsequent message-handling for that platform event. Not a thread blocking concern (Node single-threaded), but the router is sequential per platform event — slow recall slows that path.
- System message in inbound.db pollutes message history slightly (visible to the agent's archive search if not filtered out)
- New `system` message subtype/content shape needs a small contract: `{ subtype: 'recall_context', facts: [...] }` — formatter needs to know to format this differently than action-result system messages

**Pattern alignment:** Aligns with `docs/architecture.md` § Message Flow ("Host writes message to session's DB → Host calls wakeUpAgent"). Adds one more thing host can write to inbound.db — fits the model. Codex's earlier recommendation explicitly considered this approach and noted it's coherent (referenced the archive.db consumption pattern).

**Confidence:** High — every primitive needed already exists. archive.db cross-process reads work (TRUNCATE journal). System messages are formatted natively. mnemon recall via host CLI is a 5-line wrapper. The only new code is `src/modules/memory/` (the host-side MemoryStore + recall function) and a small change in `src/router.ts` to call it.

**Assumptions:**
- Recall p50 ≤ 750ms warm (per C5; validated by Ollama bench data)
- The router hot-path latency increase (100-750ms typical) is acceptable
- A `system` message with `subtype: 'recall_context'` won't be confused with action-result system messages by the formatter — straightforward to disambiguate via the subtype field

### Option C: Hybrid — in-container recall via local mnemon binary, host-side classifier

**Approach:** Container retains mnemon binary + RW mount of the group's local store (preserved infrastructure from PR #68). Agent-runner poll-loop calls `mnemon recall <query> --store <name>` directly (via the existing wrapper) before each `provider.query()`. Host runs only the chat-stream classifier daemon, which writes facts via the same mnemon CLI from the host side. Both sides write to the same SQLite store DB; concurrent writes coordinated via the wrapper's existing flock (already deployed at `~/.mnemon/data/<store>/.write.lock`). Tool-output capture also writes via mnemon.

**Pros:**
- Lowest recall latency (no router hot path, no IPC; just local SQLite + Ollama localhost)
- Reuses the wrapper's flock + STORE-pinning + mount-narrowing security infrastructure verbatim — most defense-in-depth wins from PR #68 are preserved
- Synthesise task naturally runs in this model (agent already has mnemon access for daily synth)

**Cons:**
- Container-side recall code = container-side memory dependency. The MemoryStore "interface" leaks: the agent-runner has to know about mnemon CLI shape, MNEMON_STORE env var, etc.
- Same fundamental drift surface as Option A — recall code in agent-runner must be provider-agnostic, but it's now also the path that hooks the SDK. PR #68's failure mode is structurally possible again.
- Two writers (host daemon + container, via the wrapper flock) on the same SQLite file = more contention, more failure modes; even with flock the operational model is harder to reason about
- Container needs MNEMON_STORE env, mnemon RW mount, ollama reachability — every config bug we hit in PR #68 stays possible

**Pattern alignment:** Closest to PR #68's deployment shape. Reuses more of the existing wrapper infrastructure but inherits more of the failure surface.

**Confidence:** Medium-Low — the wrapper's security defenses are good and worth keeping, but coupling the agent-runner to mnemon CLI shape goes against C7 (MemoryStore interface). The flock-based two-writer model works but adds operational complexity.

**Assumptions:**
- Two writers on one SQLite file via flock are stable enough for production
- Container can reach Ollama on host:11434 reliably (current proxy config supports this)

## Recommendation

**Selected:** Option B (Host-side recall injection via inbound.db system message)

**Justification:**

Three constraint clusters drive this choice:

1. **C7 + C8 (MemoryStore interface + cross-provider portability):** Option B is the only option where the storage primitive never crosses the host/container boundary or the provider abstraction. The MemoryStore is a single Node module on the host; agent-runner has no recall code at all. Adding a future provider (OpenCode, Bedrock-routed Claude) requires zero changes to memory infrastructure. Options A and C both put memory code in agent-runner, which means future providers re-derive that code (the PR #68 failure mode).

2. **C1 + C4 (async writes + fail-open):** Option B's failure mode is the cleanest: timeout the host-side recall, skip the system-message write, agent processes user message without context. No retries to muddle, no concurrent writers to coordinate, no cross-process state to recover. Options A and C have the same fail-open semantics but with more state to manage.

3. **C9 (surgical cleanup):** Option B has the smallest code footprint. The agent-runner's mnemon hooks (`container/agent-runner/src/modules/mnemon/*`) are pure-removed without replacement. Wrapper keeps its security defenses but loses phase gating + metrics. Container.json `mnemon` block becomes `memory: { enabled: boolean }`. Options A and C add new code in the provider layer + agent-runner core to replace what's removed.

The latency concern (recall in router hot path adds 100-750ms typical) is real but bounded by C5's hard timeout (1500ms). The platform message → inbound.db → wakeContainer flow already takes ~200ms-2s for warm containers; recall adds within an acceptable envelope. Slack/Discord users do not perceive sub-second response delays.

The "system message pollutes archive" concern is mitigated by skipping `kind: 'system'` rows in the chat-stream classifier (it should only classify `chat` / `chat-sdk` rows, not its own injections — natural separation).

**Confidence note:** High — every primitive validated against existing code. The only new operational risk is router hot-path latency, which is bounded and fail-open.

**M13 — Secret redaction layer (cycle-2):**

`MemoryStore.remember()` invokes a centralized redaction step before the mnemon-impl write. Implementation in `src/modules/memory/secret-redactor.ts`:

```ts
export interface RedactionResult {
  shouldStore: boolean;
  redactedContent?: string;          // when partial redaction is meaningful
  reason?: string;                   // for telemetry
}

export function redactSecrets(fact: FactInput): RedactionResult {
  const text = fact.content;

  // Hard refusals — block fact entirely
  const HARD_BLOCK_PATTERNS = [
    /-----BEGIN (?:RSA |EC |DSA |OPENSSH )?PRIVATE KEY-----/,
    /[Bb]earer\s+[A-Za-z0-9_\-\.=]{20,}/,
    /\b(?:sk-|sk-ant-|xai-|gho_|ghp_|gsk_)[A-Za-z0-9_\-]{20,}/,
    /\bAKIA[0-9A-Z]{16}\b/,                                  // AWS access key id
    /\beyJ[A-Za-z0-9_\-]{20,}\.eyJ[A-Za-z0-9_\-]{20,}\./,    // JWT (header.payload.signature)
    /\b[Aa]uth(?:orization)?\s*[:=]\s*['""]\w+['""]/,
    /\b(?:[Pp]assword|[Aa]pi[_-]?key|[Ss]ecret)\s*[:=]\s*\S+/,
  ];

  for (const pattern of HARD_BLOCK_PATTERNS) {
    if (pattern.test(text)) {
      return { shouldStore: false, reason: `secret pattern matched: ${pattern.source}` };
    }
  }

  return { shouldStore: true };
}
```

**M11 — Subprocess lifecycle on timeout (mnemon-impl):**

```ts
async recall(agentGroupId, query, opts = {}) {
  const timeoutMs = opts.timeoutMs ?? 1500;
  const limit = opts.limit ?? 5;
  const controller = new AbortController();
  if (opts.signal) opts.signal.addEventListener('abort', () => controller.abort());

  const child = spawn('/home/ubuntu/.local/bin/mnemon', [
    'recall', query,
    '--store', agentGroupId,
    '--limit', String(limit),
  ], { signal: controller.signal });

  const timeoutId = setTimeout(() => {
    controller.abort();
    // Grace period: SIGTERM via abort signal; SIGKILL after 500ms if still alive
    setTimeout(() => { try { child.kill('SIGKILL'); } catch {} }, 500);
  }, timeoutMs);

  try {
    const stdout = await collectStdout(child);
    return parseRecallJSON(stdout, query);
  } finally {
    clearTimeout(timeoutId);
  }
}
```

The `AbortController` + `controller.abort()` causes Node to send SIGTERM to the child process; the secondary `setTimeout` ensures SIGKILL grace if SIGTERM doesn't take effect within 500ms. This prevents file descriptor / process accumulation under sustained Ollama hangs.

The interface boundary keeps swappability to sqlite-vec/etc. (note — only `MemoryStore` is a public interface; `mnemon-impl` is internal-only).

#### 2. Auto-recall injection — central hook at `writeSessionMessage` (M1, M3, M5, M10, S3, S8)

**Hook location:** `src/session-manager.ts`'s `writeSessionMessage` function (or equivalent central inbound-write boundary), NOT `src/router.ts`. This covers ALL chat-kind inbound writes uniformly: platform user messages from router, agent-to-agent messages from delivery action handler, retries from host-sweep. Task-kind messages (synthesise scheduled tasks, other scripted tasks) explicitly opt out.

**Coverage scope** (M1 + M4 + M5 resolution — Q1 narrower scope):
- `kind: 'chat'` AND `channelType !== 'agent'` → recall fires (real user platform messages)
- `kind: 'chat'` AND `channelType === 'agent'` → recall does NOT fire (agent-to-agent introductions, approval-card responses, scheduling notifications, self-mod replies — internal-system writes that happen to use the chat kind)
- `kind: 'chat-sdk'` → recall fires (Chat SDK-routed user messages)
- `kind: 'system'` → recall does NOT fire (host-action results, no chat input — also includes our own recall_context writes, which prevents amplification)
- `kind: 'task'` → recall does NOT fire (any scheduled task — synthesise reads mnemon directly via Bash, other scheduled tasks may also opt in via Bash invocation if they want context)
- `kind: 'webhook'` → recall fires (chat-equivalent input from external integrations)

The hook NO LONGER tries to discriminate task content via a fictional `wakeAgent` field (that field doesn't exist in the codebase — actual fields are `trigger` and parsed-content `script`). The `kind === 'task'` branch always returns false; scheduled tasks self-manage their mnemon access.

**Pseudocode** (in `src/modules/memory/recall-injection.ts`):

```ts
// Called from writeSessionMessage before the inbound write
export async function maybeInjectRecall(params: {
  agentGroupId: string;
  sessionId: string;
  inboundMessage: SessionMessageInput;     // the message about to be written
  routing: RoutingAddr;
}): Promise<void> {
  if (!memoryEnabledForGroup(params.agentGroupId)) return;

  // M5: discriminate internal-system kind:'chat' writes by channel_type
  if (!shouldRecallForKind(params.inboundMessage.kind, params.routing.channelType)) return;

  const queryText = extractRecallQueryText(params.inboundMessage, params.sessionId);
  if (!shouldRecall(queryText)) return;     // noise pre-filter (length, single-emoji, ack-only)

  // S3/cycle-1 — gate on engagement: only fire when the message will actually wake the agent
  if (params.inboundMessage.trigger === 0) return;

  let recallResult: RecallResult | null = null;
  try {
    recallResult = await memoryStore.recall(params.agentGroupId, queryText, {
      limit: 5,
      timeoutMs: 1500,
    });
  } catch (err) {
    log.warn('Recall failed, fail-open', { err, sessionId: params.sessionId });
    healthRecorder.recordRecallFailOpen(params.agentGroupId, err);
    return;
  }

  if (!recallResult || recallResult.facts.length === 0) return;

  // M10 cycle-1 — namespace ID consistently with user-message namespacing
  // M2 cycle-3 — `writeSessionMessageRaw` doesn't exist; use existing `insertMessage` from
  // src/db/session-db.ts directly. Bypassing writeSessionMessage prevents recursion
  // (the recall hook is *invoked from* writeSessionMessage; recursing would loop).
  const recallId = `recall-${params.inboundMessage.id}`;  // already namespaced via messageIdForAgent in router
  const sessionDb = getInboundDbForSession(params.agentGroupId, params.sessionId);
  insertMessage(sessionDb, {
    id: recallId,
    kind: 'system',
    timestamp: params.inboundMessage.timestamp,
    platform_id: params.routing.platformId,
    channel_type: params.routing.channelType,
    thread_id: params.routing.threadId,
    content: JSON.stringify({
      subtype: 'recall_context',
      text: formatRecallSystemMessage(recallResult),
      query: queryText,
      recallVersion: RECALL_FORMAT_VERSION,    // S4 cycle-1 — versioned format
    }),
    trigger: 0,
  });
  updateSessionLastActive(params.sessionId);
}
```

**Recall query strategy** (S8 resolution):
- For first-turn-of-thread messages (no prior user msgs in this thread within last 24h): use raw user text, normalized (strip @mentions, attachment metadata, length-cap to 500 chars)
- For follow-up messages (thread has ≥ 2 prior user messages): concatenate last 3 user-runs from this thread, length-cap to 800 chars total. Helps short follow-ups like "yes do that" inherit the thread's semantic context.

**`shouldRecall(text)` noise pre-filter:**
- Length ≥ 20 chars OR ≥ 4 words (after stripping mentions/whitespace)
- Not single-emoji
- Not in ack-only allow-list: `['ok', 'yes', 'no', 'sure', 'k', 'lol', 'cool', 'nice', 'thanks', 'thx', 'np', '👍', '🙏', '👌', ...]`
- Configurable via daemon config

**`shouldRecallForKind(kind, channelType)`:**
- `chat` AND `channelType !== 'agent'` → `true` (real user platform messages)
- `chat` AND `channelType === 'agent'` → `false` (a2a / approval responses / internal plumbing)
- `chat-sdk` → `true`
- `webhook` → `true`
- `task` → `false` (scheduled tasks self-manage mnemon access)
- `system` → `false`

The first signal short-circuits — we don't need to parse content. `channelType: 'agent'` is the existing discriminator at `src/router.ts` and friends; it's set whenever the host writes to a session DB on behalf of internal-system flow.

#### 3. memory-daemon (host systemd unit)

`scripts/memory-daemon.ts` runs as the systemd service. Single process, two responsibilities:

**Chat-stream classifier loop** (60s tick):
```
for each enabled group (memory.enabled in container.json, refreshed each sweep):
  for each thread_id with messages since scan_cursor:
    -- M5 cycle-3: filter agent-to-agent / internal-system traffic from archive
    SELECT id, role, text, sent_at, channel_type
      FROM messages_archive
      WHERE agent_group_id = ?
        AND channel_type != 'agent'        -- exclude a2a, internal-system writes
        AND role IN ('user', 'assistant')   -- exclude system rows; recall_context is also excluded by this
        AND sent_at > scan_cursor
      ORDER BY sent_at ASC

    1. Group consecutive same-role runs
    2. Pair (user-run, assistant-run) when both stable (last assistant > 120s ago)
       OR (user-run alone) when orphan (>10 min since last user msg)
    3. For each stable/orphan pair (per-pair atomic — M8 cycle-1, M10 cycle-2):
       - Build classifier prompt: [user turn] + [assistant response]
       - Haiku call with strict JSON schema: {worth_storing, facts:[{content, category, importance, entities, source_role}]}
       - For each worth_storing fact (with deterministic idempotencyKey per M3 cycle-3):
         - Skip if pair already in processed_pairs
         - Call MemoryStore.remember(agentGroupId, fact, { idempotencyKey: ... })
       - Once all facts written: insert processed_pairs once
       - Advance success_watermark to last sent_at of pair
       - Advance scan_cursor (always, even on poison — M6 cycle-3)
```

**M6 cycle-3 — retries are a separate dispatch path:**
```
-- Pull due retries from dead_letters via index, NOT from archive scan
SELECT * FROM dead_letters
  WHERE poisoned_at IS NULL
    AND next_retry_at <= now()
  ORDER BY next_retry_at ASC

For each due retry:
  attempt classification
  on success: delete dead_letters row + insert processed_pairs + advance success_watermark
  on failure: increment failure_count, set next_retry_at via backoff, set poisoned_at if count >= 3
```

The scan_cursor advances past every inspected pair (including retry-pending and poisoned). The success_watermark advances only on success. Operator replay = items in `dead_letters` (not gap-derived).

**Source-file watcher** (inotify-driven):
```
on new file in groups/<g>/sources/inbox/:
  1. Read content
  2. Compute sha256 of canonicalized content
  3. Skip if (sha256, extractor_version, prompt_version) in ingest-state.processed_sources
  4. Call Haiku with extraction prompt for this content type
  5. For each worth_storing fact:
     - Call MemoryStore.remember(agentGroupId, fact)
  6. Insert into ingest-state.processed_sources
  7. Move source file to groups/<g>/sources/processed/<date>/<original-name>
```

**State DB:** `data/mnemon-ingest.db` (new, separate from archive.db / v2.db)

```sql
CREATE TABLE processed_pairs (

  thread_id TEXT,
  user_run_first_id TEXT NOT NULL,    -- first archive id in user run
  user_run_last_id TEXT NOT NULL,

  assistant_run_last_id TEXT,

  classified_at TEXT NOT NULL,
  facts_written INTEGER NOT NULL,
  -- M6 (cycle 2): is_orphan in PK so orphan-then-stable transitions don't collide.
  -- Orphan row gets PK (group, user_first_id, version, version, 1); stable pair gets (..., 0).
  PRIMARY KEY (agent_group_id, user_run_first_id, classifier_version, prompt_version, is_orphan)

-- M9: agent_group_id added to PK so same content shared across groups ingests independently
CREATE TABLE processed_sources (
  agent_group_id TEXT NOT NULL,
  source_path TEXT NOT NULL,
  content_sha256 TEXT NOT NULL,
  extractor_version TEXT NOT NULL,
  prompt_version TEXT NOT NULL,
  classified_at TEXT NOT NULL,
  facts_written INTEGER NOT NULL,
  PRIMARY KEY (agent_group_id, content_sha256, extractor_version, prompt_version)

-- M11 (cycle 2): split scan cursor from success watermark.
-- scan_cursor advances past poisoned items (so we don't re-attempt indefinitely);
-- last_classified_sent_at advances ONLY on successful classification (per HARD constraint C19).
-- Operator replay command can re-classify items between scan_cursor and last_classified_sent_at
-- after a prompt fix.
CREATE TABLE watermarks (
  agent_group_id TEXT NOT NULL,
  thread_id TEXT,                      -- null = group-level (rare, for non-threaded channels)
  last_classified_sent_at TEXT NOT NULL,    -- success watermark (C19 invariant)
  scan_cursor TEXT NOT NULL,                -- cursor past everything inspected (incl. poisoned/skipped)
  PRIMARY KEY (agent_group_id, thread_id)
);

-- S10: dead-letter schema for poison items
CREATE TABLE dead_letters (
  id TEXT PRIMARY KEY,
  item_type TEXT NOT NULL,             -- 'turn-pair' | 'source-file'
  item_key TEXT NOT NULL,              -- pair id or source_path
  agent_group_id TEXT NOT NULL,
  failure_count INTEGER NOT NULL DEFAULT 0,
  last_error TEXT NOT NULL,
  last_attempted_at TEXT NOT NULL,
  next_retry_at TEXT,                  -- null when poisoned
  poisoned_at TEXT,                    -- non-null = stop retrying, log for operator review
  operator_notes TEXT
);
CREATE INDEX idx_dead_letters_retry ON dead_letters(next_retry_at) WHERE poisoned_at IS NULL;
```

**Retry / dead-letter policy (S10):**
- Failures retried with exponential backoff: 60s, 5m, 30m
- After 3 retries, mark `poisoned_at`. Daemon advances past poisoned items but logs to `dead_letters`.
- Health JSON exposes `deadLettersCount` per group + oldest unpoisoned retry's `next_retry_at`.

**Per-pair atomic processing (M8 cycle-1 + M10/M11 cycle-2):**

```
for each (group, thread) with new messages since scan_cursor:
  pairs = group_into_turn_pairs(messages)  # returns stable + orphan pairs

  for pair in pairs:
    pair_key = (agent_group_id, user_run_first_id, classifier_version, prompt_version, is_orphan)

    if pair_key in processed_pairs:
      advance_scan_cursor(agentGroupId, thread, pair.last_sent_at)
      continue

    if pair_key in dead_letters:
      if poisoned: advance_scan_cursor only; do NOT advance success watermark
      else if next_retry_at < now: retry below
      else: continue without advancing scan_cursor (will re-poll next sweep)

    try:
      facts = call_haiku_classifier(pair)   # may raise
      facts = redact_secrets(facts)         # M13 (cycle 2): centralized redaction; may drop facts

      # M10 (cycle 2): per-fact idempotency at the MemoryStore boundary.
      # If a partial-failure retry happens, facts already-written are no-op'd by the key check.
      for fact_index, fact in enumerate(facts):
        idempotency_key = sha256(f"{pair_key}|{fact_index}|{classifier_version}|{prompt_version}")
        memoryStore.remember(agentGroupId, fact, { idempotencyKey: idempotency_key })

      insert processed_pairs(*pair_key, facts_written=len(facts), classified_at=now)
      advance_success_watermark(agentGroupId, thread, pair.last_sent_at)
      advance_scan_cursor(agentGroupId, thread, pair.last_sent_at)

    except err:
      record_or_increment_dead_letter(pair_key, err)
      # if this attempt poisoned the item, advance scan_cursor past it (M11)
      if dead_letters[pair_key].failure_count >= 3:
        advance_scan_cursor(agentGroupId, thread, pair.last_sent_at)
```

**Watermark vs scan-cursor distinction (M11):**
- `scan_cursor` = "I've inspected up to here." Advances on success AND on poisoning. Used for what to read next from archive.db.
- `last_classified_sent_at` = "I've successfully classified up to here." Advances ONLY on success. Operators can replay items between `scan_cursor` and `last_classified_sent_at` (the gap = poisoned items) after a prompt or extractor fix.

##### Atomic write protocol (M3 resolution, applies to all 4 surfaces)

All writers create `<final-path>.tmp`, write content, then `fs.rename()` to the final path. Daemon's inotify watcher uses event mask `IN_CLOSE_WRITE | IN_MOVED_TO`, NOT `IN_CREATE`. This skips partial-write reads.

##### Bind mount (M3 resolution)

`/workspace/agent/sources/` already maps to `groups/<g>/sources/` via the existing `/workspace/agent` bind mount at `src/container-runner.ts` (group folder mount). No new mount entry needed; the existing bind covers it. The `enable-memory.ts` script creates `groups/<g>/sources/{inbox,articles,docs,transcripts,clips,media,processed}/` so the directory exists before the daemon's first sweep.

##### Daemon ordering invariant (M3 resolution)

If the daemon's inotify watcher starts before any `enable-memory.ts` has run for any group, no source dirs exist and inotify silently does nothing. Resolution: daemon's startup creates the watcher only AFTER scanning `groups/*/container.json`, and lazily-creates new watchers on each 60s sweep when it sees a newly-enabled group. Existing watchers persist; new groups pick up on next sweep.

#### 9. Host-prerequisite verification (M12 — retro learning #1 mitigation)

`scripts/verify-memory-prereqs.sh` runs at:
- Build acceptance test (`/team-build` Day-1 gate, blocks on failure)
- memory-daemon startup (refuses to start, writes `data/memory-health.json` with `prereqVerification.ok: false`, exits non-zero)

Checks:
```bash
#!/usr/bin/env bash
set -euo pipefail

errors=()

# 1. Ollama service active
if ! systemctl is-active --quiet ollama; then
  errors+=("ollama systemd service not active")
fi

# 2. nomic-embed-text model pulled
if ! ollama list 2>/dev/null | grep -q "^nomic-embed-text"; then
  errors+=("ollama model nomic-embed-text not pulled")
fi

# 3. mnemon binary at expected version
mnemon_version=$(/home/ubuntu/.local/bin/mnemon --version 2>/dev/null | awk '{print $NF}' || echo "missing")
expected_version="0.1.2"  # pinned in container/Dockerfile MNEMON_VERSION ARG
if [[ "$mnemon_version" != "$expected_version" ]]; then
  errors+=("mnemon binary expected $expected_version, got $mnemon_version")
fi

# 4. Disk free for ~/.mnemon and data/mnemon-ingest.db
mnemon_dir="${HOME}/.mnemon"
mkdir -p "$mnemon_dir"
free_gb=$(df -BG --output=avail "$mnemon_dir" | tail -1 | tr -d 'G ')
if [[ "$free_gb" -lt 1 ]]; then
  errors+=("disk free in ${mnemon_dir}: ${free_gb}GB (need ≥ 1GB)")
fi

# 5. inotify watch capacity for ~10 groups × 10 dirs
max_watches=$(cat /proc/sys/fs/inotify/max_user_watches 2>/dev/null || echo 0)
if [[ "$max_watches" -lt 1024 ]]; then
  errors+=("inotify max_user_watches=${max_watches} (need ≥ 1024 for ~7-10 groups)")
fi

# 6. sqlite3 CLI available (for daemon's ingest-state DB management)
if ! command -v sqlite3 >/dev/null; then
  errors+=("sqlite3 CLI not in PATH")
fi

if [[ ${#errors[@]} -gt 0 ]]; then
  echo "Memory layer prerequisites FAILED:" >&2
  for e in "${errors[@]}"; do echo "  - $e" >&2; done
  exit 1
fi

echo "Memory layer prerequisites OK"
```

Daemon imports this check at startup:
```ts
import { execSync } from 'child_process';
try {
  execSync('bash scripts/verify-memory-prereqs.sh', { stdio: 'inherit' });
} catch (err) {
  writeUnhealthyHealthJson({ prereqVerification: { ok: false, error: String(err) } });
  process.exit(2);  // systemd will record + restart, but if prereq is the actual problem, restarting won't help — operator sees the unhealthy state
}
```

Build acceptance gate (in `/team-build` plan): runs the script before declaring a builder Day-1 acceptance complete. Failed prereq blocks the build.

Failure modes:
- Ollama not installed → operator runs `curl -fsSL https://ollama.com/install.sh | sh && sudo systemctl enable --now ollama`
- nomic-embed-text not pulled → operator runs `ollama pull nomic-embed-text`
- mnemon version drift → operator runs `~/.local/bin/mnemon` upgrade or rebuild container Dockerfile

## Decision Record Updates

*Appended to `docs/specs/mnemon-rearchitecture/decisions.yaml` — see `skills/shared/decision-record-schema.md`*

**Decisions:**
- **D12:** Chose Option B (host-side recall via system-message injection in router). Rejected Option A (container-side QueryInput extension) because it puts recall code in agent-runner and every provider must implement it correctly — same drift surface as PR #68. Rejected Option C (in-container recall via local mnemon binary) because it couples agent-runner to mnemon CLI shape, violating C7 (MemoryStore interface) and inheriting PR #68's coupling. Constraint refs: [C1, C4, C7, C8, C9].
- **D13:** Chose single host daemon (chat-stream classifier + source-ingestion in one process) over separate systemd units. Rationale: low operational complexity, single health JSON, single set of telemetry. Per-component process split is a follow-up if fault isolation becomes an issue. Constraint refs: [C18, C25].
- **D14:** Chose new `kind: 'system' + subtype: 'recall_context'` over a new top-level message kind. Rationale: agent-runner formatter already handles `kind: 'system'`; minor branch on subtype is smaller surface change than introducing a new kind. Constraint refs: [C8].
- **D15:** Chose tool-output capture as a configurable list of MCP tool names with explicit hash-key logic per tool over a generic "all tool results captured" approach. Rationale: tool-result schemas vary widely; generic capture would need per-tool deserialization anyway. Initial list is reviewable. Constraint refs: [C14, C17].
- **D16:** Chose to remove `groups/<g>/.mnemon-metrics.jsonl` per-group telemetry files in favor of single host-side `data/memory-health.json`. Rationale: per-container telemetry was a PR #68 architectural artifact (hooks emitting from container); host-side daemon owns telemetry now. Constraint refs: [C9, C25].
