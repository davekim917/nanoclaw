## Brief Reference

Extend the `create_agent` MCP tool so an admin agent can pick a provider (Claude or Codex initially; OpenCode and others as one-file follow-ups) and pass a provider-specific `provider_config` object (`{model?, effort?}` for Claude; `{model?, reasoning_effort?}` for Codex). Validation must happen before any persistent state is written; config must be sticky across container restarts; existing callers (no `provider`, no `provider_config`) keep current behavior. PR-ready against `qwibitai/nanoclaw` upstream.

Full brief: `.context/specs/create-agent-provider/brief.md`.

### Flagged Constraints

- **C4 ‚Äî "Schemas live in `src/providers/<name>.ts`":** The brief specified the host path `src/providers/<name>.ts`. Research (C11) shows the container is the source of truth for provider availability and already has `zod` available. Co-locating schemas with the **container-side** provider files (`container/agent-runner/src/providers/<name>.ts`) better satisfies the *spirit* of C4 (schema lives with the provider, no central enum) while avoiding the validation-surface mismatch the brief did not anticipate. The brief text *"or co-located there"* leaves room for this refinement, but it is a deliberate choice that should be visible. Documented as decision **D4**.
- **C6 ‚Äî Fire-and-forget narrowed:** Validation errors return *synchronously* via the MCP `isError` content block ‚Äî no system message is ever written for invalid calls. Fire-and-forget semantics still apply to the *creation* step (success notification arrives via `notifyAgent`). This is stricter than the brief, not looser, and trivially satisfies C1.
- **C2 ‚Äî admin-only, waived (D5).** The `/team-review` fact-check found that the admin gate described in `agents.ts:8` does not exist ‚Äî `mcp-tools/index.ts` unconditionally registers `agents.ts`, and `handleCreateAgent` has no `user_roles` check. C2 was carried forward from the brief as "Validated"; it is not. This PR does **not** add admin enforcement. Rationale for the waiver: the *qualitative* attack surface is unchanged (any agent can already spawn arbitrary Claude sub-agents with arbitrary CLAUDE.local.md content via the existing `create_agent` flow); this PR adds *quantitative* risk (spawning Codex/OpenCode agents can consume non-Claude API budget). That risk widens but does not cross a new authorization boundary. Fixing the gate is a separate concern that should track its own issue. Full decision, risk acknowledgment, and follow-up responsibility recorded as **D5**.

## Research Summary

- **Project patterns:**
  - Two parallel registries: container-side `provider-registry.ts` (`AgentProvider` factories ‚Äî Claude, mock baseline; Codex/OpenCode added by `/add-*` skills) and host-side `provider-container-registry.ts` (only providers needing mounts/env ‚Äî Claude/mock omitted) ‚Äî `src/providers/provider-container-registry.ts:1-26`.
  - The `create_agent` flow today: container MCP handler writes a `system` message via `writeMessageOut`; host's `delivery.ts` dispatches the action to `handleCreateAgent`; that handler hardcodes `agent_provider: null` (`src/modules/agent-to-agent/create-agent.ts:81`) and calls `notifyAgent` to surface success.
  - Per-group config persists in `groups/<folder>/container.json`, mounted read-only at `/workspace/agent/container.json`. Existing fields include `defaultModel` and `defaultEffort` (Claude-specific; flow into env vars at spawn ‚Äî `src/container-runner.ts:1206-1224`). The container runner reads container.json once at startup via `container/agent-runner/src/config.ts:29`.
  - The container's `RunnerConfig` interface (`container/agent-runner/src/config.ts:12`) currently exposes only `provider`, `assistantName`, `groupName`, `agentGroupId`, `maxMessagesPerPrompt`, `mcpServers`. New fields (e.g., `providerConfig`) would extend this.
  - `AgentProvider` interface (`container/agent-runner/src/providers/types.ts:1`) and `QueryInput` (line 58): per-query `model` and `effort` are first-class fields the SDK passes to its query call; Claude provider already honors them (`claude.ts:824-825`).
  - Active churn: most recent commit on this module was `46b19dc refactor(modules): extract agent-to-agent as registry-based module` ‚Äî registry pattern is current best practice in this area. Match it.
- **Library docs:**
  - `zod@^4.0.0` already a runtime dep in `container/agent-runner/package.json`. Not yet imported anywhere in container source ‚Äî this work would be the first user.
  - Claude Agent SDK `effort` field: first-class since Opus 4.6 per the inline comment at `types.ts:88`. Verified by reading the existing claude provider ‚Äî `effort: input.effort as EffortLevel` is passed straight through (`claude.ts:825`).
  - Codex CLI honors `model` and `model_reasoning_effort` via `~/.codex/config.toml`; the host-side codex provider already mounts a per-session `~/.codex` directory (per `/add-codex` skill `SKILL.md:140`). For `CODEX_MODEL`, env-var passthrough is also supported.
- **Existing code (key cites for design decisions):**
  - `container/agent-runner/src/mcp-tools/agents.ts:31-66` ‚Äî current `create_agent` MCP handler. ~35 lines, clean to extend.
  - `src/modules/agent-to-agent/create-agent.ts:37-126` ‚Äî current host handler. The `agent_provider: null` line is the one this work flips.
  - `src/container-config.ts:57-143` ‚Äî `ContainerConfig` interface; this is where a `providerConfig` field would land.
  - `src/container-runner.ts:417-440` ‚Äî provider name resolution (`resolveProviderName` + `resolveProviderContribution`). No change needed; new code consumes this unchanged.

## Options

### Option A: Host-side validation; schemas in `src/providers/<name>.ts` (literal brief reading)

**Approach:** Add a host-side metadata registry that holds Zod schemas alongside the existing container-config functions. Each provider's `src/providers/<name>.ts` registers both. Add a baseline `src/providers/claude.ts` (currently missing) that registers only a schema. `handleCreateAgent` validates `provider` against the registry and `provider_config` against the schema before any DB write; failures send `notifyAgent` error.

**Pros:**
- Validation happens in the same process that writes the DB row ‚Äî straightforward transaction boundary.
- Strict literal compliance with brief constraint C4's path.

**Cons:**
- Requires adding `zod` to root `package.json` (host) ‚Äî subject to the project's `minimumReleaseAge: 4320` rule. `zod@4.x` has been GA for months, so this is likely fine, but it's a new top-level dep.
- Requires creating a new `src/providers/claude.ts` whose only purpose is schema registration ‚Äî the host-side claude file does not exist today because Claude needs no host-side container-config (`src/providers/index.ts` comment block lines 1-7).
- Validation feedback arrives **asynchronously** via `notifyAgent` (the MCP handler has already returned `ok` by the time validation runs). The agent sees "Creating agent‚Ä¶" and then a separate error message ‚Äî confusing UX.
- Two provider registries to keep aligned: container-side knows what can run; host-side knows what's valid. Drift risk.

**Pattern alignment:** Mirrors `src/channels/channel-registry.ts` and the existing `provider-container-registry.ts` pattern, but in a parallel new file. Brief C4 literal compliance.

**Confidence:** Medium ‚Äî the new `zod` host dep adds friction with the supply-chain rule (would land but adds a verification step), and the dual-registry drift problem is a long-term liability.

**Assumptions:**
- `zod@4.x` passes the `minimumReleaseAge` gate at install time. (Likely true; verifiable via `pnpm view zod time`.)
- Claude is fine to "fake-register" host-side just for schema purposes.

### Option B: Container-side validation; schemas co-located in `container/agent-runner/src/providers/<name>.ts` (recommended)

**Approach:** Each container-side provider module exports both its factory (existing) and an optional Zod `configSchema` for `provider_config`. The container's `provider-registry.ts` gains a `getProviderConfigSchema(name)` lookup. The `create_agent` MCP handler in `agents.ts` validates `provider` against `listProviderNames()` and `provider_config` against the schema **before** writing the system action ‚Äî failures return synchronously as MCP `isError` content. On success, the action payload (now including `provider` and the validated `provider_config`) is sent to the host, which sets `agent_groups.agent_provider` and writes `providerConfig: Record<string, unknown>` into the new agent's `container.json`. The new container reads `container.json.providerConfig` at startup; the provider's factory or `query()` translates it into per-query SDK options.

**Pros:**
- **No new host dependency.** `zod@^4.0.0` is already a runtime dep in `container/agent-runner/package.json`. (Constraint C7.)
- **Single source of truth for "what providers are valid."** The container's registry is, by construction, the set of providers the running container image can actually instantiate. Host-side mirroring becomes unnecessary.
- **Validation feedback is synchronous.** The agent sees the `isError` content block in the same turn ‚Äî no fire-and-forget back-channel for errors. C1's "no orphan state" is satisfied trivially because no system message is ever written for a failed validation.
- Schema is **truly co-located** with the provider runtime (factory and schema in the same file). Skill installation pattern unchanged: `/add-codex` already drops a file at `container/agent-runner/src/providers/codex.ts` ‚Äî that file just gains a schema export.
- Per-group persistence uses a single new field (`providerConfig`) on `container.json`; no new DB column or migration.

**Cons:**
- Schema location differs from brief C4's literal text (host-side path). Treated as a deliberate refinement (see Flagged Constraints) but reviewers should agree.
- Container-side validation means a stale container image (skill installed on host but container not yet rebuilt) will reject the new provider. This is *correct* behavior (the container can't actually run it) but the error is "Unknown provider: codex" rather than "Run /add-codex" ‚Äî message wording matters.
- Adds a generic `providerConfig?: Record<string, unknown>` blob to `ContainerConfig`. Slightly less type-safe at the host layer than dedicated fields would be ‚Äî but the only consumer is the matching provider, which validates again on read.

**Pattern alignment:** Matches the existing container-side registry pattern (`container/agent-runner/src/providers/provider-registry.ts:1-33`) and skill-install convention (`/add-codex` SKILL.md). Preserves brief C4's *intent* (schema co-located with provider, no central enum) at a different physical location.

**Confidence:** High ‚Äî every assumption is grounded in code reads, no new deps, the validation surface mismatch from C11 is resolved correctly, and the change is the smallest viable PR for upstream.

**Assumptions:**
- A1 ‚Äî `container.json.providerConfig` field added to `RunnerConfig` in `container/agent-runner/src/config.ts` is sufficient for the container to receive per-group config without env-var plumbing on the host. **Validation:** Confirmed by reading `config.ts:29-49` ‚Äî the loader already maps fields from the JSON file directly.
- A2 ‚Äî Setting `agent_groups.agent_provider` (instead of `null`) does not break any existing read path. **Validation:** Confirmed by reading `resolveProviderName` (`container-runner.ts:417-423`) ‚Äî non-null values are honored, and the existing migration column already accepts arbitrary text.

### Option C: Hybrid ‚Äî container-side validation + map Claude config to existing `defaultModel`/`defaultEffort` fields

**Approach:** Same as Option B for validation (container-side, synchronous error), but on the host's persistence side, special-case Claude: write `provider_config.model` to `container.json.defaultModel` and `provider_config.effort` to `container.json.defaultEffort` (the existing fields that already plumb through env vars in `container-runner.ts:1206-1224`). Other providers use the new generic `providerConfig` blob.

**Pros:**
- Reuses existing wiring for Claude ‚Äî `defaultModel` and `defaultEffort` already flow into the right env vars; nothing new to test for the Claude path.
- `container.json` for a Claude agent stays "shaped like every other Claude agent on this host" ‚Äî easier for someone reading the JSON to understand.

**Cons:**
- **Asymmetric.** Claude config takes a host-side translation path; everyone else uses a generic blob. Two persistence paths to maintain forever.
- The existing `defaultModel` field is documented as "for the bare `opus` alias" specifically (`container-config.ts:113-118`) ‚Äî semantic mismatch with arbitrary user-supplied models like `claude-sonnet-4-6`. Either we change the field's documented purpose (bigger change) or we accept the drift.
- Future Claude knobs (e.g., thinking budget) would either need new dedicated fields *or* fall into the generic blob anyway, defeating the symmetry argument.

**Pattern alignment:** Partially matches existing patterns (Claude path), partially introduces new pattern (`providerConfig` blob). Less coherent than Option B.

**Confidence:** Medium ‚Äî works, but the asymmetry creates drift risk and the existing field's documented purpose doesn't quite fit.

**Assumptions:**
- The "for the bare `opus` alias" semantics of `defaultModel` can be quietly broadened. (Probably yes, but it's a documentation/intent change.)

## Recommendation

**Selected:** Option B.

**Justification:**

- Resolves the validation-surface mismatch (C11) at the right layer: the container is the actual source of truth for what providers can run, and validating there means we never lie to the agent about what's possible.
- Zero new host dependencies (C7 satisfied trivially); container already has `zod`.
- Synchronous validation feedback in the same MCP turn ‚Äî no fire-and-forget back-channel for the error path. C1 satisfied trivially because no system message is ever written for a failed call.
- Symmetric persistence model: every provider's config rides on the same `providerConfig` field; no two-track translation. Future providers (OpenCode and beyond) plug in identically.
- Smallest realistic diff for an upstream PR: 4-5 files touched, no new tables, no new top-level deps.
- Matches the upstream churn direction (the most recent agent-to-agent commit was a registry-extraction refactor ‚Äî the registry-based pattern is what the maintainers are currently building toward).

**Refinement of brief constraint C4:** Schemas live in `container/agent-runner/src/providers/<name>.ts` (container-side path) rather than `src/providers/<name>.ts` (host-side path). The brief's text *"or co-located there"* leaves room; the spirit (schema with provider, no central enum) is preserved. This is **decision D4** below.

**Confidence:** High. All assumptions in the Assumptions Log are validatable with code reads or short test runs; no spike needed.

## Implementation Surface (informational ‚Äî `/team-plan` decomposes)

Files expected to change (rough scope, not a task list):

- `container/agent-runner/src/providers/provider-registry.ts` ‚Äî add `registerProviderConfigSchema(name, schema)` + `getProviderConfigSchema(name)`. The container-side Claude registration must export a `configSchema`; every future skill-installed provider must do the same. When `getProviderConfigSchema(name)` returns `undefined` (provider is registered via `registerProvider` but has not declared a `configSchema`), the MCP handler treats it as `z.strictObject({})` ‚Äî i.e., **empty `provider_config` passes; any non-empty `provider_config` fails validation**. Addresses S2.
- `container/agent-runner/src/providers/claude.ts` ‚Äî export and register:
  ```ts
  configSchema = z.strictObject({
    model: z.string().min(1).optional(),
    effort: z.enum(['low', 'medium', 'high', 'xhigh', 'max']).optional(),
  })
  ```
  **`effort` is enum-constrained** to the SDK's full `EffortLevel` type (`container/agent-runner/node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts:462` ‚Äî five values, including `'max'` for Opus 4.6/4.7 per `sdk.d.ts:1338`). `model` is string-only with `.min(1)` to reject empty strings (model namespace is open-ended; bad ids surface at first wake ‚Äî acceptable). Use `z.strictObject(...)` (Zod 4 idiom) not `z.object(...).strict()`. Addresses M3 + M5 + W1.
  Read `configSchema`-validated values from a new constructor option (or a `query()` parameter sourced from `container.json`) and merge into existing per-query env/options.
- `src/container-config.ts` ‚Äî **also widen `ContainerConfig.defaultEffort` type** from `'low' | 'medium' | 'high' | 'xhigh'` to include `'max'`, matching the SDK. The existing field is stale relative to the SDK; this work surfaces the drift. One-line change.
- `container/agent-runner/src/providers/types.ts` ‚Äî extend `ProviderOptions` with `providerConfig?: Record<string, unknown>` (or surface via a separate channel ‚Äî `/team-plan` decides exact shape).
- `container/agent-runner/src/providers/factory.ts` ‚Äî pass `providerConfig` into the factory at construction.
- `container/agent-runner/src/mcp-tools/agents.ts` ‚Äî extend the input schema with `provider` + `provider_config`; validate against the registry; include validated values in the system action payload.
  - **MCP `inputSchema` (JSON Schema) stays coarse: `provider_config: { type: 'object', additionalProperties: true }`.** Fine-grained shape is Zod's job inside the handler. Document this split in a comment so future maintainers don't try to tighten the JSON Schema and create drift. Addresses S5.
  - **However (S12 fix):** the `provider_config` JSON Schema entry MUST include a `description` field that enumerates per-provider shapes so the calling LLM has a signal without having to guess. Example: `"Provider-specific config. Shape depends on 'provider'. Claude: { model?: string, effort?: 'low'|'medium'|'high'|'xhigh'|'max' }. Codex: { model?: string, reasoning_effort?: 'low'|'medium'|'high' }. Unknown keys are rejected."`. Keep it in sync with the Zod schemas ‚Äî a one-line comment above the description reminds future editors. (Longer-term, migrating to `z.toJSONSchema()` for single-source Zod ‚Üí JSON Schema generation is noted in R6 as a follow-up.)
- `container/agent-runner/src/config.ts` ‚Äî add `providerConfig: Record<string, unknown>` to `RunnerConfig`; populate from `container.json`.
- `container/agent-runner/src/index.ts` ‚Äî pass `providerConfig` through to the provider factory.
- `src/container-config.ts` ‚Äî add `providerConfig?: Record<string, unknown>` to `ContainerConfig`; include in `readContainerConfig`'s field list. Add a JSDoc comment pointing to `container/agent-runner/src/providers/<name>.ts` as the source of truth for per-provider schemas (addresses S7 ‚Äî keeps the dual-artifact pattern discoverable to someone reading `container.json` cold).
- `src/modules/agent-to-agent/create-agent.ts` ‚Äî accept `provider` + `providerConfig` from the action content. **Critical invariant (M1 + M6 fix):** on successful creation, three writes must land consistently: `agent_groups.agent_provider = provider` (SQLite), `container.json.provider = provider` (JSON file), and `container.json.providerConfig = providerConfig` (same JSON file).
  These cannot commit atomically ‚Äî SQLite and the filesystem are separate stores with no cross-store transaction. The design specifies a **sequenced-write-with-rollback** ordering that degrades safely on any failure:
  1. `initGroupFilesystem(newGroup, { instructions })` ‚Äî creates `groups/<folder>/`. On failure: propagate; no DB or config state exists yet.
  2. `updateContainerConfig(folder, c => { c.provider = provider; c.providerConfig = providerConfig ?? {}; })` ‚Äî **write `container.json` FIRST**, before the DB insert. On failure: propagate; no DB row references this folder yet. Caller may want to clean up the folder from step 1, but even if they don't, there's no DB pointer to a dangling config ‚Äî the folder is just orphaned-on-disk (discoverable via sweep or manual cleanup).
  3. `createAgentGroup({ ..., agent_provider: provider })` ‚Äî DB INSERT second. On failure: **rollback by removing `groups/<folder>/`** (or at minimum: delete `container.json` so a later re-attempt starts fresh). This is essential ‚Äî a DB row pointing to a misconfigured `container.json` is the split-brain M1/M6 were supposed to prevent.
  4. `createDestination(...)` √ó 2 (bidirectional), `writeDestinations(...)`, `notifyAgent(...)` ‚Äî unchanged from today. If any of these fail after step 3 succeeds, the agent exists with valid provider config but missing/partial destinations ‚Äî a separate, recoverable failure mode (not a split-brain).

  Do **not** describe this as "atomic." The word is wrong and would mislead an implementer into omitting the rollback.
- **Paired upstream/providers commit (M7 fix) ‚Äî required in the same PR scope.** This PR's changes span two branches to avoid a rollout window where Codex is unreachable:
  - **Trunk** ‚Äî everything above (registry extension, Claude schema, host persistence, MCP changes).
  - **`upstream/providers`** ‚Äî paired commit adding `configSchema` export to `container/agent-runner/src/providers/codex.ts`. Shape: `z.strictObject({ model: z.string().min(1).optional(), reasoning_effort: z.enum(['low','medium','high']).optional() })`. Also update `/add-codex`'s SKILL.md to note that a Codex-capable install now requires a providers-branch version that contains the schema export.

  Both commits land together as one coherent PR. Upstream reviewers see the schema concept, the Claude consumer, and the Codex consumer in one change ‚Äî which makes the registry contract testable end-to-end and closes the M7 "strictObject({}) rejects valid Codex config" window at merge time.

  Without this paired commit, D8's fallback (`z.strictObject({})` for unknown providers) would block Brief Success Criterion #1 on any install that has run `/add-codex` but not yet received the schema update. Shipping the schema now removes the window entirely.
- **Tests:**
  - `bun:test` for the container-side validation path ‚Äî success cases for Claude schema, failure cases for each validation branch (unknown provider; unknown key; bad enum value for `effort`; empty string `model`; valid `'max'` effort is accepted).
  - `vitest` for the host-side persistence path ‚Äî verify that after `handleCreateAgent` completes successfully, `agent_groups.agent_provider`, `container.json.provider`, and `container.json.providerConfig` are all set to the chosen values. **Also add a failure-injection test** that simulates `createAgentGroup` throwing after `updateContainerConfig` succeeds, and asserts the rollback removes `groups/<folder>/` (M6 invariant).

  - **Add a `test.todo('admin enforcement ‚Äî see followup issue for C2 gate')` marker (S9 fix)** at the top of the new test file. Keeps the D5-waived gap visible in the test suite until a separate PR closes it.

## Risks & Open Questions

- **R1 ‚Äî Stale-image error message wording.** If the host has `/add-codex` installed but the container image hasn't been rebuilt, validation will say "Unknown provider: codex." This is correct but could read as a bug to the agent. Mitigation: the error message includes `Registered: <list>` (already in `provider-registry.ts:25`), which makes the diagnosis obvious. If we want a friendlier message, we can add a host-side preflight check that warns when `src/providers/codex.ts` exists on disk but `listProviderNames()` doesn't include `codex` ‚Äî out of scope for this PR.
- **R2 ‚Äî Provider-side translation contract.** Each provider must define how `provider_config` translates into its SDK call. Claude maps cleanly (`model` ‚Üí `model`, `effort` ‚Üí `effort` ‚Äî already first-class). Codex needs to map `model` ‚Üí `CODEX_MODEL` env (or `~/.codex/config.toml`) and `reasoning_effort` ‚Üí `model_reasoning_effort` in `config.toml`. The codex provider file (added by `/add-codex`) is responsible for this; the contract is "read your slice of `providerConfig`, do whatever you need to apply it." Documented in the design but enforced by per-provider tests.
- **R3 ‚Äî Codex round-trip requires a one-time providers-branch sync on this fork (addresses M4; sharpened by M7).** The `/add-codex` skill pulls codex provider source from `origin/providers`. On this fork `origin/providers` is stale ‚Äî it has OpenCode but not Codex. Codex provider files DO exist on `upstream/providers`. The `M7` paired commit (see Implementation Surface) adds the `configSchema` export to `upstream/providers` as part of this PR. After the PR merges upstream, the user must sync their fork's providers branch:
  ```bash
  # Verbose form (recommended ‚Äî produces clear errors if fork has diverged)
  git fetch upstream providers
  git checkout providers
  git merge --ff-only upstream/providers
  git push origin providers
  ```
  If `git merge --ff-only` fails, the fork has divergent providers commits (OpenCode customizations, private provider work). In that case, merge `upstream/providers` into the local `providers` branch manually (resolving conflicts as needed) and push. Do not force-push unless the divergent commits are known to be disposable.
  Then re-run `/add-codex` to pick up the new files (including the schema export).
  Unlike the original M4, this PR no longer leaves a window where Codex is registered but its schema is not (M7 paired commit closes that window). Addresses M4 + S10.
- **R4 ‚Äî Host-side re-validation of the system-action payload.** The design has the container validate `provider_config` synchronously, then ship the validated values through a system action the host trusts and persists. The system action is a trust boundary the design does not close. In practice, the only writer is the same MCP handler in the same container, so this is theoretical today. Deferred: a lightweight host-side envelope validator (`z.strictObject({ provider: z.string(), provider_config: z.record(z.unknown()).optional() })` at the entry to `handleCreateAgent`) can be added later without touching the provider-specific schemas. Not blocking this PR.
- **R5 ‚Äî No schema-version field on persisted `providerConfig`.** Each provider's slice of `providerConfig` is persisted indefinitely in `container.json`. When a provider's schema evolves (renamed key, enum change), older persisted blobs may silently misapply or fail at first wake. This PR does not add a version field. Deferred: the first provider schema change that is not strictly backward-compatible will force the question, at which point a `schemaVersion` key + per-provider migrator is the natural fix. Noted so a future `/team-plan` for that change picks this up.
- **Q1 ‚Äî Future overrides at session/turn level.** The brief explicitly defers per-session and per-turn overrides. The `providerConfig` blob in `container.json` is the per-group sticky layer. Sessions and turns would layer on top later ‚Äî not addressed here, but the design does not preclude them.
- **Q2 ‚Äî `.strict()` / `z.strictObject` trade-off.** Strict validation (reject unknown keys) catches typos and guards against silent-acceptance of agent-fabricated keys. The cost: when a future schema version adds a new knob, older callers that pass it fail. This PR keeps strict because (a) `create_agent` is agent-driven and LLMs will sometimes invent plausible keys ‚Äî silent stripping would hide that, (b) the provider schemas are small and well-enumerated, so "unknown key" is a more likely signal of a bug than of forward-compat drift. Documented explicitly so it is a conscious choice, not a silent default. Addresses S3.
- **R6 ‚Äî Future migration to single-source Zod via `z.toJSONSchema()`.** Reviewer B's pattern check flagged that the dominant 2025-2026 MCP pattern is to generate the JSON Schema from Zod (single source of truth) rather than maintaining two separate artifacts. This PR does not adopt that pattern ‚Äî the MCP `inputSchema` is hand-written with `additionalProperties: true` and a descriptive `description` field (S12 fix). Reason for deferral: `z.toJSONSchema()` would require either (a) dispatching the inputSchema per-provider at call time (more ambitious, needs MCP SDK support) or (b) unioning all provider schemas into one JSON Schema (awkward and still lossy because the caller picks one provider at a time). Follow-up PR territory. Noted here so future work picks it up.
- **R7 ‚Äî Maintenance-time split-brain is not closed by this PR (S8 acknowledgment).** D6 fixes the CREATE path; `agent_groups.agent_provider` and `container.json.provider` can still diverge if a future admin migration or self-mod flow writes one without the other. This is a pre-existing structural asymmetry (host reads `agent_groups`; container reads `container.json`) and not introduced by this PR. Deferred with the same rationale as M2's waiver ‚Äî filed for follow-up. A future PR should introduce a `setAgentProvider(groupId, provider)` helper that is the sole mutation path for both stores, and add constraint C12 ("these two fields MUST stay in sync"). Not in scope here.
- **R8 ‚Äî Container-side re-validation on read (S11 acknowledgment).** The provider factory on container startup reads `container.json.providerConfig` and consumes the blob without re-parsing against its `configSchema`. If `container.json` is hand-edited or mutated by a future self-mod path, the factory gets unvalidated input. Low cost to fix in the factory contract: "Factories MUST `configSchema.parse()` their slice before use." Noted here; recommend implementers land it with this PR if the provider-side changes are already being touched, but do not block on it.
