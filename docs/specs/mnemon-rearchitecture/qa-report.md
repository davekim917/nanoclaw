### From swarm re-run (after the F3 partial fix)

4. **Sweep path also needed symlink protection** [HIGH, security-reviewer]
   `src/memory-daemon/index.ts:72` was using `statSync` (follows symlinks) while the watcher used `lstatSync`. Cross-tenant symlink attack via the 60s sweep path. **Fix**: replaced with `lstatSync` + `isSymbolicLink` + `realpathSync` validation against `inboxRealPath`. Now mirrors the watcher's defense.

5. **writeSessionMessage await cascade** [CRITICAL, adversarial+concurrency+domain]
   `writeSessionMessage` was made async to preserve recall ordering, but **8 of 9 production callers** still fired-and-forget the now-async function. Container wakes raced the recall+inbound write. The original F3 fix only covered `router.ts:701`. **Fix**: added `await` to all 8 remaining callers + made enclosing functions async — `writeSystemResponse` (now async), `notifyAgent` in `primitive.ts` (now async), `notifyAgent` in `create-agent.ts` (now async), `notify` closure in `response-handler.ts` (now async), plus call sites in `interactive/index.ts`, `self-mod/apply.ts`, `agent-to-agent/agent-route.ts`, `scheduling/actions.ts`. Also updated `ApprovalHandlerContext.notify` type signature to `Promise<void>` and propagated `await notify(...)` to all 5 call sites in `self-mod/apply.ts` + `response-handler.ts`. All 411 tests still pass.

6. **SourceIngester watcher silently no-op'd** [CRITICAL, adversarial+domain]
   `reconcileWatchers`'s `setImmediate` callback called `processInboxFile(..., getIngestDb())`, hitting the legacy Database-only branch in the dual-overload signature, which short-circuits without calling the classifier. Files dropped into the inbox were silently skipped by the watcher fast-path; they only got processed on the 60s sweep. **Fix**: added `setRuntime(store, health)` to `SourceIngester`, called from `index.ts` startup. Watcher's `setImmediate` now passes the real `MemoryStore`. Tests still pass via the existing test seam.

7. **Recall content lacked untrusted-data boundary** [HIGH, security-reviewer]
   `formatRecallContext` injected raw fact content into a `kind: system` message with no boundary marker. Per OWASP ASI06 (Memory and Context Poisoning) and the Palo Alto Unit 42 Bedrock demo, an attacker-controlled fact could carry a "[System: ignore previous instructions ...]" payload that the agent would treat as authoritative. **Fix**: wrapped recalled facts with `RECALL_PREAMBLE` framing + `<recall-data>` boundary tags telling the model to treat the block as untrusted reference data, not instructions.

## Remaining SHOULD-FIX (non-blocking; user discretion)

- **Idempotency cache persistence** (Codex F8, concurrency-reviewer) — `MnemonStore.idempotencyCache` is an in-memory Set, not persisted to `mnemon-ingest.db`. On daemon restart, retries can re-write already-successful facts. Mitigation: the classifier's deterministic per-fact idempotency key (sha256 of `pair_key|fact_index|versions`) means re-writes are content-identical, and mnemon's graph layer dedupes on content. Documented as known follow-up.
- **inotify partial-write race** (concurrency-reviewer) — `setImmediate` in the watcher dispatcher only defers to next event loop tick, not until file write fully flushes. Mitigation: classifier reads file content via `readFileSync` after the watcher fires; partial reads result in a hash mismatch with the eventual full-write hash, so the file gets re-classified on the 60s sweep. Acceptable.
- **`processed_pairs` INSERT + `deleteAfterSuccess` not in same transaction** (concurrency-reviewer) — crash-between leaves dead_letter row pointing at a now-classified pair. Cost: one wasted Anthropic API call on next sweep when the retry runs. Documented as a perf issue, not a correctness one.
- **WebFetch hook size cap** (domain-reviewer) — `createMemoryCaptureBashHook` writes raw response bodies with no size limit. Bounded only by disk; a large `gws gmail search` could write a multi-MB file to the inbox. Mitigation: classifier respects context window (200K for Haiku 4.5).
- **Daemon recall-failure health signal** (domain-reviewer) — when `memoryEnabledForGroup` filesystem read fails, only a `log.warn` fires; no `memory-health.json` field. Defense-in-depth.
- **Anthropic prompt caching** (domain-reviewer) — classifier system prompt is ~400 tokens repeated per call. Adding `cache_control: { type: 'ephemeral' }` would save ~80% input tokens. Cost optimization, not correctness.
- **systemd hardening — broader RestrictAddressFamilies/RestrictNamespaces/MemoryDenyWriteExecute** (security-reviewer) — added NoNewPrivileges/ProtectSystem/ProtectHome/PrivateTmp/ReadWritePaths in initial pass. Additional hardening directives are defense-in-depth; the existing path-traversal protections are the primary control.
- **Mid-turn poll-loop recall_context filter** (adversarial-reviewer) — `trigger !== 1 && !(m.kind === 'system')` lets any system-kind message through if it has subtype `recall_context`. Practical risk: low, since recall_context is host-written only and the container can't write to inbound.db. Documented as worth-tightening to also check `m.id.startsWith('recall-')`.

## Escalations (PRE-EXISTING or design-deferred — user judgment)

- **F1 [CRITICAL, PRE-EXISTING]** Container has `/workspace/archive.db` mounted with ALL groups' messages (`src/container-runner.ts:573-575`). Pre-existing in the codebase, not introduced by this PR. The rearchitecture's recall feature relies on archive.db being read-only-via-tool. Per-tenant archive copy or host-mediated thread-search API is the long-term fix. **Recommend follow-up PR.**
- **F5** `block-mnemon-real-hook` substring matcher bypass — defense-in-depth on top of the wrapper's `MNEMON_READ_ONLY=1` enforcement. The mount narrowing is the primary control. Tightening would require either a stronger matcher (regex on quoted/script paths) or filesystem-level enforcement (`chmod 700 mnemon-real`).
- **F6 / K8** Source-file prompt injection — recall-context boundary now mitigates the recall path (see Fix #7 above). The remaining gap is that source-ingest's classifier is still fed untrusted content as a user prompt; a malicious source file could persuade the classifier to extract instructions disguised as "facts". The recall-side framing now blocks the agent from executing those, but the persistence of attacker-influenced content in the store remains.

## Codex round 2 (post-fix adversarial)

After applying the swarm + Codex round 1 fixes above, ran Codex once more against the now-larger diff. 5 new findings; 4 substantive, 1 was the pre-existing F5 escalation (already documented).

19. **Source-ingest mirror of classifier action-result bug** [CRITICAL]
    `src/memory-daemon/source-ingest.ts` had the same silent-success bug fixed in classifier.ts: `remember()` returning `action: 'skipped'` (or any non-add/update/replace result) without a `factId` was being counted as written, hiding mnemon write failures. **Fix**: mirrored the classifier action check — only `'added' | 'updated' | 'replaced'` count as success; everything else routes to dead_letters with the actual error context.

20. **TOCTOU residual race in processInboxFile** [HIGH]
    The earlier round-1 fix added `lstatSync` + `realpathSync` validation before reading, but still called `readFileSync(filePath)` afterward — a tiny window where a container with write access to its inbox could swap the file for a symlink. **Fix**: switched to `openSync(filePath, O_RDONLY | O_NOFOLLOW)` + `fstatSync(fd)` + `readlinkSync('/proc/self/fd/<fd>')` prefix check + chunked `readSync` from the fd. The kernel itself refuses to follow a symlink at open time, and the fd's `/proc/self/fd` readlink is the canonical post-resolution path. No re-resolution, no race.

21. **Memory-capture hooks fired without MNEMON_STORE** [MEDIUM]
    `container/agent-runner/src/providers/claude.ts` registered the WebFetch + Bash PostToolUse hooks unconditionally. Groups without memory enabled (no `MNEMON_STORE` env var) would still write capture files to nonexistent inbox paths and emit error logs every WebFetch. **Fix**: hook registration gated on `process.env.MNEMON_STORE` — only memory-enabled groups install the capture hooks.

22. **extractRouting picks recall_context as reply anchor** [MEDIUM]
    `formatter.ts:113` used `messages[0]` to derive routing, including `inReplyTo`. Since recall_context system rows are inserted before their paired inbound message, they could become the first row in the batch — making outbound replies attach to `recall-<messageId>` instead of the real user message. **Fix**: anchor selection now skips `kind === 'system'` rows (`messages.find(m => m.kind !== 'system') ?? messages[0]`). Added behavioral test asserting that a `recall-m1` system row preceding a chat row `m1` produces `inReplyTo === 'm1'`. All 17 poll-loop.test.ts tests pass.

## Daemon endpoint wiring (F3) — fixed end-to-end after pushback

User pushed back on the "F3 critical but out of scope, defer" framing. Investigated and fixed end-to-end. The daemon now actually works against api.anthropic.com — smoke-tested 200 OK with substituted credentials.

**Discoveries during the fix:**
1. The default URL `http://127.0.0.1:10254/api/anthropic` was wrong — 10254 is the OneCLI Next.js dashboard, returns 404. The actual gateway HTTPS-CONNECT proxy is on 10255.
2. Wrapping ExecStart in `onecli run` injects HTTPS_PROXY=http://x:<vault-token>@127.0.0.1:10255 + `CLAUDE_CODE_OAUTH_TOKEN=placeholder` + CA cert env. The proxy substitutes the literal "placeholder" Bearer with the operator's real OAuth token at request time.
3. **Node 20's native `fetch` ignores HTTPS_PROXY** — `NODE_USE_ENV_PROXY` is a Node 24+ feature even though `onecli run` sets the env var. Without a manual proxy dispatcher, requests bypass the gateway entirely.
4. The OneCLI vault's Anthropic secret is configured as type=anthropic which maps to OAuth Bearer auth (matching `CLAUDE_CODE_OAUTH_TOKEN`), not x-api-key.

**Fix:**
- `src/memory-daemon/anthropic-client.ts` — added `import { EnvHttpProxyAgent, fetch as undiciFetch } from 'undici'`. Lazy-init dispatcher inside callClassifier (not at module load — see Codex round 4 F1 below). EnvHttpProxyAgent honors NO_PROXY for non-OneCLI deployments.
- Auth precedence: prefers `x-api-key` when ANTHROPIC_API_KEY is set; falls back to `Authorization: Bearer ${CLAUDE_CODE_OAUTH_TOKEN}` with `anthropic-beta: oauth-2025-04-20` when only OAuth is set.
- `data/systemd/nanoclaw-memory-daemon.service` — ExecStart now `/home/ubuntu/.local/bin/onecli run -- /usr/bin/env node dist/memory-daemon/index.js`. Comment block documents the operator prerequisite (default OneCLI agent must have an Anthropic vault entry assigned).
- `package.json` — `undici@6.24.1` added as direct dep (was previously transitive through discord.js).

**End-to-end smoke test** (run from repo root): `onecli run -- node smoke.mjs` against api.anthropic.com via the daemon's exact code path returned `status: 200` with `"text":"OK"`. Verified twice (once with the initial wiring, once after Codex round 4 follow-ups).

### Gap 2 — Wiki synthesis ran on the agent's default model (Sonnet)
