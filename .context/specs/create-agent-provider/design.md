# Design: `create_agent` provider + per-provider config

> Generated by `/team-design` — 2026-04-24
> Saved to: `.context/specs/create-agent-provider/design.md`

## Brief Reference

Extend the `create_agent` MCP tool so an admin agent can pick a provider (Claude or Codex initially; OpenCode and others as one-file follow-ups) and pass a provider-specific `provider_config` object (`{model?, effort?}` for Claude; `{model?, reasoning_effort?}` for Codex). Validation must happen before any persistent state is written; config must be sticky across container restarts; existing callers (no `provider`, no `provider_config`) keep current behavior. PR-ready against `qwibitai/nanoclaw` upstream.

Full brief: `.context/specs/create-agent-provider/brief.md`.

### Flagged Constraints

- **C4 — "Schemas live in `src/providers/<name>.ts`":** The brief specified the host path `src/providers/<name>.ts`. Research (C11) shows the container is the source of truth for provider availability and already has `zod` available. Co-locating schemas with the **container-side** provider files (`container/agent-runner/src/providers/<name>.ts`) better satisfies the *spirit* of C4 (schema lives with the provider, no central enum) while avoiding the validation-surface mismatch the brief did not anticipate. The brief text *"or co-located there"* leaves room for this refinement, but it is a deliberate choice that should be visible. Documented as decision **D4**.
- **C6 — Fire-and-forget narrowed:** Validation errors return *synchronously* via the MCP `isError` content block — no system message is ever written for invalid calls. Fire-and-forget semantics still apply to the *creation* step (success notification arrives via `notifyAgent`). This is stricter than the brief, not looser, and trivially satisfies C1.

## Research Summary

- **Project patterns:**
  - Two parallel registries: container-side `provider-registry.ts` (`AgentProvider` factories — Claude, mock baseline; Codex/OpenCode added by `/add-*` skills) and host-side `provider-container-registry.ts` (only providers needing mounts/env — Claude/mock omitted) — `src/providers/provider-container-registry.ts:1-26`.
  - The `create_agent` flow today: container MCP handler writes a `system` message via `writeMessageOut`; host's `delivery.ts` dispatches the action to `handleCreateAgent`; that handler hardcodes `agent_provider: null` (`src/modules/agent-to-agent/create-agent.ts:81`) and calls `notifyAgent` to surface success.
  - Per-group config persists in `groups/<folder>/container.json`, mounted read-only at `/workspace/agent/container.json`. Existing fields include `defaultModel` and `defaultEffort` (Claude-specific; flow into env vars at spawn — `src/container-runner.ts:1206-1224`). The container runner reads container.json once at startup via `container/agent-runner/src/config.ts:29`.
  - The container's `RunnerConfig` interface (`container/agent-runner/src/config.ts:12`) currently exposes only `provider`, `assistantName`, `groupName`, `agentGroupId`, `maxMessagesPerPrompt`, `mcpServers`. New fields (e.g., `providerConfig`) would extend this.
  - `AgentProvider` interface (`container/agent-runner/src/providers/types.ts:1`) and `QueryInput` (line 58): per-query `model` and `effort` are first-class fields the SDK passes to its query call; Claude provider already honors them (`claude.ts:824-825`).
  - Active churn: most recent commit on this module was `46b19dc refactor(modules): extract agent-to-agent as registry-based module` — registry pattern is current best practice in this area. Match it.
- **Library docs:**
  - `zod@^4.0.0` already a runtime dep in `container/agent-runner/package.json`. Not yet imported anywhere in container source — this work would be the first user.
  - Claude Agent SDK `effort` field: first-class since Opus 4.6 per the inline comment at `types.ts:88`. Verified by reading the existing claude provider — `effort: input.effort as EffortLevel` is passed straight through (`claude.ts:825`).
  - Codex CLI honors `model` and `model_reasoning_effort` via `~/.codex/config.toml`; the host-side codex provider already mounts a per-session `~/.codex` directory (per `/add-codex` skill `SKILL.md:140`). For `CODEX_MODEL`, env-var passthrough is also supported.
- **Existing code (key cites for design decisions):**
  - `container/agent-runner/src/mcp-tools/agents.ts:31-66` — current `create_agent` MCP handler. ~35 lines, clean to extend.
  - `src/modules/agent-to-agent/create-agent.ts:37-126` — current host handler. The `agent_provider: null` line is the one this work flips.
  - `src/container-config.ts:57-143` — `ContainerConfig` interface; this is where a `providerConfig` field would land.
  - `src/container-runner.ts:417-440` — provider name resolution (`resolveProviderName` + `resolveProviderContribution`). No change needed; new code consumes this unchanged.

## Options

### Option A: Host-side validation; schemas in `src/providers/<name>.ts` (literal brief reading)

**Approach:** Add a host-side metadata registry that holds Zod schemas alongside the existing container-config functions. Each provider's `src/providers/<name>.ts` registers both. Add a baseline `src/providers/claude.ts` (currently missing) that registers only a schema. `handleCreateAgent` validates `provider` against the registry and `provider_config` against the schema before any DB write; failures send `notifyAgent` error.

**Pros:**
- Validation happens in the same process that writes the DB row — straightforward transaction boundary.
- Strict literal compliance with brief constraint C4's path.

**Cons:**
- Requires adding `zod` to root `package.json` (host) — subject to the project's `minimumReleaseAge: 4320` rule. `zod@4.x` has been GA for months, so this is likely fine, but it's a new top-level dep.
- Requires creating a new `src/providers/claude.ts` whose only purpose is schema registration — the host-side claude file does not exist today because Claude needs no host-side container-config (`src/providers/index.ts` comment block lines 1-7).
- Validation feedback arrives **asynchronously** via `notifyAgent` (the MCP handler has already returned `ok` by the time validation runs). The agent sees "Creating agent…" and then a separate error message — confusing UX.
- Two provider registries to keep aligned: container-side knows what can run; host-side knows what's valid. Drift risk.

**Pattern alignment:** Mirrors `src/channels/channel-registry.ts` and the existing `provider-container-registry.ts` pattern, but in a parallel new file. Brief C4 literal compliance.

**Confidence:** Medium — the new `zod` host dep adds friction with the supply-chain rule (would land but adds a verification step), and the dual-registry drift problem is a long-term liability.

**Assumptions:**
- `zod@4.x` passes the `minimumReleaseAge` gate at install time. (Likely true; verifiable via `pnpm view zod time`.)
- Claude is fine to "fake-register" host-side just for schema purposes.

### Option B: Container-side validation; schemas co-located in `container/agent-runner/src/providers/<name>.ts` (recommended)

**Approach:** Each container-side provider module exports both its factory (existing) and an optional Zod `configSchema` for `provider_config`. The container's `provider-registry.ts` gains a `getProviderConfigSchema(name)` lookup. The `create_agent` MCP handler in `agents.ts` validates `provider` against `listProviderNames()` and `provider_config` against the schema **before** writing the system action — failures return synchronously as MCP `isError` content. On success, the action payload (now including `provider` and the validated `provider_config`) is sent to the host, which sets `agent_groups.agent_provider` and writes `providerConfig: Record<string, unknown>` into the new agent's `container.json`. The new container reads `container.json.providerConfig` at startup; the provider's factory or `query()` translates it into per-query SDK options.

**Pros:**
- **No new host dependency.** `zod@^4.0.0` is already a runtime dep in `container/agent-runner/package.json`. (Constraint C7.)
- **Single source of truth for "what providers are valid."** The container's registry is, by construction, the set of providers the running container image can actually instantiate. Host-side mirroring becomes unnecessary.
- **Validation feedback is synchronous.** The agent sees the `isError` content block in the same turn — no fire-and-forget back-channel for errors. C1's "no orphan state" is satisfied trivially because no system message is ever written for a failed validation.
- Schema is **truly co-located** with the provider runtime (factory and schema in the same file). Skill installation pattern unchanged: `/add-codex` already drops a file at `container/agent-runner/src/providers/codex.ts` — that file just gains a schema export.
- Per-group persistence uses a single new field (`providerConfig`) on `container.json`; no new DB column or migration.

**Cons:**
- Schema location differs from brief C4's literal text (host-side path). Treated as a deliberate refinement (see Flagged Constraints) but reviewers should agree.
- Container-side validation means a stale container image (skill installed on host but container not yet rebuilt) will reject the new provider. This is *correct* behavior (the container can't actually run it) but the error is "Unknown provider: codex" rather than "Run /add-codex" — message wording matters.
- Adds a generic `providerConfig?: Record<string, unknown>` blob to `ContainerConfig`. Slightly less type-safe at the host layer than dedicated fields would be — but the only consumer is the matching provider, which validates again on read.

**Pattern alignment:** Matches the existing container-side registry pattern (`container/agent-runner/src/providers/provider-registry.ts:1-33`) and skill-install convention (`/add-codex` SKILL.md). Preserves brief C4's *intent* (schema co-located with provider, no central enum) at a different physical location.

**Confidence:** High — every assumption is grounded in code reads, no new deps, the validation surface mismatch from C11 is resolved correctly, and the change is the smallest viable PR for upstream.

**Assumptions:**
- A1 — `container.json.providerConfig` field added to `RunnerConfig` in `container/agent-runner/src/config.ts` is sufficient for the container to receive per-group config without env-var plumbing on the host. **Validation:** Confirmed by reading `config.ts:29-49` — the loader already maps fields from the JSON file directly.
- A2 — Setting `agent_groups.agent_provider` (instead of `null`) does not break any existing read path. **Validation:** Confirmed by reading `resolveProviderName` (`container-runner.ts:417-423`) — non-null values are honored, and the existing migration column already accepts arbitrary text.

### Option C: Hybrid — container-side validation + map Claude config to existing `defaultModel`/`defaultEffort` fields

**Approach:** Same as Option B for validation (container-side, synchronous error), but on the host's persistence side, special-case Claude: write `provider_config.model` to `container.json.defaultModel` and `provider_config.effort` to `container.json.defaultEffort` (the existing fields that already plumb through env vars in `container-runner.ts:1206-1224`). Other providers use the new generic `providerConfig` blob.

**Pros:**
- Reuses existing wiring for Claude — `defaultModel` and `defaultEffort` already flow into the right env vars; nothing new to test for the Claude path.
- `container.json` for a Claude agent stays "shaped like every other Claude agent on this host" — easier for someone reading the JSON to understand.

**Cons:**
- **Asymmetric.** Claude config takes a host-side translation path; everyone else uses a generic blob. Two persistence paths to maintain forever.
- The existing `defaultModel` field is documented as "for the bare `opus` alias" specifically (`container-config.ts:113-118`) — semantic mismatch with arbitrary user-supplied models like `claude-sonnet-4-6`. Either we change the field's documented purpose (bigger change) or we accept the drift.
- Future Claude knobs (e.g., thinking budget) would either need new dedicated fields *or* fall into the generic blob anyway, defeating the symmetry argument.

**Pattern alignment:** Partially matches existing patterns (Claude path), partially introduces new pattern (`providerConfig` blob). Less coherent than Option B.

**Confidence:** Medium — works, but the asymmetry creates drift risk and the existing field's documented purpose doesn't quite fit.

**Assumptions:**
- The "for the bare `opus` alias" semantics of `defaultModel` can be quietly broadened. (Probably yes, but it's a documentation/intent change.)

## Recommendation

**Selected:** Option B.

**Justification:**

- Resolves the validation-surface mismatch (C11) at the right layer: the container is the actual source of truth for what providers can run, and validating there means we never lie to the agent about what's possible.
- Zero new host dependencies (C7 satisfied trivially); container already has `zod`.
- Synchronous validation feedback in the same MCP turn — no fire-and-forget back-channel for the error path. C1 satisfied trivially because no system message is ever written for a failed call.
- Symmetric persistence model: every provider's config rides on the same `providerConfig` field; no two-track translation. Future providers (OpenCode and beyond) plug in identically.
- Smallest realistic diff for an upstream PR: 4-5 files touched, no new tables, no new top-level deps.
- Matches the upstream churn direction (the most recent agent-to-agent commit was a registry-extraction refactor — the registry-based pattern is what the maintainers are currently building toward).

**Refinement of brief constraint C4:** Schemas live in `container/agent-runner/src/providers/<name>.ts` (container-side path) rather than `src/providers/<name>.ts` (host-side path). The brief's text *"or co-located there"* leaves room; the spirit (schema with provider, no central enum) is preserved. This is **decision D4** below.

**Confidence:** High. All assumptions in the Assumptions Log are validatable with code reads or short test runs; no spike needed.

## Risks & Open Questions

- **R1 — Stale-image error message wording.** If the host has `/add-codex` installed but the container image hasn't been rebuilt, validation will say "Unknown provider: codex." This is correct but could read as a bug to the agent. Mitigation: the error message includes `Registered: <list>` (already in `provider-registry.ts:25`), which makes the diagnosis obvious. If we want a friendlier message, we can add a host-side preflight check that warns when `src/providers/codex.ts` exists on disk but `listProviderNames()` doesn't include `codex` — out of scope for this PR.
- **R2 — Provider-side translation contract.** Each provider must define how `provider_config` translates into its SDK call. Claude maps cleanly (`model` → `model`, `effort` → `effort` — already first-class). Codex needs to map `model` → `CODEX_MODEL` env (or `~/.codex/config.toml`) and `reasoning_effort` → `model_reasoning_effort` in `config.toml`. The codex provider file (added by `/add-codex`) is responsible for this; the contract is "read your slice of `providerConfig`, do whatever you need to apply it." Documented in the design but enforced by per-provider tests.
- **Q1 — Future overrides at session/turn level.** The brief explicitly defers per-session and per-turn overrides. The `providerConfig` blob in `container.json` is the per-group sticky layer. Sessions and turns would layer on top later — not addressed here, but the design does not preclude them.
