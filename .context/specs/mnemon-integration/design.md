### Flagged Constraints

- **C3 — "ContainerConfig zod schema":** Brief says "zod schema" but verified at `src/container-config.ts:1-218` that the codebase uses **plain TypeScript interfaces with a manual property whitelist** in `readContainerConfig` (lines 188-213) — NOT zod. Unknown keys silently drop because `readContainerConfig` enumerates explicit fields one-by-one rather than a schema validator passing them through. **Spirit of the constraint is correct** (must explicitly add `mnemon` field or it disappears on round-trip), but the **mechanism** the brief named was wrong. Implementation: extend the `ContainerConfig` interface AND add the explicit property in the `readContainerConfig` whitelist. No zod migration in scope.

- **C11 — host-gateway wiring:** Linux Docker requires `--add-host=host.docker.internal:host-gateway` for containers to reach host services. Verified at `src/container-runner.ts:360-361` that `NO_PROXY` is being set per-container, but I did not find an `--add-host` flag in the spawn args. **Validation needed during build:** does v2 already pass `--add-host=host.docker.internal:host-gateway`, or do we need to add it? Likely add it as part of `applyMnemonEnv` when `mnemon.embeddings === true`.

## Research Summary

**Project patterns** (from CLAUDE.md and `src/modules/agent-to-agent/`):
- Modules under `src/modules/<name>/` follow shape: `index.ts` (public API), feature TS files, optional `db/` subfolder, colocated `*.test.ts`.
- Module exports a small surface; container-runner.ts and other callers import via `./modules/<name>/index.js`.
- Self-mod config mutations route through `updateContainerConfig(folder, mutate)` at `src/container-config.ts:237` — the canonical TS path for any container.json change.
- Hook delivery in `providers/claude.ts:947` uses the SDK's `hooks` parameter (object keyed by event name, value is array of `{ matcher?, hooks: [...] }`). Existing hooks include `createSanitizeBashHook`, `createPreCompactHook(assistantName)`, etc. — all imported from sibling files and called as factories.
- Dockerfile uses `ARG <NAME>_VERSION=x.y.z` followed by a `RUN curl -fsSL ... -o /tmp/X` install block; pattern at lines 118-127 (RENDER/RAILWAY/SUPABASE).

**Mnemon CLI surface** (verified earlier via `mnemon --help`):
- Commands: `remember`, `recall`, `link`, `related`, `search`, `forget`, `gc`, `embed`, `viz`, `status`, `store`, `setup`.
- Default data dir `~/.mnemon`. Per-store isolation via `--store <name>` or `MNEMON_STORE` env.
- `mnemon status` outputs JSON by default (verified earlier — no `--json` flag needed; structure includes `total_insights`, `edge_count`, `db_size_bytes`, `top_entities`, etc.).
- Embeddings via `MNEMON_EMBED_ENDPOINT` (default `http://localhost:11434`) and `MNEMON_EMBED_MODEL` (default `nomic-embed-text`).
- `mnemon setup --target` supports `claude-code` and `openclaw` only — not `nanoclaw`. Hook scripts (Prime/Remind/Nudge) deployed by setup are at `.claude/hooks/mnemon/{prime,user_prompt,stop}.sh` (verified earlier in `/tmp/mnemon-test/`).

**Ollama nomic-embed-text** (Ollama docs, T1):
- Endpoint: `POST /api/embeddings` with `{ "model": "nomic-embed-text", "prompt": "..." }`.
- Model size: 274MB. Pulled via `ollama pull nomic-embed-text`.
- Linux install: `curl -fsSL https://ollama.com/install.sh | sh` (root, installs systemd service).

**SQLite WAL + concurrent writers** (sqlite.org docs, T1):
- WAL allows multiple concurrent readers + one writer; no concurrent writers without `busy_timeout` retry.
- Recommended `busy_timeout` for multi-process writers: 5000ms+. Mnemon v0.1.2's DSN has WAL but no `busy_timeout` (Codex verified by reading the binary).
- `flock` (POSIX advisory lock) at the OS layer is the standard mitigation when the application doesn't set `busy_timeout` itself.

**RRF hybrid retrieval** (Microsoft Azure AI Search docs, T1):
- Default fusion algorithm for combining BM25 + vector scores.
- Mnemon implements its own variant (verified in mnemon README); fully internal — we don't need to configure RRF, just enable embeddings.

## Options

### D1 — Hook implementation strategy [REVISED in cycle 1]

**Critical correction from cycle 1 review:** mnemon's `user_prompt.sh` is a 5-line static reminder (`echo "[mnemon] Evaluate: recall needed? After responding, evaluate: remember needed?"`). It does NOT call `mnemon recall`. The LLM is the one that decides to invoke `mnemon recall` via Bash tool calls, prompted by the static reminder. Earlier draft of D1 was based on a false premise that the hook itself ran recall.

Given this, the hook has two responsibilities only:
1. Inject the reminder text into the LLM's context (Prime via SessionStart, Remind via UserPromptSubmit).
2. Emit per-turn metric rows (Stop hook, side-effect-only).

No mnemon CLI invocation from the hook itself — the LLM does it through Bash.

#### Option A: Pure TypeScript hook callbacks (selected)

Three TS hook factories in `container/agent-runner/src/modules/mnemon/hooks.ts`:

- `createMnemonPrimeHook(store, phase)` — SessionStart hook. Returns `additionalContext` containing the mnemon guide reference + current phase ("Phase 1 shadow" or "Phase 2 live"). Loaded from `~/.mnemon/prompt/guide.md` mounted RO.
- `createMnemonRemindHook(store, phase)` — UserPromptSubmit hook. Returns `additionalContext` with the reminder text. In Phase 1: text is "[mnemon] Phase 1 shadow — only consider remember; recall not yet active." In Phase 2: text is mnemon's standard "[mnemon] Evaluate: recall needed?...".
- `createMnemonNudgeHook(store)` — Stop hook. **Side-effect only** (see D7 + MF3). Emits one row to per-container metrics file. Does NOT return `additionalContext` (SDK doesn't support it on Stop).

**Pros:**
- TS-native, matches existing hook style in `claude.ts:947`.
- No Bash dependency surface in hot path beyond mnemon CLI calls the LLM makes (which were always going to happen).
- Easy to test via vitest with mocked SDK input.

**Pattern alignment:** Matches `createPreCompactHook(assistantName)` factory at line 965. Same shape.

**Confidence:** High — verified mnemon's actual hook script behavior in cycle 1 fact-check.

**Assumptions:**
- `~/.mnemon/prompt/guide.md` content is stable across mnemon patch versions (validated A7 → re-validate on MNEMON_VERSION bump in /update-container schema migration step).

#### Option B: Bake mnemon's .sh scripts and shell out *(REJECTED)*

Considered as a way to track upstream behavior automatically. **Rejected** because the scripts are 5-20 lines of `echo` + `cat`; baking them adds shell layer with no fidelity benefit (there's no recall logic in them to "preserve").

### D2 — Shadow-mode toggle mechanism

Phase 1 (week 1) writes recall results to `shadow.log` instead of injecting them. Phase 2 (Day 8 onward) injects normally. How is the shadow boolean expressed?

#### Option A: container.json field `mnemon: { shadow: true }`

Add a third field to the mnemon block. Hook reads it via the spawn-time env (host writes `MNEMON_SHADOW_MODE=true` based on container.json).

**Pros:**
- Per-group toggle, visible in the same place the operator already configures mnemon.
- Operator can flip it via `pnpm exec tsx scripts/mnemon-phase2.ts <group>` (a tiny script that flips one boolean and triggers respawn).

**Cons:**
- Adds a third field to the mnemon block; brief said "exactly { enabled, embeddings }" (C9). **Constraint violation.**

**Verdict:** Rejected on C9.

---

#### Option B: Separate `shadow_mode` file flag at `~/.mnemon/data/<store>/.shadow_mode`

When the file exists, hooks operate in shadow mode. `enable-mnemon.ts` creates it on rollout; a separate `phase2-mnemon.ts` script removes it after the week.

**Pros:**
- Doesn't modify container.json schema — keeps C9 clean.
- Trivial implementation: the hook just checks `fs.existsSync(SHADOW_FLAG_PATH)`.
- Operator can flip without a container respawn (next session sees the new state).

**Cons:**
- Toggle state lives outside the per-group source-of-truth (container.json). Slightly less discoverable.
- Filesystem state vs config state inconsistency.

**Pattern alignment:** Filesystem flags are used elsewhere in v2 (e.g., heartbeat at `/workspace/.heartbeat`). Established pattern.

**Confidence:** High.

---

#### Option C: Auto-graduation by enable-time timestamp

`enable-mnemon.ts` writes the enable timestamp into a per-store metadata file. Hooks check `(now - enable_time) > 7 days` to decide shadow vs inject.

**Pros:**
- Auto-graduates without any operator action on Day 8.
- Self-documenting timeline.

**Cons:**
- **Violates C7's "Phase 2 transition is gated on the success criteria"** — auto-graduation flips Phase 2 even if gates don't pass.
- Operator loses the "decide whether to graduate" moment, which is the whole point of the gates.

**Verdict:** Rejected on C7.

### D3 — flock serialization shape

C16 requires per-store flock around mnemon writes to prevent SQLite contention. Where does the lock acquisition live?

#### Option A: TS hook function acquires flock before exec'ing mnemon

In the Remind/Nudge hook callbacks, acquire `~/.mnemon/data/<store>/.write.lock` via `proper-lockfile` (or similar) before shelling out to mnemon. Release after the CLI exits.

**Pros:**
- Fine-grained — only writes are serialized, reads (recall) proceed unblocked.
- TS-native, easy to test.
- No host-side daemon to maintain.

**Cons:**
- Adds an npm dependency (or we use bare `flock` shell command).
- Hook latency includes lock acquisition time (typically <1ms uncontended, more under contention).

**Pattern alignment:** v2 already uses file locks elsewhere (e.g., `.claude/scheduled_tasks.lock` per .gitignore). Consistent.

**Confidence:** High.

---

#### Option B: Shell `flock` command in front of every mnemon CLI call

`flock /path/to/.write.lock mnemon remember ...` — POSIX `flock(1)` utility wraps the call in a lock on the file descriptor.

**Pros:**
- No npm dependency.
- Standard POSIX tool, already on Linux containers.
- One-liner change.

**Cons:**
- Same as A but slightly less testable (harder to assert lock behavior in unit tests).

**Pattern alignment:** Lower — v2's existing locks are TS-managed via the npm `proper-lockfile` ecosystem; using shell flock would introduce a new pattern.

**Confidence:** High.

---

#### Option C: Single host-side mnemon-write daemon serializes ALL writes globally

A small Node process listens on a unix socket; hooks send writes to it; daemon serializes and shells out to mnemon.

**Pros:**
- Most rigorous — guaranteed total ordering of writes.

**Cons:**
- **Single point of failure across all groups.** If the daemon dies, all mnemon writes hang or fail.
- Adds infrastructure to maintain.
- Per-store flock already gives us per-store ordering; cross-store ordering isn't required.

**Verdict:** Rejected — overengineered for the problem.

### D4 — Scheduled task creation by enable-mnemon.ts [REVISED in cycle 1]

**Cycle 1 corrections:**
- The path `data/v2-sessions/<agent_group_id>/inbound.db` IS valid (filesystem confirms — `ls data/v2-sessions/ag-1776377699463-2axxhg/` shows `inbound.db` at the agent_group level alongside `sess-*` subdirs). Reviewer A's claim that this path doesn't exist was wrong.
- However, the design treated `wire-scheduled-tasks.ts` as scheduling API. It's not — it's a one-shot setup script. Need to extract the insertion logic into a reusable function.

#### Option A: Refactor wire-scheduled-tasks.ts to expose `scheduleTask(taskDef)` (selected)

Extract the existing insert logic from `wire-scheduled-tasks.ts` (per Reviewer A's reading at lines 471-529 — the insertion machinery) into a function exported from `src/db/scheduled-tasks.ts` (or similar). Both `wire-scheduled-tasks.ts` and `enable-mnemon.ts` call it.

```ts
// src/db/scheduled-tasks.ts (new)
export interface TaskDef { id, groupFolder, agentGroupId, platformId, channelType, cron, processAfter, seriesId, prompt }
export function scheduleTask(def: TaskDef): void {
  // Open agent_group-level inbound.db, idempotent insert/upsert via series_id
```

Both call sites use the same idempotent insert. `enable-mnemon.ts` calls it three times (synth/GC/reconcile).

**Pros:**
- Single source of truth for the scheduling insert; idempotency lives in one implementation.
- Direct, no LLM round-trip cost.
- If v2's scheduled-task storage shape changes, one file moves; both call sites pick it up.

**Cons:**
- Requires a small refactor of wire-scheduled-tasks.ts (in scope for this build).

#### Option B: Direct duplicated insert *(REJECTED — was prior selection)*

### D5 — Observability emit cadence

Per-group metrics into `data/mnemon-metrics.jsonl`. When are rows appended?

#### Option A: Host-side periodic script (every 5 min) reads `mnemon status` per enabled group

A small Node script runs via systemd timer or cron every 5 min, iterates groups with `mnemon.enabled === true`, runs `mnemon status --store <name>`, parses JSON, appends row to `data/mnemon-metrics.jsonl`.

**Pros:**
- Decouples observability from the hot path — no per-turn cost.
- Single source of truth (host filesystem).
- 5-min granularity sufficient for everything except hook-failure rate (which is an aggregate over time anyway).

**Cons:**
- Misses per-turn signal (e.g., individual hook latency). Needs separate per-turn capture for hook timing.
- Adds a new periodic process.

---

#### Option B: Container-side hook on Stop appends one row per turn

The Stop hook writes a row each time a turn completes. Captures: hook fired counts (from the agent's session), hook failures (try/catch in the hook), p95 latency (from per-turn timing).

**Pros:**
- Captures per-turn detail — the only place we can measure per-turn hook latency.
- No separate periodic process needed for that signal.

**Cons:**
- Adds latency to every Stop event.
- File I/O contention across multiple containers writing the same JSONL file (need flock here too, or per-container files merged later).

---

#### Option C: Both — per-turn append from container + periodic snapshot from host

Container Stop hook emits per-turn rows (timing, failure counts). Host periodic script emits per-store rows (DB size, embedding success). Same file, different row types.

**Pros:**
- Captures both per-turn and aggregate signals at appropriate cadences.
- Clean separation: container measures what only it can see; host measures what only it can see.

**Cons:**
- Two writers to the same file → need lock coordination, OR per-source files merged at query time.

**Pattern alignment:** Solved via per-source files at `data/mnemon-metrics-turns.jsonl` (container-written, one per session container) + `data/mnemon-metrics-stores.jsonl` (host-written). Query CLI reads both and joins.

**Confidence:** High.

### D6 — Backup script implementation [REVISED in cycle 1]

C13 explicitly says `sqlite3 .backup` or WAL-checkpoint-then-copy. Cycle 1 review (MF5) flagged that picking better-sqlite3 introduces cross-SQLite-library risk (Go binary's bundled SQLite vs better-sqlite3's bundled SQLite) for marginal benefit. Switching to **Option A** (`sqlite3` CLI) per C13 literal.

#### Option A: Bash + sqlite3 CLI, retention pruning in shell (selected)

```bash
# scripts/mnemon-backup.sh — invoked nightly via cron
# For each store under ~/.mnemon/data/, run sqlite3 .backup and prune retention windows.
DATE=$(date +%Y-%m-%d)
mkdir -p ~/backups/.mnemon-${DATE}
for STORE_DIR in ~/.mnemon/data/*/; do
  STORE=$(basename "$STORE_DIR")
  sqlite3 "${STORE_DIR}/mnemon.db" ".backup ~/backups/.mnemon-${DATE}/${STORE}.db"
done
# Prune: keep last 7 daily; keep one per week for 4 weeks; delete rest.
# (Logic in companion script scripts/mnemon-retention.sh — short, well-tested awk + find)
```

**Pros:**
- Matches C13 literal phrasing.
- Same SQLite library writing as reading the backup (mnemon's own); zero cross-library risk.
- `sqlite3` CLI guaranteed present on Linux containers/hosts.

**Cons:**
- Retention pruning in shell needs care (`find -mtime` + per-week tracking via a manifest file).

**Pattern alignment:** Bash for ops scripts is standard for v2's host-side cron jobs.

**Confidence:** High.

#### Option B: TS wrapper invoking sqlite3 via child_process *(alternative)*

If TS preferred for retention logic, wrap shell sqlite3 from `scripts/mnemon-backup.ts` using `execFile('sqlite3', ['db', '.backup', ...])`. Same backup mechanism, retention logic in TS. Acceptable hybrid; selected if operator finds bash retention painful.

#### Option C: better-sqlite3 backup API *(REJECTED — was prior selection, now reversed)*

Cycle 1 fact-check confirmed cross-library risk wasn't justified. Switching to A.

### D7 — Shadow-mode delivery mechanism [REVISED in cycle 1]

**Critical correction from cycle 1 review:** Earlier draft assumed the hook captured `mnemon recall` output. Cycle 1 fact-check revealed the hook doesn't run recall at all — the LLM does. So D7 is reframed: shadow mode controls **what reminder text the LLM sees**, which determines whether it invokes recall.

#### Option A: Modified reminder text per phase (selected)

The Remind hook's `additionalContext` content depends on the rollout phase:

- **Phase 1 (shadow):** Hook returns "[mnemon] Phase 1 shadow — only consider remember; recall not yet active. Skip recall this turn." LLM doesn't invoke `mnemon recall` because the prompt explicitly tells it not to.
- **Phase 2 (live):** Hook returns mnemon's standard reminder, which prompts recall. LLM invokes `mnemon recall` via Bash naturally.

Phase resolution comes from `data/mnemon-rollout.json` (see SF2 below) — host-mounted RO into container at `/workspace/agent/.mnemon-rollout.json`.

Plus: every `mnemon remember` call still writes facts during Phase 1, so the store accumulates real signal. Operator validates shadow mode by inspecting `mnemon recall <query>` from the shell during the week — same DB, same result; just no LLM-side injection until Day 8.

**Pros:**
- No `shadow.log` file needed — shadow data IS the live mnemon DB; operator queries it directly.
- No Bash interception complexity.
- Phase 2 transition is "operator runs `phase2-mnemon.ts <group>`" which updates the rollout JSON.

**Cons:**
- "Shadow" semantics are slightly different from the original framing (no parallel log; the live DB is the validation surface).
- If the LLM disregards the Phase 1 prompt and calls `mnemon recall` anyway, we'd inject results. Mitigation: the Phase 1 prompt is explicit; agent compliance is the same standard as other behavioral instructions.

Each hook factory lives in `container/agent-runner/src/modules/mnemon/hooks.ts`. Behavior per hook:

### Prime (SessionStart)
- Reads `~/.mnemon/prompt/guide.md` content (mounted from host RW data dir).
- Returns it as `additionalContext`. SDK supports `additionalContext` on SessionStart.
- No CLI shell-out; pure file read + return.

### Remind (UserPromptSubmit)
- Reads phase from `/workspace/agent/.mnemon-rollout.json` (host-mounted RO).
- Returns `additionalContext` containing one of:
  - **Phase 1 (shadow):** "[mnemon] Phase 1 shadow — only consider remember; recall not yet active. Skip recall this turn."
  - **Phase 2 (live):** "[mnemon] Evaluate: recall needed? After responding, evaluate: remember needed?"
- No CLI shell-out; the LLM decides whether to invoke `mnemon recall` itself based on the prompt.

### Nudge (Stop) — side-effect only [MF3 — Stop doesn't support additionalContext]
- Acquires per-store `flock` (write window — measurement-side, not mnemon-side).
- Appends one row to `data/mnemon-metrics/turns/<agent_group_id>__<session_id>.jsonl` with: timestamp, phase, hook latencies, hook fire counts, any failure class observations from this turn.
- Returns `{ continue: true }`. Does NOT return additionalContext (SDK doesn't support it on Stop).

### Common discipline (all three hooks)
1. Time wall-clock (for the latency gate).
2. Catch errors via `failure-class.ts` (SF6):
   - **Recoverable** (Ollama timeout, network blip, transient disk write fail): log WARN, continue with empty/default content.
   - **Blocking** (mnemon binary missing, store DB inaccessible, schema mismatch detected, ≥3 write failures in window): log ERROR, **emit an unhealthy event row to `/workspace/agent/.mnemon-metrics.jsonl`** with `{event_type: "unhealthy", reason: ..., store: ...}`. **Container hooks NEVER write to `data/mnemon-rollout.json`** — that file is host-owned and RO-mounted into the container. The host-side metrics-collector aggregates unhealthy events and writes derived state to `data/mnemon-health.json` per store. Phase 2 graduation reads `data/mnemon-health.json` and refuses if `phase: 'unhealthy'`. (Cycle 3 F1 fix.)
3. Never fail the agent's turn — return safe defaults; the agent doesn't know mnemon exists, only that no extra context arrived.

## Mnemon CLI wrapper (replaces PreToolUse Bash gate) [REVISED in cycle 2 — MF8/MF10/MF11]

**Cycle 2 found** that PreToolUse hooks cannot mutate Bash commands (advisory only — allow/deny + reason), so the prior plan to wrap mnemon calls via PreToolUse was unworkable. Also: a per-container `/tmp/mnemon-<store>.lock` doesn't serialize across containers (each container has its own /tmp). And the LLM may ignore the Phase 1 reminder if mnemon's skill content tells it to recall.

**Solution: a wrapper script at `/usr/local/bin/mnemon`** (the canonical PATH location) shadowing the real binary at `/usr/local/bin/mnemon-real`. The wrapper is bash, baked into the container image. It centralizes three concerns: lock acquisition, phase gating, and observability.

### Dockerfile addition

```dockerfile

    && mv /usr/local/bin/mnemon /usr/local/bin/mnemon-real \
    && ln -s /usr/local/bin/mnemon-wrapper.sh /usr/local/bin/mnemon \
    && /usr/local/bin/mnemon --version
```

### Wrapper script (container/mnemon-wrapper.sh)

```bash
#!/bin/bash
# Mnemon CLI wrapper. Centralizes:
#   1. Phase gating: in shadow mode, recall/search/related return empty + log the query.
#   2. Lock acquisition: write commands acquire host-shared flock per-store.
#   3. Observability: every invocation emits a metric event.
set -euo pipefail

SUBCMD="${1:-}"
shift || true
STORE="${MNEMON_STORE:-default}"
ROLLOUT_FILE="/workspace/agent/.mnemon-rollout.json"
LOCK_FILE="${HOME}/.mnemon/data/${STORE}/.write.lock"
METRICS_FILE="/workspace/agent/.mnemon-metrics.jsonl"
REAL=/usr/local/bin/mnemon-real

# Resolve phase. FAIL CLOSED: missing/malformed rollout file or missing store entry = treat as 'shadow'.
# Per Cycle 3 F3 fix: the wrapper's safety property is "shadow blocks recall regardless"; defaulting
# to 'live' on missing data inverted that guarantee. Containers that reach this wrapper are mnemon-enabled
# (binary path-shadowed only when MNEMON_STORE is set), so a missing/invalid rollout is an error state.
PHASE=$(jq -r ".[\"${STORE}\"].phase // \"shadow\"" "${ROLLOUT_FILE}" 2>/dev/null || echo "shadow")
case "$PHASE" in
  shadow|live|unhealthy) ;;
  *) PHASE="shadow" ;;  # unknown values -> shadow
esac

emit_metric() {
  printf '{"ts":"%s","store":"%s","phase":"%s","cmd":"%s","action":"%s"}\n' \
    "$(date -u +%Y-%m-%dT%H:%M:%SZ)" "$STORE" "$PHASE" "$SUBCMD" "$1" \
    >> "$METRICS_FILE" 2>/dev/null || true
}

case "$SUBCMD" in
  recall|search|related)
    if [ "$PHASE" = "shadow" ]; then
      emit_metric "shadow-blocked"
      # Empty result, mnemon's standard JSON shape:
      echo '{"results":[]}'
      exit 0
    fi
    emit_metric "passthrough"
    exec "$REAL" "$SUBCMD" "$@"
    ;;
  remember|link|forget|embed|gc)
    # Acquire host-shared lock at the host inode (mounted at the same path inside container).
    mkdir -p "$(dirname "$LOCK_FILE")"
    emit_metric "write-locked"
    exec flock "$LOCK_FILE" "$REAL" "$SUBCMD" "$@"
    ;;
  status|viz|version|help|"--version"|"--help"|"")
    # Read-only and informational commands — pass through unmediated.
    emit_metric "passthrough"
    exec "$REAL" "$SUBCMD" "$@"
    ;;
  store|setup)
    # Cycle 3 Codex#4 fix: store and setup mutate filesystem state. Lock and audit them.
    # `setup` is also operator-only — should never originate from the LLM in normal operation.
    mkdir -p "$(dirname "$LOCK_FILE")" 2>/dev/null || true
    emit_metric "admin-locked"
    exec flock "$LOCK_FILE" "$REAL" "$SUBCMD" "$@"
    ;;
  *)
    # Unknown subcommands — fail closed; emit unhealthy event.
    emit_metric "unknown-subcmd"
    echo "[mnemon-wrapper] Unknown subcommand: $SUBCMD" >&2
    exit 2
    ;;
esac
```

### Why this works

- **Locking is host-shared** because `~/.mnemon/data/<store>/` is RW-mounted from host into container at `/home/node/.mnemon/data/<store>/`. Same inode; `flock(2)` advisory locks work cross-process across the host/container boundary.
- **Recall block is deterministic** — even if the LLM ignores the Phase 1 reminder text and calls `mnemon recall`, the wrapper returns empty results. No injection possible during shadow.
- **Phase resolution is idempotent and deterministic** — read from `data/mnemon-rollout.json` (host-mounted RO at `/workspace/agent/.mnemon-rollout.json`). Hook callbacks read the same file the wrapper does; single source of truth.
- **No PreToolUse hook needed for locking** — the wrapper IS the gate. SDK hooks become much simpler: SessionStart returns guide, UserPromptSubmit returns reminder text per phase, Stop emits per-turn metrics. None of them need to know about locks or phase.
- **Observability events** from the wrapper land in `/workspace/agent/.mnemon-metrics.jsonl` (the per-container metrics file the host metrics-collector merges).
