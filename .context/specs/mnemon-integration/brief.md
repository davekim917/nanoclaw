# Brief: Mnemon Integration

> Generated by `/team-brief` — 2026-04-26
> Saved to: `.context/specs/mnemon-integration/brief.md`

Integrate **mnemon** (LLM-supervised persistent memory binary, `github.com/mnemon-dev/mnemon` v0.1.2) and **Ollama embeddings** (`nomic-embed-text`) into NanoClaw v2 per-agent-group containers. Scoped to **Illysium** (`agent_group_id ag-1776377699463-2axxhg`, Slack data-engineering consulting workspace) for first rollout. Designed to be enable-per-group via 2-command operation; designed to compose with the Karpathy LLM Wiki layer (already shipped at step 1) where mnemon is canonical for atomic facts and the wiki is a derived narrative synthesis the operator reviews in Obsidian.

## Requirements

- Mnemon binary v0.1.2 baked into the container image via Dockerfile pinned ARG (`MNEMON_VERSION`) plus per-arch SHA256 verification. Auto-tracked by `/update-container` audit loop and weekly `task-container-updates`.
- `~/.mnemon/` data dir RW bind-mounted into containers; per-group fact isolation via `MNEMON_STORE=<store-name>` env var.
- Ollama runs as a host systemd service with `nomic-embed-text` pulled. Port 11434 firewalled to the Docker bridge subnet only. Containers reach it via `host.docker.internal:host-gateway`.
- Mnemon's recall/remember reminders integrated into the existing programmatic SDK hook chain at `container/agent-runner/src/providers/claude.ts:947`. Single hook channel — no `mnemon setup`-style settings.json patching.
- New host module at `src/modules/mnemon/` (matches `src/modules/agent-to-agent/` convention): `index.ts` (mounts + env), `store.ts` (idempotent `ensureStore` wrapper), `claude-deploy.ts` (translates mnemon hook scripts into SDK programmatic hook entries), `config.ts` (zod schema additions for `ContainerConfig`), `index.test.ts`.
- `ContainerConfig` zod schema explicitly adds a `mnemon` block: `{ enabled: boolean, embeddings: boolean }`. No fictional fields.
- Per-group rollout in TypeScript: `scripts/enable-mnemon.ts` and `scripts/disable-mnemon.ts`. Uses the same TS config-mutation path as self-mod flows, not shell JSON edits. Disable preserves the mnemon DB intact.
- New container skill `container/skills/mnemon-companion/SKILL.md`: source-of-truth matrix, `sources → mnemon → wiki` flow, "always write mnemon first", what never goes in mnemon (rules, transient state, secrets).
- `container/skills/wiki/SKILL.md` reflow on mnemon-enabled groups: agent writes atomic facts to mnemon, wiki pages compiled by a scheduled synthesise task. (Direct-write flow preserved as the default for groups without mnemon.)
- Auto-tracked drift via two scheduled tasks:
  - `task-container-updates` (existing) — extended with `MNEMON_VERSION → mnemon-dev/mnemon` ARG mapping.
  - `task-host-deps-updates` (NEW) — weekly check on `ollama --version` and `nomic-embed-text` model digest.
- Per-group scheduled tasks created by `enable-mnemon.ts`:
  - **Synthesise** (default daily 03:00 ET) — agent reads mnemon for the group, updates wiki pages, appends to wiki/log.md.
  - **GC** (default Sunday 04:00 ET) — agent runs `mnemon gc` for the group, reviews retention suggestions.
  - **Reconcile** (NEW, default Sunday 05:00 ET) — `task-mnemon-wiki-reconcile` cross-checks mnemon's entity graph against wiki pages, flags orphans for operator review.
- `scripts/mnemon-restore.sh` written **and tested** before Illysium goes live (Day 1 deliverable). Documents the recovery path; restore drill executed once with verifiable output.
- `scripts/mnemon-backup.sh` — nightly host cron. Uses `sqlite3 .backup` (or WAL checkpoint + copy), NOT bare `rsync` against the live DB file. Retention: 7 daily + 4 weekly = 11 snapshots, ~1 month coverage. Lands at `~/backups/.mnemon-YYYY-MM-DD/`.
- `/update-container` flow updated with a schema-migration smoke check: on `MNEMON_VERSION` bump, spin up a temp copy of one enabled store's DB against the new binary, run a smoke query, gate the bump on success.
- Per-store host-side `flock` serialization wrapping mnemon writes (mitigates SQLite multi-writer contention since mnemon's DSN sets WAL but no `busy_timeout`). Upstream issue filed requesting `busy_timeout` in the connection string.
- Phase 1 = **shadow mode for one week**: hooks fire, `remember` writes happen, `recall` runs and writes results to `~/.mnemon/data/<store>/shadow.log` instead of returning them as `additionalContext`. Day 8 = full enable (one boolean flip — recall results start injecting).
- Per-group observability emitted as JSON Lines at `data/mnemon-metrics.jsonl` (rotated daily) by a host-side periodic script. Fields: hook fired counts (Prime/Remind/Nudge), hook failures, p95 hook wall-clock, store size (insights/edges/db_size_bytes), last successful remember/recall timestamp, embedding success rate, Ollama availability. Ad-hoc query CLI: `pnpm exec tsx scripts/mnemon-metrics.ts`.
- `docs/mnemon.md` — operator manual: rollout, embedding reindex policy on `nomic-embed-text` digest change, deletion/reconciliation workflow, restore drill, troubleshooting.
- `groups/illysium/container.json` updated with `{ "mnemon": { "enabled": true, "embeddings": true } }` block as the first enrolled group.

## Constraints

- [HARD] Hook delivery via SDK programmatic chain at `container/agent-runner/src/providers/claude.ts:947`. **Do NOT use `mnemon setup`'s settings.json patching mechanism.** Mixing hook channels was flagged as HIGH drift in prior peer review.
- [HARD] Mnemon binary baked into the container image via `ARG MNEMON_VERSION` + per-arch SHA256 ARGs (`MNEMON_SHA256_arm64`, `MNEMON_SHA256_amd64`). **Do NOT bind-mount the host binary.** Bind-mount bypasses the `/update-container` audit loop.
- [HARD] `ContainerConfig` zod schema must explicitly add the `mnemon` block. Unknown keys are silently dropped on round-trip — verified in `src/container-config.ts`.
- [HARD] Per-store isolation via `MNEMON_STORE=<name>` env var. No cross-group memory leakage — each agent group has its own store name and its own DB file.
- [HARD] Module path is `src/modules/mnemon/` (matches `src/modules/agent-to-agent/` convention). No one-off colocations in `container-runner.ts` or root `src/`.
- [HARD] Rollout scripts in TypeScript (`scripts/enable-mnemon.ts`, `scripts/disable-mnemon.ts`). They must call the same TS config-mutation path used by self-mod flows. Bash JSON edits are not permitted.
- [HARD] Phase 1 = exactly one week of shadow mode before Phase 2 full enable. Phase 2 transition is gated on the success criteria below.
- [HARD] No PreCompact integration in this scope. Mnemon v0.1.2's binary does not expose a Compact handler (verified). Defer until upstream ships a real handler.
- [HARD] No fictional config fields. The `mnemon` block in `container.json` is exactly `{ enabled, embeddings }` — no `synthesiseCron`, no `gcCron` (those are scheduled tasks created by `enable-mnemon.ts`, not config).
- [HARD] Multi-arch SHA256 verification in the Dockerfile is required despite establishing a stricter precedent than the existing RENDER/RAILWAY/SUPABASE pattern (which version-pins but doesn't verify). Mnemon has access to all the agent's facts; provenance integrity is non-negotiable.
- [HARD] Ollama runs on the host (NOT in the container, NOT as a sidecar). Port 11434 firewalled to the Docker bridge subnet. Containers reach via `host.docker.internal`.
- [HARD] Wiki canonical role inverts when mnemon is enabled for a group: `sources → mnemon → wiki` flow. Wiki becomes derived synthesis, not the canonical store. Already forward-pointered in `container/skills/wiki/SKILL.md` and `groups/illysium/CLAUDE.local.md`.
- [HARD] Backup mechanism uses `sqlite3 .backup` or WAL-checkpoint-then-copy. **Do NOT use bare `rsync` against the live DB file** — peer review flagged the consistency risk.
- [HARD] Schema migration verification on `MNEMON_VERSION` bump: spin up a temp copy of one enabled store's DB against the new binary, run a smoke query, gate the bump on success. Refuse the bump on failure with operator-readable diagnostic.
- [HARD] Day 1 deliverable: `scripts/mnemon-restore.sh` written **and tested with verifiable output** before Illysium goes live. Backups are worthless without a tested restore path.
- [HARD] Per-store `flock` serialization wraps mnemon writes (host-side). Mitigates SQLite multi-writer contention that mnemon's DSN does not handle natively (WAL set, no `busy_timeout`).
- [SOFT] Ollama-down behavior: degrade gracefully (skip vector signal; return keyword + entity + graph results), log a WARN line, surface in the weekly `task-host-deps-updates` report. **Do NOT fail the agent's turn over an embedding service blip.**
- [SOFT] Default schedules: synthesise daily 03:00 ET, GC Sunday 04:00 ET, reconcile Sunday 05:00 ET. Operator can override per-group when invoking `enable-mnemon.ts`.
- [SOFT] Backup retention: 7 daily + 4 weekly snapshots. Operator can adjust if disk constraints change.

## Non-Goals (Explicitly Out of Scope)

- Touching Claude Code's auto-memory system (`~/.claude/.../memory/MEMORY.md`). It's independent, agent-internal, and remains untouched.
- Wiring a PreCompact handler for mnemon. v0.1.2's binary has no real handler; defer until upstream supports it.
- Adding `synthesiseCron` / `gcCron` fields to `container.json`. Those cadences live in NanoClaw's scheduled-task system, not in container.json.
- Bind-mounting the mnemon binary from the host into containers. Rejected — bypasses `/update-container` audit.
- Deploying mnemon hooks via `.claude/settings.json` file patching. Rejected — mixes hook channels.
- Including mnemon in `/update-nanoclaw` audit. That skill handles upstream NanoClaw merges; mnemon version drift is a `/update-container` concern.
- Rolling mnemon out to other groups before Illysium clears Phase 2. One-group-at-a-time rollout is the discipline.
- Building mnemon-as-MCP-server abstraction. Hook chain integration is the established pattern; MCP can't cover SessionStart/Stop and would split the hook channel.
- Treating the wiki as a portability backup for mnemon. Provider portability rides on porting the mnemon DB + binary directly and rewiring hooks per provider. Wiki is the human audit/curation surface.
- Modifying CLAUDE.local.md / CLAUDE.md to act as a memory layer. Those are rules and instructions, not memory.

## Key Concepts

- **Canonical fact store (mnemon):** The SQLite four-graph (temporal/entity/causal/semantic) DB is the source of truth for atomic insights, edges, importance, decay metadata. Implements the MAGMA paper architecture (arXiv:2601.03236).
- **Derived synthesis (wiki):** Markdown pages compiled FROM mnemon facts by an agent-driven scheduled task. Operator-readable and editable in Obsidian. NOT the canonical store when mnemon is enabled.
- **Layering principle:** Today, no-mnemon groups treat the wiki as canonical. When mnemon is enabled for a group, the wiki becomes derived. Forward-pointered in `container/skills/wiki/SKILL.md` and `groups/illysium/CLAUDE.local.md` already.
- **Shadow mode (Phase 1):** Hooks fire, `remember` writes happen normally, `recall` runs but writes results to `~/.mnemon/data/<store>/shadow.log` instead of returning them as `additionalContext`. Day 8 transition: change one boolean to switch from log-only to inject.
- **LLM-supervised pattern:** Per mnemon's own taxonomy — host LLM (Claude) is the supervisor making judgment calls about what to remember/recall; the binary handles deterministic computation (storage, graph indexing, search, decay). Distinct from LLM-embedded (Mem0, Letta, MemGPT).
- **Synthesise operation:** An agent-driven NanoClaw scheduled task. Reads mnemon insights for the group, updates wiki pages, appends to wiki/log.md. NOT a mnemon CLI command — that was fictional in earlier drafts.
- **Reconcile operation:** A scheduled task (`task-mnemon-wiki-reconcile`) that cross-checks mnemon's entity graph against wiki pages, flags orphans for operator review. Distinct from wiki-lint (which checks wiki internal consistency).
- **Embedding reindex policy:** When `nomic-embed-text` model digest changes, full `mnemon embed` rerun across all enabled stores is required. Documented in `docs/mnemon.md` as a required step in the container-update flow.
- **Schema migration verification:** On `MNEMON_VERSION` bump, the `/update-container` flow spins up a temp copy of one enabled store's DB against the new binary, runs a smoke query, gates the bump on success.
- **Per-store flock serialization:** Host-side `flock` lock per `<store>.lock` file wrapping mnemon write calls. Mitigates SQLite multi-writer contention since mnemon v0.1.2's DSN sets WAL but no `busy_timeout`.
