## Notes

### Biggest concrete finding for v2

**v2's `shutdown()` does not stop active containers synchronously.** `v2/src/index.ts:226-242` only stops polls + tears down adapters. v1's `GroupQueue.shutdown(10000)` is explicitly load-bearing per its comment block — without it, systemd `TimeoutStopSec` stalls every restart because the container subprocesses linger in the cgroup. v2 has the same pattern available (`stopContainer` is already imported in `container-runner.ts`), it's just not wired to the shutdown path. This is the single most actionable HIGH-severity gap in Cat 1.

### Recovery notice gap

v1 sends a user-visible "your last message wasn't processed, please resend" notice on startup for in-flight threads. v2's retry-on-wake covers most cases silently, but for messages that hit `markMessageFailed` (5 retries, still stuck processing), the user gets no DM. This is architecturally different (v2 is more resilient, doesn't need a notice for transient restarts) but leaves a blind spot for permanent-failed messages.

### prompt_too_long auto-recovery missing

v1's `index.ts:2132-2199` is a silent auto-recovery path that Dave likely relies on in long sessions. v2 has no equivalent host-side fallback. This is HIGH because `prompt_too_long` is a frequent load-bearing failure, and the v1 behavior (nuke session, retry with summary) is invisible to the user in the happy case but visible in the failure case if missing.

### Sender allowlist is a config dead letter in v2

`v2/src/config.ts:22` still exports `SENDER_ALLOWLIST_PATH` but no v2 code consumes it. Either port the drop-mode semantics to `unknown_sender_policy` or remove the constant. Risk of operator confusion if they set the config file and nothing reads it.

### Areas I couldn't fully verify (flag for peer reviewers)

- **Cross-cutting**: whether `formatOutbound`'s strip→style→scrub order is preserved in v2 delivery path. If not, styled markdown fragments may bypass the scrubber.
- **Scheduling module**: `computeNextRun` drift-anchoring and advance-before-execute double-dispatch guard from v1 `task-scheduler.ts` — need the scheduling reviewer to confirm `recurrence.ts` preserves both.
- **Container-side**: `/compact` in v1 forwards pre-compact messages into the session BEFORE issuing the slash command so they're in the SDK context pre-compaction. v2's container runs the SDK-native `/compact` — confirm same semantics (the add-compact skill hints at a deliberate design, worth verifying).
- **Auto-save on plain teardown (not PreCompact)**: v1's `cleanupThreadWorkspace` auto-commits on every session exit regardless of cause; v2's container-side autosave fires on `pre-compact` but the unconditional teardown-path autosave needs confirmation from container-side audit.

Not read for Cat 1 deliverable.
